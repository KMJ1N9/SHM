### PROMPT-01：package.json 版本号捏造

- **日期：** 2026-06-05
- **场景：** 代码生成
- **任务描述：** 初始化 uni-app 项目，配置 package.json 依赖版本

#### 失败版本

**Prompt：**（隐式）在创建 package.json 时直接写入了 `3.0.0-4020420240930001` 等版本号

**输出问题：**
- `@dcloudio/uni-app`、`@dcloudio/uni-mp-weixin`、`@dcloudio/vite-plugin-uni` 三个包版本号不存在于 npm registry
- npm install 报 `ETARGET: No matching version found`
- 版本号看起来像真实格式（`3.0.0-<commit-ish>`），但实际是从训练数据中拼凑的，不是真实查询到的

#### 优化版本

**Prompt：** 先用 `npm view <package> versions --json` 查询真实版本列表，确认 Vue 3 对应 alpha 版本线的最新 tag，再写入 package.json

**改进点：**
- 不依赖训练数据中的版本号——npm 版本随时在变
- 区分 Vue 2（2.x 稳定线）和 Vue 3（3.0.0-alpha 线）对应的版本线
- 使用 exact version（不用 `^`），因为 dcloudio 自定义版本格式不兼容 semver

**效果对比：**
- 优化前：npm install 直接失败，需要 3 次尝试才找到正确版本
- 优化后：一次 `npm install` 成功，667 个包安装完成

#### 经验总结

**任何 package.json 中的版本号都必须通过 `npm view` 验证真实存在。** 特别是中国厂商的包（dcloudio、cnpm 等），版本格式可能与 semver 标准差异大，训练数据中的版本号可能已过期或完全捏造。

### PROMPT-02：治理文档创建为"暂无"空壳

- **日期：** 2026-06-05
- **场景：** 文档生成
- **任务描述：** 完成完整开发.md §4.4 的知识管理体系（ADR/复盘/事故/技术债/Prompt 失败案例）

#### 失败版本

**Prompt：** "把未完成的全部做了"（来自前一 session）

**输出问题：**
- `adr/README.md` — 只有模板 + "暂无"占位行
- `tech-debt.md` — 所有统计为零 + "暂无"
- `prompt-failures.md` — 案例列表为空
- `postmortems/README.md` — 只有模板
- `postmortems/incidents.md` — 只有模板
- 这些文件被创建为"预置模板"，但没有一个真正可用
- 用户反馈："不能创建了之后当个摆设"

#### 优化版本

**思路：** 区分"当前可填充"与"当前阶段合理为空"：

| 文档 | 当前能否填 | 行动 |
|------|:--:|------|
| ADR | 能 | 从技术架构文档提炼 5 个已有决策 |
| 技术债 | 能 | 录入 alpha 依赖、JSON 字段、偏移分页 4 项 |
| Prompt 失败 | 能 | 记录版本号捏造、空壳文档 3 个案例 |
| Bug 复盘 | 不能（没代码） | 保留模板，标注"待开发阶段" |
| 事故记录 | 不能（没上线） | 保留模板，标注"待上线后" |

**改进点：**
- 创建预置文件时必须区分"即刻可用"和"将来预留"
- 即刻可用的不能在创建后留空
- 将来预留的应在文件中明确标注启用条件

**效果对比：**
- 优化前：7 份治理文档，5 份是纯模板（71% 空壳率）
- 优化后：ADT 5 份 + 技术债 4 项 + Prompt 案例 3 条，空壳归零

#### 经验总结

**"创建目录/模板"不等于"完成任务"。** 当任务要求建立知识管理体系时，应该：
1. 分类：哪些现在就能填（已有信息，只需提取）→ 立即填
2. 哪些现在不能填（信息不存在）→ 标注启用条件，让用户知道为什么空
3. 绝不能清一色"暂无"——那是应付任务，不是完成任务

### PROMPT-03：旧产物未随新建一起清理

- **日期：** 2026-06-05
- **场景：** 代码生成
- **任务描述：** 在 miniprogram/components/ 下创建 6 个文档定义的 .vue 占位组件

#### 失败版本

**Prompt：**（来自前一 session）创建组件骨架

**输出问题：**
- 创建了 6 个新 .vue 文件
- 但 components/ 下遗留了 6 个旧空目录（empty-state, goods-card, image-upload, search-bar, skeleton, toast）
- 这些是之前另一轮尝试创建的，与文档定义的组件名不匹配，且全是空目录
- 用户需要额外提醒"清理它"才处理

#### 优化版本

**思路：** 新建结构的同时，`ls` 检查目录下已有内容 → 对比文档定义 → 删除不匹配的旧产物

**改进点：**
- 创建新文件前先扫描目标目录已有内容
- 对比新旧结构差异
- 将清理作为同一次操作的一部分，不留在事后

**效果对比：**
- 优化前：用户发现 6 个垃圾目录，需额外提问
- 优化后：一次操作完成创建+清理，目录干净

#### 经验总结

**增删改一体。** 当你新建/修改了某个目录的内容时，检查是否有你的旧尝试留下的产物。用户不应该为你的上次尝试"收垃圾"。

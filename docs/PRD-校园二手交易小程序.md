## 一、产品概述

### 1.1 一句话描述

帮助广州应用科技学院肇庆校区学生，通过校内 C2C 二手交易平台，实现安全、可信的闲置物品面交换，解决校园闲置物品流通和交易信任问题。

### 1.2 核心价值

| 痛点 | 解决方案 |
|------|----------|
| 没有好的渠道卖二手物品（只能微信群喊） | 专注单校的瀑布流商品广场 + 分类筛选 |
| 物品交易没有保障 | 站内 IM 留证 + 客服介入 + 举报机制 |
| 没有信任机制 | 微信手机号认证（真实身份）+ 多维度互评（实名可见）|

### 1.3 产品形态

微信小程序，单校封闭生态（仅广应科肇庆校区学生可注册）。

## 二、用户体系

### 2.1 用户角色

| 角色 | 权限概述 |
|------|----------|
| 买家 | 浏览、搜索、筛选、聊天、下单、确认收货、评价、举报 |
| 卖家 | 发布商品、管理商品、聊天、举报 |
| 管理员/客服 | 用户管理、商品下架、纠纷处理、数据看板 |

### 2.2 注册与认证

- **注册方式**：微信小程序 `getPhoneNumber` 能力，用户授权后直接读取手机号完成注册/登录。注册与登录合一——首次授权即自动注册，后续授权即登录
- **认证强度**：手机号为微信实名绑定，天然具备实名属性，无需额外验证码流程
- **非本校拦截**：MVP 阶段不做学校限制（手机号无法判断是否本校学生）。通过以下方式替代：
  - 用户注册后自行填写班级、宿舍楼栋信息（选填但鼓励填写，增强交易信任）
  - 后续迭代可接入学校统一身份认证平台加强学校绑定
- **优势**：相比邮箱注册，手机号注册体验更流畅（一键授权 vs 填邮箱+等验证码），且天然实名降低匿名作恶风险

## 三、核心功能

### 3.1 商品发布

**必填字段：**
- 商品图片（最多 6 张）
- 商品名称
- 成色/新旧程度（如：全新、95 新、9 成新、8 成新、7 成新及以下）
- 原价（¥）
- 期望售价（¥）
- 交易地点偏好（如：1 栋宿舍楼下、食堂门口、图书馆门前）
- 是否接受议价（是/否）
- 商品分类/标签

**发布页 UX**：字段较多，发布页需**分步/分组**设计，避免单页过长（如：第一步上传图片 → 第二步填写商品信息 → 第三步确认发布）。

**审核机制**：发布即上架 + 事后抽查/举报下架（自动审核仅在发布时即时拦截明显违规内容，极少数触发高风险规则但不明确的情况进入人工复核，详见技术架构 §16.2）

**商品类目**：全品类，重点类目包括教材、电子产品、生活用品、宿舍家具、运动器材等。

### 3.2 商品浏览与发现

**首页：**
- 瀑布流布局（参考闲鱼）
- 默认按发布时间倒序（新品优先）
- 基于浏览历史的推荐（后续迭代，MVP 不做）

**搜索：**
- 支持按商品名称搜索
- 支持按分类搜索
- 支持按**卖家昵称**搜索（低成本、实用——用户可能记住了某个靠谱卖家但忘了商品名）

**筛选（右上角按钮，点击弹出侧边栏，参考悠悠有品）：**
- 按分类筛选
- 按成色筛选
- 按价格区间筛选

### 3.3 站内 IM（即时通讯）

- 小程序内聊天，消息可留存作为纠纷证据
- 技术方案：腾讯云 IM SDK（免费档 100 日活，Provider 抽象层可替换）
- **免责提示**：聊天界面需醒目提示——"如买卖双方私下交换微信/其他联系方式进行交易，发生纠纷后平台不负责"

### 3.4 交易流程

**MVP 阶段（B 模式 — 钱不过平台）：**

```
卖家发布商品 → 买家浏览 → 站内IM议价 →
线下约定面交点 → 当面验货/一手交钱一手交货 →
买家点击"确认收货" → 触发双方互评
```

> 平台记录交易状态，但不介入资金流转。此阶段完全免费。

**后续迭代（A 模式 — 钱过平台）：**

```
买家小程序内支付 → 资金暂存平台 →
线下面交 → 买家确认收货 → 平台打款给卖家 →
触发互评 → 平台抽取 3% 交易费
```

> A 模式需先注册个体工商户 → 申请微信支付商户号。MVP 阶段不做。

**退款流程（A 模式上线后）：**
- 纠纷判定退款 → 原路返回买家钱包 → 全额退款
- 买家申请退款但卖家已出发 → 买家承担"运费"（补偿卖家）

### 3.5 评价体系

- **互评时机**：确认收货后触发。面交场景用户可能忘记点"确认收货"，增加**超时自动确认**机制——买家/卖家任一方点击"已面交"后，另一方 3 天内未确认则系统自动确认并触发互评
- **评价维度**：多维度（沟通态度、守时程度、商品与描述一致度等）
- **实名可见**：评价不可匿名
- **差评处理**：差评不直接触发限制，但差评会扣减信誉分，信誉分低于阈值将自动触发功能限制（见 §3.5.1）

### 3.5.1 信誉分体系

> 信誉分体系是评价体系的量化补充——用户行为通过积分量化，自动触发功能限制，减少客服人工介入。

| 项目 | 分值 |
|------|:--:|
| 新用户初始分 | 100 |
| 信誉分上限 | 200 |
| 举报成立（被举报方） | -30 |
| 完成交易（卖家） | +2 |
| 收到好评（4 星及以上） | +1 |
| 收到差评（2 星及以下） | -5 |

| 信誉分区间 | 限制 |
|-----------|------|
| ≥ 60 | 正常使用全部功能 |
| 30 ~ 59 | 禁止发布商品，可浏览/购买/聊天 |
| < 30 | 禁止发起交易，仅可浏览 |
| ≤ 0 | 自动标记为高风险用户，通知管理员人工审核决定是否封禁 |

信誉分低于 60 分的用户，可通过积极完成交易逐步恢复分数。

### 3.6 举报机制

| 项目 | 说明 |
|------|------|
| 举报入口 | 商品详情页 + 用户个人主页 |
| 举报类型 | 商品与描述严重不符、对方辱骂/骚扰、疑似骗子、其他 |
| 客服响应 SLA | 3 小时内响应 |
| 争议办结 SLA | 3 个工作日内 |

### 3.7 客服工作流（适配单人开发）

> 单人维护客服 SLA 是核心瓶颈。MVP 阶段用「表单收集 + 集中处理」降低实时响应压力。

**用户端：**
- 举报/申诉入口统一为**表单页**（非实时聊天客服）
- 表单字段：问题类型、涉及商品/订单、对方昵称、问题描述、上传截图证据
- 提交后自动回复："已收到，客服将在 3 小时内响应"

**客服端（后台）：**
- 工单列表按提交时间排序，支持标记：待处理 / 处理中 / 已解决
- 封号等操作为**一键按钮**，不写额外代码

| 层 | 选型 | 理由 |

## 八、MVP 验证标准

> MVP 的核心目标是验证"校园内是否有真实二手交易需求，且平台模式能满足此需求"（见 §五 商业模式）。没有指标就无法判断何时从 MVP 进入下一阶段。本章定义验证通过的具体标准。

### 8.1 冷启动说明

校园二手交易是典型的**双边市场**——买家需要足够的商品可逛，卖家需要足够的买家来询价。从零启动时双方都不会自发出现。

**开发者需在正式推广前完成以下准备工作：**

1. **种子商品**：自己或邀请 5-10 位朋友，提前发布 15-20 件真实商品（覆盖不同分类：教材、电子产品、生活用品、运动装备等），确保首页瀑布流至少能刷出 2 屏内容
2. **种子用户**：邀请 10-15 位同学作为首批用户，承诺他们"第一批看到新品"
3. **推广渠道**：准备好班级群/宿舍群/校园墙/宿舍楼公告栏的推广文案和小程序码

**以下指标阈值的前提是上述冷启动已完成。** 未做冷启动直接推广导致的数据低迷，不是需求不存在——是启动方式不对。

### 8.2 观察周期

| 节点 | 时间 | 目的 |
|------|------|------|
| 上线 | T+0 | 小程序通过微信审核，分享到第一个班级群 |
| 健康检查 | T+14 天 | 非正式检查——确认不是"完全没人用"。若注册 < 30，检查推广是否到位 |
| 正式评估 | T+42 天（第 6 周） | 按 §8.4 判定标准做通过/待定/不通过决策 |
| 最终评估 | T+70 天（第 10 周） | 仅当第 6 周为"待定"时启动，为最后一次评估 |

> **⚠️ 时机警告**：避免在考试周前 2 周内发布。最佳发布窗口：开学后 2-4 周（学生安顿后有闲置处理需求）或学期结束前 4 周（毕业生离校前清宿舍）。暑假/寒假期间暂停评估——用户离校，面交无法进行，指标无参考意义。

### 8.3 指标体系

全部 11 项指标均可通过 SQL 从现有数据库表直接计算，无需额外埋点基础设施（各指标 SQL 查询语句见 [技术架构文档 §十七](技术架构文档.md) Analytics 模块）。

#### 8.3.1 需求验证（M1-M3）——"有人来吗？有人发吗？"

| # | 指标 | 计算方法 | 数据源 | 成功阈值 | 理由 |
|:--:|------|----------|--------|:--:|------|
| M1 | 累计注册用户 | `COUNT(*) FROM users` | `users` 表 | ≥ 150 | 肇庆校区估约 6,000-10,000 在校生，150 ≈ 1.5%-2.5% 渗透率。低于此数说明要么推广没到位，要么学生无需求 |
| M2 | 累计发布商品数 | `COUNT(*) FROM products WHERE deleted_at IS NULL` | `products` 表 | ≥ 60 | 60 件商品 ≈ 首页瀑布流 3 屏内容。若不足，买家打开后认为"东西太少"而流失 |
| M3 | 独立卖家数 | `COUNT(DISTINCT user_id) FROM user_events WHERE event='publish_product'` | `user_events` 表 | ≥ 20 | 排除"一个人清宿舍"的假象——20 个独立卖家说明二手交易需求是分布式的 |

**判定**：M1-M3 全部达标，需求验证通过。M1 达标但 M2/M3 不达标：说明有人好奇注册但没人愿意发商品——发布流程或激励有问题。

#### 8.3.2 交易闭环（M4-M6）——"能成交吗？"

| # | 指标 | 计算方法 | 数据源 | 成功阈值 | 理由 |
|:--:|------|----------|--------|:--:|------|
| M4 | 累计成交订单 | `COUNT(*) FROM orders WHERE status='completed'` | `orders` 表 | ≥ 25 | 6 周 25 单 ≈ 每天 0.6 单。面交模式下的最低可持续节奏。低于此数说明平台未形成交易习惯 |
| M5 | 订单完成率 | `completed / (completed + cancelled)` | `orders` 表 | ≥ 55% | 面交场景约时间/改主意/找到更便宜的都会导致取消，55% 是健康线。低于 40% 说明约定→见面流程有系统性问题 |
| M6 | 商品动销率 | `COUNT(DISTINCT product_id FROM orders) / COUNT(*) FROM products` | `orders` / `products` | ≥ 25% | 至少 1/4 的商品被下单过，验证供给质量——不是随便什么东西都有人要 |

**判定**：M4 必须达标（这是最核心的验证信号）。M5 单独不达标 → 检查面交约定机制。M6 单独不达标 → 商品类目或定价偏高。

#### 8.3.3 信任与质量（M7-M8）——"安全吗？"

| # | 指标 | 计算方法 | 数据源 | 成功阈值 | 理由 |
|:--:|------|----------|--------|:--:|------|
| M7 | 纠纷率 | `COUNT(orders WHERE status='disputed') / COUNT(orders WHERE status IN ('completed','disputed'))` | `orders` 表 | ≤ 8% | MVP 用户基数小，单次纠纷拉高比例，8% 是合理上限。超过则说明商品描述失实或恶意用户存在 |
| M8 | 互评完成率 | `双方均评价的订单数 / 已完成订单数`（`GROUP BY order_id HAVING COUNT(*)=2` / 已完成订单总数） | `reviews` 表 | ≥ 40% | 互评是信任飞轮的核心。面交场景下用户已见面，评价意愿应高于纯线上交易。40% 双方评价是及格线 |

**判定**：M7 和 M8 任一不达标 → 信任机制需要加强。M7 不达标优先检查举报处理时效和处罚力度；M8 不达标优先检查评价入口是否明显、完成订单后的评价提醒是否触达。

#### 8.3.4 可持续性（M9-M11）——"能活下去吗？"

| # | 指标 | 计算方法 | 数据源 | 成功阈值 | 理由 |
|:--:|------|----------|--------|:--:|------|
| M9 | 7 日回访率 | 任意一周有事件的用户中，下一周仍至少有一次事件的用户占比 | `user_events` 表 | ≥ 20% | 不是"下载用一次就删"。20% 周留存对工具型小程序是健康线。低于此数：产品缺乏召回机制 |
| M10 | 第 6 周活跃卖家 | 评估前 7 天内有 `publish_product` 事件的独立用户数 | `user_events` 表 | ≥ 5 | 验证供给侧在持续——第 6 周仍有 ≥ 5 人在发新商品，说明平台没有被废弃 |
| M11 | 单人运维时间 | 开发者自行记录（处理工单 + 回复举报 + 下架商品 + 排查 bug） | 人工 | ≤ 5 小时/周 | **专为单人开发设计的指标。** 若 MVP 阶段就需要 >5h/周运维，说明自动化不足，扩张后必然崩溃。第 4-6 周每天记录，取周均值 |

**判定**：M11 单独不达标但其他指标都好 → 正循环信号——需求真实但自动化需跟上。优先开发自动回复模板、自动举报分类、批量处理工具。M9+M10 同时不达标 → 平台正在"死亡"——用户来过但不再回来。

### 8.4 综合判定（第 6 周）

| 结果 | 条件 | 含义 | 下一步 |
|:--:|------|------|------|
| ✅ **验证通过** | 11 项中 ≥ 8 项达标，且 **M4（成交数）和 M7（纠纷率）必须达标** | 校园内确实存在可被平台满足的二手交易需求 | 进入下一阶段：① 注册个体工商户 ② 接入微信支付 ③ A 模式资金担保 ④ 付费置顶功能 |
| ⚠️ **待定** | 5-7 项达标 | 需求可能成立但存在阻碍，有改善空间 | 延长观察 4 周到第 10 周。根据 §8.5 诊断短板并针对性优化，第 10 周重新评估。第 10 周仍未通过 → 视为不通过 |
| ❌ **不通过** | ≤ 4 项达标 | 需求假设可能不成立，或执行有重大问题 | 检查三个根本原因：① 校园规模是否不足以支撑独立的二手交易平台？② 微信群的"随时喊"模式是否更适合学生习惯？③ 产品本身（流程/体验/速度）是否有致命缺陷？根据结论决定是 pivot 还是终止 |

### 8.5 常见失败模式与诊断

| 失败模式 | 特征信号 | 可能原因 | 建议动作 |
|----------|----------|----------|----------|
| **没人来** | M1 < 150 | 推广未触达目标用户 | 检查推广渠道：班级群是否被屏蔽？校园墙是否有互动？考虑宿舍楼贴小程序码（扫码即用，最直接） |
| **来了不发** | M1 达标，M2/M3 不达标 | 发布流程有摩擦；用户习惯微信群直接发图 | 自己走一遍发布流程，计时——如果 >2 分钟，简化；观察 `user_events` 中 `publish_product` 事件在哪个步骤断流 |
| **发了不买** | M2 达标，M4 不达标 | 商品价格/品相不佳；IM 沟通后买家流失；首页推荐策略有问题 | 检查 `click_want` → `create_order` 转化率；如果 click_want 很多但 create_order 很少 → IM 环节有问题；如果 click_want 就很少 → 商品本身或首页排序有问题 |
| **买了不见** | M4 达标，M5 < 40% | 面交约定机制不可靠；鸽子率高 | 考虑在下单后增加"约定面交时间地点"功能；确认收货流程检查是否有阻碍（如必须双方确认才能完成） |
| **纠纷多** | M7 > 8% | 商品描述不实；或存在恶意用户 | 检查纠纷工单的原告/被告分布——如果是少数用户反复被举报 → 针对性处理；如果是分散的 → 加强商品发布时的如实描述引导和照片规范 |
| **没人评** | M4 达标，M8 < 40% | 评价入口不明显；评价流程太复杂；用户没动力 | 检查确认收货后的评价提醒弹窗是否触达；考虑在 IM 会话中自动发送评价提醒系统消息 |
| **一锤子买卖** | M9 < 20% 且 M10 < 5 | 用户没有理由回来——交易完成后就流失了 | 增加召回机制：上新通知、降价提醒、收藏商品状态变更 Push。同时检查是否有内容持续吸引回访（如首页是否每次打开都一样） |
| **运维吃不消** | M11 > 5h/周 | 客服工单太多或处理流程低效 | 分析工单类型分布：重复性问题 → 自动回复模板；纠纷判定耗时 → 标准化裁决规则减少主观判断；恶意举报 → 提高举报门槛（信誉分限制） |

### 8.6 数据看板建议

当前管理后台 Dashboard API（`GET /api/admin/dashboard`）已返回 `today`（今日活跃用户/新增商品/新增订单/完成订单/新增举报）和 `total`（累计用户/商品/订单/纠纷率）。建议增加以下字段以支持 MVP 指标追踪：

```
// 建议新增的 Dashboard 响应字段
{
  mvp_metrics: {
    unique_sellers:         18,     // M3：独立卖家数
    sell_through_rate:      "28%",  // M6：商品动销率
    review_completion_rate: "45%",  // M8：互评完成率
    weekly_active_sellers:  6,      // M10：近 7 天活跃卖家
    weekly_retention:       "22%"   // M9：7 日回访率（按上一周活跃用户计算）
  }
}
```

> 实现成本极低——在上述现有表上增加 5 条 SQL 聚合查询。Dashbaord 页面新增一个"MVP 指标"卡片区域，评估日打开即可一眼判断达标情况。

## 九、风险与待确认事项

| 风险/待确认 | 说明 |
|-------------|------|
| 用户身份真实性 | 手机号无法判断是否本校学生，MVP 阶段通过用户自填班级/宿舍信息建立信任，后续可接入学校统一身份认证 |
| 微信小程序审核 | 涉及 C2C 交易类目，可能需补充资质。建议提前提交审核看反馈；获取手机号能力需通过微信认证 |
| 个人主体限制 | 微信小程序个人主体无法开通支付分账，后续升级需注册企业/个体户 |
| 单人维护 | 客服已改为表单收集 + 集中处理模式，初期用户少可应付。IM 消息通知可后续加微信提醒防遗漏 |

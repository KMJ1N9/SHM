## 一、测试策略概述

### 1.1 金字塔模型

```
┌────────────────────────────────────┐
│  E2E（MVP 不做）                    │  ← 资源有限，暂不投入
├────────────────────────────────────┤
│  API 集成测试（~30 条）             │  ← supertest + 真实路由 + MySQL 测试库
│  关键用户流程端到端验证              │
├────────────────────────────────────┤
│  单元测试（~90 条）                 │  ← vitest + 纯函数 / mock 依赖
│  逻辑密集模块全覆盖                  │
└────────────────────────────────────┘
```

### 1.2 测试原则

| 原则 | 说明 |
|------|------|
| **关键路径 100%** | 涉及资金/信誉/安全的代码必须测试（订单状态机、信誉分计算、JWT 验证） |
| **快速反馈** | 全部测试 < 30s，每次 `git push` 前本地跑通 |
| **可重复** | 每个测试独立运行，`beforeEach` 清空数据库重建 schema |
| **读得懂** | 测试名中文描述：`describe('订单状态机') + it('pending 7 天后超时取消')` |
| **只 mock 网络边界** | `config/`、`repository/`、`services/` 内部不做 mock，用真实测试数据库验证 |

### 1.3 范围边界

| 类型 | 说明 |
|------|------|
| **测** | 服务端业务逻辑、API 端到端流程、定时任务逻辑、错误码覆盖 |
| **不测** | 前端 uni-app 页面（MVP 阶段不做 E2E）、腾讯云 IM SDK 本身、微信 API 本身、COS SDK 本身 |
| **部分测** | IM 系统消息 payload 格式验证（mock TIM SDK）、COS STS 凭证生成逻辑（mock COS SDK） |

## 二、测试环境

### 2.1 技术栈

| 层 | 工具 | 用途 |
|----|------|------|
| 测试运行器 | **vitest** | 单元测试 + 集成测试，原生 ESM |
| 断言库 | vitest 内置（兼容 Jest API） | `expect()`, `toEqual()`, `toMatchObject()` |
| HTTP 测试 | **supertest** | Express app 无需启动监听 |
| 数据库 | MySQL `campus_market_test` | 独立测试库，自动迁移 + 种子 |
| Mock | vitest `vi.fn()` | 微信 API、COS 上传、IM REST API |

### 2.2 数据库配置

```env
# .env.test
DB_HOST=127.0.0.1
DB_USER=root
DB_PASSWORD=
DB_NAME=campus_market_test
```

- 测试数据库与开发/生产数据库**物理隔离**
- 每个 `describe` 块 `beforeAll` 创建独立 app 实例，`afterAll` 关闭连接池
- 每个 `it` 用例 `beforeEach` 清空所有表并重新执行 migration（`SET FOREIGN_KEY_CHECKS = 0` → `DROP TABLE` → migration → `SET FOREIGN_KEY_CHECKS = 1`）

### 3.1 订单状态机（P0 — 12 条）

> **文件：** `server/__tests__/unit/services/order-state.test.js`
> **被测函数：** `orderService.validateTransition(currentStatus, nextStatus, actor)`
> **原则：** 覆盖所有合法转换 + 非法转换拒绝 + 边界参与者校验

#### 3.1.1 合法转换（7 条）

| ID | 用例 | 当前状态 | 目标状态 | 操作者 | 预期 |
|:--:|------|:--:|:--:|--------|------|
| OS-001 | 任一方点击"已面交"→订单进入 met | `pending` | `met` | 买家 | ✅ 允许，`met_at` 记录时间 |
| OS-002 | 卖家点击"已面交"同样有效 | `pending` | `met` | 卖家 | ✅ 允许 |
| OS-003 | 买家取消订单 | `pending` | `cancelled` | 买家 | ✅ 允许，`cancelled_by='buyer'`，商品恢复 `active` |
| OS-004 | 卖家取消订单 | `pending` | `cancelled` | 卖家 | ✅ 允许，`cancelled_by='seller'`，商品恢复 `active` |
| OS-005 | 买家确认收货→订单完成 | `met` | `completed` | 买家 | ✅ 允许，`confirmed_at` 记录时间 |
| OS-006 | 买家举报交易纠纷→进入争议 | `pending` | `disputed` | 系统（举报触发） | ✅ 允许，关联工单创建 |
| OS-007 | 面交后举报→进入争议 | `met` | `disputed` | 系统（举报触发） | ✅ 允许，商品同时冻结 |

#### 3.1.2 非法转换（5 条）

| ID | 用例 | 当前状态 | 目标状态 | 预期 |
|:--:|------|:--:|:--:|------|
| OS-008 | 已完成订单不能回到面交状态 | `completed` | `met` | ❌ 拒绝，错误码 `3001` |
| OS-009 | 已取消订单不能回到面交状态 | `cancelled` | `met` | ❌ 拒绝，错误码 `3001` |
| OS-010 | 已完成订单不能取消 | `completed` | `cancelled` | ❌ 拒绝，错误码 `3001` |
| OS-011 | 不能跳过面交直接完成 | `pending` | `completed` | ❌ 拒绝，错误码 `3001` |
| OS-012 | 非交易双方不能操作订单 | `pending` | `met` | ❌ 拒绝，错误码 `1005` |

> **设计依据：** 状态流转图来自 [技术架构 §7.1](技术架构文档.md)。`disputed` 状态只能由举报举报触发，由客服裁决为 `completed` 或 `cancelled`，不可由用户直接操作。

### 3.2 信誉分计算（P0 — 11 条）

> **文件：** `server/__tests__/unit/services/credit.test.js`
> **被测函数：** `creditService.calculate(newScore, delta, reason)`
> **业务规则来源：** [PRD §3.5.1](PRD-校园二手交易小程序.md) + [技术架构 §十八](技术架构文档.md)

| ID | 用例 | 初始分 | 操作 | 预期分数 | 说明 |
|:--:|------|:--:|------|:--:|------|
| CS-001 | 新用户初始分 | — | 注册 | 100 | 与 PRD 一致 |
| CS-002 | 完成交易（卖家）+2 | 100 | `order_completed` | 102 | 买家不加分 |
| CS-003 | 收到好评（4 星及以上）+1 | 102 | `positive_review` | 103 | 评价方给 ≥4 星 |
| CS-004 | 收到差评（2 星及以下）-5 | 103 | `negative_review` | 98 | 评价方给 ≤2 星 |
| CS-005 | 举报成立 -30 | 100 | `report_upheld` | 70 | 被举报方扣分 |
| CS-006 | 扣分不跌破 0 | 20 | `report_upheld`（-30） | 0 | 硬底线，不可为负 |
| CS-007 | 加分不超 200 上限 | 199 | `order_completed`（+2） | 200 | 硬上限，不可超出 |
| CS-008 | 上限边界：刚好 200 不再增加 | 200 | `positive_review`（+1） | 200 | 封顶 |
| CS-009 | 低于 60 分触发发布限制 | 59 | — | — | 返回 `4008` 错误码 |
| CS-010 | 低于 30 分触发交易限制 | 29 | — | — | 返回 `4009` 错误码 |
| CS-011 | ≤0 分标记高风险 | 0 | — | — | 通知管理员（系统消息 type=`credit_alert`） |

> **注意：** 3 星评价不触发任何信誉分变动。信誉分变动必须记录 `credit_score_logs` 表（含 `delta`、`reason`、`metadata`）。

### 3.3 JWT 双 Token 认证（P0 — 8 条）

> **文件：** `server/__tests__/unit/services/jwt.test.js`
> **被测函数：** `jwtService.signAccess()`, `jwtService.signRefresh()`, `jwtService.verify()`

| ID | 用例 | 预期 |
|:--:|------|------|
| JWT-001 | 签发 Access Token → 包含 `userId`、`role`、`token_version` | ✅ payload 含预期字段 |
| JWT-002 | Access Token 过期 → 验证返回 `TokenExpiredError` | ❌ 错误码 `1002` |
| JWT-003 | Access Token 签名伪造 → 验证拒绝 | ❌ 错误码 `1002` |
| JWT-004 | Refresh Token 签发 → 30 天有效期 | ✅ `exp` = `iat` + 30 天 |
| JWT-005 | Refresh Token 过期 → 需重新授权登录 | ❌ 错误码 `1002` |
| JWT-006 | `token_version` 变更后旧 Access Token 失效 | ❌ 错误码 `1003`（与 1002 区分） |
| JWT-007 | 有效 Token → `req.user` 注入正确 | ✅ `req.user = { userId, role, tokenVersion }` |
| JWT-008 | Authorization header 缺失或格式错误 | ❌ 错误码 `1001` |

> **1002 vs 1003 区别：** 1002 是 token 自身过期/伪造，前端静默跳转登录；1003 是被动失效（管理员封禁触发 version+1），前端弹窗提示后跳转。

## 六、测试数据管理

### 6.1 隔离规则

| 规则 | 实现方式 |
|------|---------|
| 每个 `describe` 独立 app 实例 | `beforeAll` 创建 pool + app，`afterAll` 关闭 pool |
| 每个 `it` 独立数据 | `beforeEach` 清空所有表并重新执行 migration |
| 不共享可变状态 | 禁止 `let sharedVar` 跨用例修改 |
| 外部调用全部 mock | `vi.mock('../../services/wechat.js')` 等 |
| CI 每个测试文件独立线程 | vitest `pool: 'threads'`，`isolate: true` |

### 6.2 测试数据库初始化流程

```
beforeAll:
  1. mysql.createPool({ database: 'campus_market_test' })
  2. runMigrations(pool, 'up')  → 建表
  3. createApp(pool)            → Express app 实例

beforeEach:
  1. SET FOREIGN_KEY_CHECKS = 0
  2. DROP TABLE IF EXISTS `所有业务表`（倒序，先删子表）
  3. SET FOREIGN_KEY_CHECKS = 1
  4. runMigrations(pool, 'up')  → 重建空表
  5. 按需插入测试种子数据

afterAll:
  1. pool.end()
```

### 6.3 工厂函数规范

- `createTestUser(overrides)` — 返回 user 对象，默认 `credit_score=100, role='user', status='active'`
- `createTestProduct(sellerId, overrides)` — 返回 product 对象，默认 `status='active'`
- `createTestOrder(buyerId, productId, overrides)` — 返回 order 对象，默认 `status='pending'`
- 所有工厂函数支持 `overrides` 参数按需覆盖
- 工厂函数不直接插入数据库，由调用方负责 `INSERT`

## 七、Mock 策略

### 7.1 原则

> **只 mock 网络边界。** `config/`、`repository/`、`services/` 内部的 MySQL 调用、业务逻辑**不做 mock**，用真实测试数据库验证。

### 7.2 Mock 清单

| 外部依赖 | Mock 方式 | 返回值示例 |
|---------|----------|-----------|
| 微信 `getPhoneNumber` API | `vi.fn().mockResolvedValue(...)` | `{ phone: '13800138000' }` |
| 微信 `getAccessToken` | `vi.fn().mockResolvedValue(...)` | `{ access_token: 'mock_token', expires_in: 7200 }` |
| 腾讯云 IM REST API | `vi.fn().mockResolvedValue(...)` | `{ ActionStatus: 'OK' }` |
| COS STS | `vi.fn().mockResolvedValue(...)` | `{ credentials: {...}, expiredTime: ... }` |
| 腾讯云 COS 上传 | `vi.fn().mockResolvedValue(...)` | `{ Location: 'https://cos.example.com/xxx.jpg' }` |

### 7.3 不做 Mock 的层

| 层 | 原因 |
|----|------|
| `repository/` | 是数据访问层，用真实 MySQL 测试数据库验证 SQL 正确性 |
| `services/` | 是业务逻辑层，通过真实 DB 验证完整链路 |
| `middleware/` | JWT 中间件用真实签名+验证链路；限流器用可控时钟 |
| `utils/` | 纯函数，直接调用测试 |

## 一、技术栈总览

| 层 | 选型 | 版本/方案 |
|----|------|-----------|
| 小程序框架 | uni-app (Vue 3) | HBuilder X 开发 |
| 后端框架 | Node.js + Express | **5 层结构**（routes/controllers/services/repository/models） |
| 数据库 | MySQL + mysql2 | mysql2/promise 连接池，原生参数化 SQL，无 ORM 抽象层 |
| 即时通讯 | 腾讯云 IM SDK | 免费档（100 日活），前端直接接入，服务端通过 IM Provider 抽象层调用 REST API |
| 文件存储 | 腾讯云 COS | 服务端 STS 临时密钥，前端直传 |
| 部署 | 腾讯云轻量服务器 | CentOS / Ubuntu |
| 进程管理 | PM2（fork 模式） | 守护 + 自动重启。**必须用 fork 而非 cluster**，否则 node-cron 定时任务会在多进程中重复执行 |
| 日志 | winston | 按天切割，3 级 logger（access/error/business） |
| 限流 | 内存令牌桶 | 全局 60req/min，敏感接口独立 10req/min |
| 敏感词 | DFA 算法 | 本地词库，覆盖标题/昵称/评论/举报内容 |
| 测试 | vitest + supertest | 单元测试 + API 集成测试（MySQL 测试数据库） |

### 为什么选型如此

- **MySQL**：开发者最熟悉的数据库，行级锁 + 完整 ALTER TABLE + 在线 DDL，几千日活场景性能充裕。`mysql2/promise` 提供 async/await 连接池，参数化查询防注入
- **mysql2（无 ORM）**：直接写 SQL 而非通过 knex 等 Query Builder——开发者对 SQL 更熟悉，出问题时 MySQL general log 搜到的 SQL 就是代码里写的，排查链路零黑盒。Repository 层封装所有数据库操作，业务层不接触 SQL
- **腾讯云 IM**：IM 是重活——消息存储、离线推送、图片消息、已读未读，自建至少 2 周，SDK 半天接入。免费档 100 日活，超量需升级（约 ¥500/月），MVP 阶段完全够用。业务层通过 IM Provider 抽象层调用，后续换 IM 服务商只需新增 Provider
- **COS + STS**：前端直传图片不经过服务器，省带宽。临时密钥 30 分钟过期，安全可控。STS 授权时限制 content-type 白名单，前端上传前校验文件头魔数
- **winston**：Node.js 最成熟的日志库，3 级 logger（access/error/business）覆盖全链路，按天切割（winston-daily-rotate-file），dev=console, prod=file
- **内存令牌桶限流**：MVP 阶段无需 Redis，内存级足够。全局 60req/min + 敏感接口独立 10req/min，防止验证码/举报/发布接口被刷
- **DFA 敏感词过滤**：O(n) 时间复杂度，本地词库无网络依赖，覆盖所有用户输入文本字段

| D7 | 系统 ↔ COS | STS 临时凭证请求（→）、数据库备份文件上传（→） | HTTPS (COS SDK) | ✅ 是 | 备份由 backup.js 定时执行（§11.5.2） |
| D8 | 用户 → COS | 商品图片（前端压缩后 ≤ 500KB/张，1-6 张） | HTTPS (COS JS SDK) | ❌ 否 | 服务端仅发放 STS 临时密钥，图片数据不经过 Express |
| D9 | Express → 腾讯云 IM | 系统消息推送（订单状态变更、举报结果、信誉分变动） | HTTPS (IM REST API) | ✅ 是 | 通过 IM Provider 抽象层调用（§八） |
| D10 | 腾讯云 IM → Express | 消息回调（违规消息通知、用户在线/离线状态） | HTTPS (回调 URL) | ✅ 是 | IM 后台配置回调地址，Express 接收后处理 |

**关键架构决策（从 DFD 中可见）**：

| 决策 | 说明 | 收益 |
|------|------|------|
| **图片直传 COS（D8 绕过 Express）** | 前端拿 STS 临时密钥后直接上传到 COS，不经过 Express | 节省服务器带宽、降低服务端负载、上传速度仅取决于用户网络 |
| **IM 消息不经过 Express（D3/D4）** | 聊天消息通过腾讯云 IM SDK 端到端传递，Express 只通过 REST API 发系统通知（D9） | 消息延迟 < 100ms、服务端零存储成本、避免 Express 成为消息瓶颈 |
| **MySQL 仅在服务端内部（无外部直连）** | 前端永远不直接连接 MySQL，所有数据库操作通过 Express → Repository → mysql2 | 安全：数据库端口不对外暴露；一致：所有写入经过业务层校验 |

## 二、性能基准与 SLA 目标

> 本章定义系统的量化性能目标与可用性承诺。所有架构决策、代码实现与测试用例均以本章数值为基准。目标值基于广州应用科技学院肇庆校区 5,000–15,000 人规模制定，超出此规模时需重新评估并调整目标。

### 2.1 规模假设

性能目标必须建立在真实规模预估之上。以下为基于校区人数的合理推断：

| 指标 | MVP 阶段（前 3 个月） | 稳定运营期（6 个月后） |
|------|----------------------|----------------------|
| 注册用户 | 500–1,500 | 3,000–8,000 |
| 日活 (DAU) | 100–300 | 500–2,000 |
| 峰值 QPS | 30–80 | 100–300 |
| 商品总量 | 200–800 | 2,000–10,000 |
| 日均订单 | 5–20 | 30–100 |

> **推论**：此量级下 MySQL + Node.js 单进程（PM2 fork）完全不是瓶颈。真正的性能瓶颈在小程序端（网络延迟、图片加载、渲染）和 COS 上传速度。

### 2.2 API 响应时间

服务端响应时间是用户体验的基础。以下指标适用于所有 `/api/*` 端点，统计口径为**服务端收到请求首字节到发送响应末字节的完整耗时**，不含客户端网络传输时间。

| 档位 | 目标 | 适用场景 | 不适用 |
|------|------|---------|--------|
| **P50** | < 80ms | 简单查询：商品详情（主键）、用户信息（主键）、分类列表（索引）、健康检查 | — |
| **P95** | < 500ms | 复杂查询：全文搜索 + 筛选 + 排序、订单列表（多表 JOIN）、管理后台看板聚合 | — |
| **P99** | < 1,000ms | 最差情况：冷查询首次加载、大结果集深分页、高峰期连接池排队 | 图片上传（COS 直传不经过服务端）、IM 消息（走腾讯云 SDK） |

**测量方式**：全局中间件在 `req` 上记录 `_startTime = Date.now()`，在 `res.on('finish')` 回调中计算差值，输出到 access log 的 `duration` 字段。生产环境通过 winston access logger 统计 P50/P95/P99 分位数。

**P50 定 80ms 而非 50ms 的理由**：Node.js 单线程事件循环 + MySQL 网络往返（同机 TCP 约 0.2–0.5ms）+ `mysql2` 驱动层 PREPARE + EXECUTE 开销 + JSON 序列化 + 业务逻辑。80ms 是真实可达值；50ms 需要 Redis 缓存命中且无数据库查询——MVP 阶段不做此要求。

**P95 < 500ms 而非 200ms 的理由**：MySQL `FULLTEXT` 中文分词（ngram parser）在数千条数据下通常耗时 100–300ms，加上筛选条件叠加、多表 JOIN、序列化与连接池排队，500ms 是合理上限。全文搜索瓶颈的触发条件为 QPS > 100 或商品 > 10 万，届时触发 ES 迁移（详见 §二十 搜索模块），在此之前 FULLTEXT 充足。

**超过 P99 阈值 1,000ms 的请求**：自动记录慢查询日志（winston warn 级别），包含 SQL 文本、参数、耗时、请求路径。此即 §2.8 慢查询阈值体系的第一级。

### 2.3 小程序端加载时间

小程序端的性能瓶颈通常在网络 IO 与图片解码，而非服务端 CPU。

| 指标 | 目标 | 测量方式 | 备注 |
|------|------|---------|------|
| 首屏冷启动 | < 2s | 微信性能面板 `App.onLaunch` → 首页 `Page.onReady` | 含微信小程序容器初始化 |
| 页面切换（warm） | < 300ms | `wx.navigateTo` 调用 → 目标页 `onReady` | 页面栈内跳转，不含首次进入 |
| 商品列表首屏（含图片） | < 1.5s | 骨架屏展示 → 数据 + 缩略图 `bindload` 完成 | 列表数据 + 首屏可见图片全部加载 |
| 图片懒加载可见延迟 | < 500ms | 图片进入视口（`IntersectionObserver`）→ `bindload` 回调 | 单张缩略图 ≤ 50KB（COS 压缩后） |

**关键优化手段**（对应开发轮次中的实施节点）：

1. **商品图片缩略图**（第 3 轮发布时即实施）：列表与卡片不加载原图，使用 COS 图片处理参数 `?imageMogr2/thumbnail/200x` 压缩为缩略图。原图仅在详情页点击大图时加载。
2. **首页数据客户端缓存**（第 11 轮实施）：`wx.setStorage` 缓存首页商品列表，有效期 5 分钟，减少重复请求。缓存过期后异步更新，不阻塞渲染。
3. **骨架屏 + 分批渲染**（第 3 轮实施）：商品列表使用骨架屏占位，数据返回后分批调用 `setData`，单次 `setData` 数据量 < 256KB（微信限制）。
4. **图片懒加载**（第 3 轮实施）：列表使用 `IntersectionObserver`（微信基础库 ≥ 2.7.0）实现图片懒加载，仅加载可视区域内的图片。

### 2.4 图片上传速度

图片通过 COS 前端直传，不经服务端中转。上传速度主要取决于用户网络质量与 COS 地域（广州）。

| 指标 | 目标 | 条件 |
|------|------|------|
| 单张上传 | < 3s | 4G 网络，图片经前端压缩后 ≤ 500KB，COS 广州区域 |
| 单张上传（Wi-Fi） | < 1s | 校园网 / 同区域 COS |
| 6 张并发上传总耗时 | < 8s | 前端并发控制在 3 路（`Promise.all` concurrency = 3），6 张分两批 |
| 前端压缩耗时（单张） | < 500ms | `uni.compressImage({ quality: 80 })`，压缩后目标 ≤ 200KB |

**并发控制说明**：6 张图片同时上传会占满弱网环境带宽，反而降低总吞吐。前端通过 `ImageUploader` 组件内置并发池（`maxConcurrency = 3`），每完成一张自动从队列取下一张。

**验证方式**：发布前在校园不同位置（教学楼、宿舍楼、食堂）分别使用 4G 与校园 Wi-Fi 做一次完整上传测试，记录实际上传耗时。此测试不计入自动化 CI，为一次性手动验证。

### 2.5 IM 消息延迟

IM 消息链路依赖腾讯云 IM SDK，大部分延迟不在服务端控制范围内。

| 指标 | 目标 | 说明 |
|------|------|------|
| 发送 → 对方收到 | < 1s | 腾讯云 IM SDK 单聊消息正常水平（同区域部署） |
| 敏感词本地过滤 | < 50ms | DFA 算法，文本 ≤ 500 字（详见 §十六 Moderation 模块） |
| 敏感词远程过滤（兜底） | 超时 3s 放行 | 可用性优先——宁可漏过滤不能阻塞消息发送，事后审计兜底 |

> **监控建议**：腾讯云 IM 控制台提供消息到达率与延迟统计面板。若持续超过 3s，优先排查腾讯云 IM 服务状态页，其次检查客户端网络。服务端仅参与 REST API 方式的管理操作（系统消息推送、群组管理），不参与实时消息中转。

### 2.6 可用性（Availability）

校园交易平台非金融级服务，可用性目标在可靠性与成本之间取平衡。

| 指标 | MVP 目标 | 说明 |
|------|---------|------|
| **月度可用率** | **99.5%** | 允许月故障时间 ≤ 3.6 小时（含计划内维护）。按 30 天计，日均允许故障 ≤ 7.2 分钟 |
| 计划内维护窗口 | 凌晨 2:00–5:00 | 尽量安排在寒暑假，提前通过系统消息通知用户 |
| **RTO**（故障恢复时间） | < 30 分钟 | 从发现故障到服务恢复可用的时间上限。操作：PM2 重启 / 数据库恢复 / 代码回滚 |
| **RPO**（数据丢失容忍） | < 24 小时 | 每日凌晨 3 点 `mysqldump` 全量备份至 COS，最坏情况丢失最近 24 小时内数据 |
| 降级模式 | 核心交易链路保持可用 | 非关键功能（搜索热词、Dashboard 统计）允许降级关闭，不影响发布/购买/聊天 |

**99.5% 而非 99.9% 的理由**：99.9% 月故障 ≤ 43 分钟，至少需要主备数据库 + 健康检查自动切换 + 双实例负载均衡。以腾讯云轻量服务器单实例的部署形态，99.9% 无法承诺。校园场景下考试周、寒暑假使用量极低，99.5% 已满足实际需求。服务稳定运营 6 个月后可评估是否升级到 99.9%。

**故障分级**：

| 级别 | 定义 | 响应 | 示例 |
|------|------|------|------|
| P0 — 全站不可用 | 所有用户无法访问 | 立即恢复（目标 < 30min） | 进程崩溃未重启、MySQL 宕机、服务器到期 |
| P1 — 核心链路中断 | 无法发布商品/下单/聊天 | 1 小时内修复 | 腾讯云 IM 欠费停服、COS 密钥过期 |
| P2 — 非核心功能异常 | 搜索慢/部分页面报错 | 下一个工作日修复 | 全文索引失效、定时任务未执行 |
| P3 — 体验降级 | 图片加载慢/页面偶发抖动 | 下一迭代修复 | COS CDN 回源慢、setData 数据量大 |

### 2.7 并发与容量

| 指标 | 目标 | 说明 |
|------|------|------|
| 并发 TCP 连接 | 200 | Node.js event loop + PM2 fork 单进程，轻松支撑 |
| 峰值 QPS | 300 | 超过此值考虑升级为 PM2 cluster 模式（需将定时任务独立为单进程） |
| 数据库连接池利用率 | < 70% | `connectionLimit = 10` 时，活跃连接 ≤ 7，保留 3 个余量应对突发流量 |
| MySQL 最大连接数 | 50 | 服务器 `max_connections` 配置，为运维连接（mysqldump/Workbench）预留空间 |

**容量规划**：300 QPS 对应同时约 100 个活跃用户持续操作（人均 3 QPS）。对于 2,000 DAU 的校园场景，峰值 QPS 通常不超过 100，300 QPS 留有 3 倍余量。

**压力验证**：第 13 轮迭代（测试 + CI + 部署）使用 ApacheBench 对以下接口做基线压测：

```bash

```

#### 商品详情（主键查询）

```bash

```

#### 用户信息（JWT 鉴权 + 主键查询）

```bash

```

压测结果记录 P50/P95/P99 并与 §2.2 目标对比，写入 `docs/benchmark-results.md`。

### 2.8 慢查询阈值

在 `db.js` 的 `query()` 函数内部实现计时，与 MySQL 慢查询日志（`slow_query_log`）互补——服务端计时覆盖网络往返，MySQL 日志只记录执行时间。

| 阈值 | 动作 | 实现方式 |
|------|------|---------|
| > 200ms | 记录慢查询日志（winston warn） | `query()` 内 `Date.now()` 差值，输出 SQL 文本（截断至 500 字符）+ 参数摘要 + 耗时 |
| > 1,000ms | 记录 + 告警（winston error） | 同上，提升至 error 级别以便日志监控 |
| > 3,000ms | 终止查询 + 返回 500 | `pool.execute({ sql, timeout: 3000 })` — mysql2 驱动层超时，抛出 `ERR_DB_QUERY_TIMEOUT`，全局错误中间件统一返回 6999 |

> **注意**：mysql2 的 `timeout` 参数控制的是查询执行超时（`QueryTimeoutError`），非连接池获取超时。连接池获取超时由 `waitForConnections` + `queueLimit` 控制——队列满后新请求立即返回 `PROTOCOL_CONNECTION_LOST`。具体实现在第 11 轮（缓存 + 性能监控）中完成。

> **SQLite → MySQL 类型对照：**
>
> | SQLite | MySQL | 原因 |
> |--------|-------|------|
> | `INTEGER PK AUTOINCREMENT` | `INT PK AUTO_INCREMENT` | 语法差异 |
> | `TEXT` (短字符串) | `VARCHAR(n)` | 有长度约束更严谨 |
> | `TEXT` (长文本) | `TEXT` | 保持一致 |
> | `REAL` | `DECIMAL(10,2)` | 价格用定点数避免浮点误差 |
> | `TEXT DEFAULT '[]'` (JSON) | `JSON` | MySQL 原生 JSON 类型，可建虚拟列索引 |
> | `datetime('now')` | `CURRENT_TIMESTAMP` | MySQL 内置函数 |
> | `INTEGER DEFAULT 1` (布尔) | `TINYINT(1)` | MySQL 习惯用法 |
> | FTS5 虚拟表 | `FULLTEXT INDEX ... WITH PARSER ngram` | MySQL 内置全文索引，支持中文 |

### 4.3 数据库连接配置（db.js 初始化）

```js
const mysql = require('mysql2/promise');

// 连接池（config 从环境变量读取）
const pool = mysql.createPool({
  host: process.env.DB_HOST || '127.0.0.1',
  port: parseInt(process.env.DB_PORT, 10) || 3306,
  user: process.env.DB_USER || 'root',
  password: process.env.DB_PASSWORD || '',
  database: process.env.DB_NAME || 'campus_market_dev',
  charset: 'utf8mb4',
  waitForConnections: true,
  connectionLimit: 10,             // MVP 阶段 10 个连接足够
  queueLimit: 0,                   // 无限排队
  enableKeepAlive: true,
  keepAliveInitialDelay: 10000,    // 10 秒心跳
});

// 从池中获取连接执行查询的便捷函数
async function query(sql, params = []) {
  const [rows] = await pool.execute(sql, params);
  return rows;
}

// 获取事务连接
async function transaction(callback) {
  const conn = await pool.getConnection();
  try {
    await conn.beginTransaction();
    const result = await callback(conn);
    await conn.commit();
    return result;
  } catch (err) {
    await conn.rollback();
    throw err;
  } finally {
    conn.release();
  }
}

module.exports = { pool, query, transaction };
```

> **为什么用连接池而非单连接：** MySQL 不同于 SQLite（文件数据库单连接足够），需要连接池管理并发请求。`mysql2/promise` 提供 async/await 支持，`pool.execute()` 内部调用 `PREPARE + EXECUTE` 实现参数化查询，天然防 SQL 注入。

### 4.6 数据库迁移策略

> **迁移系统定位：** Schema 版本管理工具。每次数据库结构变更通过 `.sql` 文件记录，确保所有环境的 schema 一致、可复现、可回滚。

#### 4.6.1 迁移文件命名规范

```
server/migrations/
├── 001_initial_schema.sql      ← 初始建表（第 1 轮）
├── 002_add_fts.sql             ← MySQL FULLTEXT 全文索引（第 5 轮）
├── 003_add_archive_tables.sql  ← 归档表
├── 004_add_failed_system_messages.sql ← IM 系统消息失败重试表
└── NNN_description.sql         ← 后续迁移（按序号递增追加）
```

| 规则 | 说明 |
|------|------|
| **前缀** | 3 位零填充序号（`001`、`002`……），全局递增不重复 |
| **分隔符** | 下划线 `_` |
| **描述** | 小写英文 + 下划线，简洁描述变更内容 |
| **不可变** | 一旦提交到版本库，**禁止修改**已应用的迁移文件。如需修正，新建一个迁移 |
| **序号分配** | 开发时取当前最大序号 +1 |

#### 4.6.2 迁移文件格式

每个迁移文件就是一个 `.sql` 文件，用注释分隔正向和逆向操作：

```sql
-- ============================================================
-- UP: 正向迁移
-- ============================================================
CREATE TABLE users (
  id            INT PRIMARY KEY AUTO_INCREMENT,
  phone         VARCHAR(11) NOT NULL UNIQUE,
  nickname      VARCHAR(50) NOT NULL DEFAULT '',
  -- ... 其余字段（见 §4.2）
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- ... 其余表

-- ============================================================
-- DOWN: 逆向迁移（按外键依赖顺序反向删除）
-- ============================================================
DROP TABLE IF EXISTS user_events;
DROP TABLE IF EXISTS notifications;
DROP TABLE IF EXISTS report_evidence;
DROP TABLE IF EXISTS product_images;
-- ... 其余表
```

> **为什么用 `.sql` 而非 `.js`：** MySQL 的 DDL 能力完整（`ALTER TABLE ... DROP COLUMN`、`MODIFY COLUMN` 等），不需要 JavaScript 逻辑处理 SQLite 式的重建表变通。纯 SQL 文件更直观，可以直接在 MySQL Workbench / 命令行中执行调试。

#### 4.6.3 迁移运行器（migrate.js）

`server/src/models/migrate.js` 实现一个轻量运行器，核心逻辑：

```
runMigrations(pool, direction = 'up')
  ├── 1. 确保 migrations 表存在（CREATE TABLE IF NOT EXISTS）
  ├── 2. 读取 migrations/ 目录下所有 .sql 文件，按序号排序
  ├── 3. 查询 migrations 表中已应用的版本号集合
  ├── 4. 过滤出待执行的迁移文件
  │     up：  未在 migrations 表中的文件
  │     down：取 migrations 表中最大 version 的那一条记录
  ├── 5. 解析 .sql 文件，提取 UP 或 DOWN 部分的 SQL
  ├── 6. 对每个待执行文件：
  │     ├── 获取连接 → 开启事务
  │     ├── 执行 SQL
  │     ├── INSERT（up）或 DELETE（down）migrations 版本记录
  │     └── 提交事务（失败则回滚 + 打印错误 + 终止）
  └── 7. 输出执行结果摘要
```

关键设计点：

- **文件解析**：用 `-- UP:` 和 `-- DOWN:` 注释标记分割，运行器按需取对应段落。无需正则匹配，直接按标记行切割字符串即可
- **事务包裹**：每个 `.sql` 文件在单独事务内执行，MySQL InnoDB 支持 DDL 事务回滚（与 SQLite 不同，MySQL 的 DDL 在 InnoDB 中是原子的）
- **串行执行**：按序号串行，后续迁移可能依赖前面的表
- **幂等性**：已应用的迁移跳过不执行

```json
    "db:migrate:reset": "node src/models/migrate.js reset",
    "db:seed": "node seeds/dev.js"
  }
}
```

| 命令 | 用途 | 使用场景 |
|------|------|---------|
| `npm run db:migrate` | 应用所有待执行的迁移 | 日常开发、部署上线 |
| `npm run db:migrate:rollback` | 回滚最后一次迁移 | 迁移出错时撤回 |
| `npm run db:migrate:status` | 显示迁移状态（已应用 / 待执行） | 排查环境不一致 |
| `npm run db:migrate:reset` | 回滚全部迁移 → 重新应用 | **仅 dev/staging**，清空重建 |
| `npm run db:seed` | 插入种子数据 | 首次搭建、测试环境重置 |

> **安全防护：** `db:migrate:reset` 通过 `NODE_ENV` 检查——production 环境下拒绝执行并打印警告。

#### 4.6.5 回滚策略

MySQL DDL 在 InnoDB 中支持原子回滚，但回滚决策仍需分场景：

```
回滚决策树：
  迁移在开发环境失败？
    → 修复 .sql 文件中的语法错误
    → npm run db:migrate:rollback 撤回
    → 修复后重新 npm run db:migrate

  迁移在生产环境失败？
    → 停止部署
    → 恢复备份：mysql < backup-<timestamp>.sql
    → 回滚代码到上一版本
    → 分析失败原因，修复后在 staging 验证，再重新部署

  迁移在生产环境成功，但代码有 bug 需要整体回滚？
    → 向前兼容优先（新字段有 DEFAULT 值，旧代码不受影响）
    → 如需回滚 schema：npm run db:migrate:rollback
```

**编写 DOWN 的准则：**

- 新建表 → `DOWN` 中 `DROP TABLE IF EXISTS`
- 新增字段 → `DOWN` 中 `ALTER TABLE ... DROP COLUMN`（MySQL 原生支持）
- 新增索引 → `DOWN` 中 `DROP INDEX ... ON ...`
- 修改字段类型 → `DOWN` 中 `ALTER TABLE ... MODIFY COLUMN ...` 改回原类型
- 如果 DOWN 会导致**不可逆的数据丢失**，在 SQL 注释中显式警告：

```sql
-- ⚠️ DOWNSIDE RISK: DOWN 会删除 data_summary 列，该列数据无法恢复
ALTER TABLE products DROP COLUMN data_summary;
```

#### 4.6.6 MySQL ALTER TABLE 能力

与 SQLite 不同，MySQL 的 `ALTER TABLE` 能力完整，**无需任何变通方案**：

| 操作 | MySQL 语法 |
|------|-----------|
| 添加列 | `ALTER TABLE t ADD COLUMN c VARCHAR(100) DEFAULT ''` |
| 删除列 | `ALTER TABLE t DROP COLUMN c` |
| 修改列类型 | `ALTER TABLE t MODIFY COLUMN c INT NOT NULL` |
| 重命名列 | `ALTER TABLE t RENAME COLUMN old TO new` |
| 添加索引 | `ALTER TABLE t ADD INDEX idx_name (col)` |
| 删除索引 | `ALTER TABLE t DROP INDEX idx_name` |

> `ADD COLUMN` 建议始终带 `DEFAULT` 值，确保旧数据行有合理默认值，也保证向前兼容。

#### 4.6.7 种子数据 vs 迁移

| 维度 | 迁移（Migration） | 种子（Seed） |
|------|------------------|-------------|
| **管理内容** | Schema 结构（表/字段/索引） | 开发测试用数据（用户/商品/订单） |
| **版本追踪** | ✅ `migrations` 表记录 | ❌ 无版本追踪 |
| **执行次数** | 一次（版本号不重复执行） | 多次（幂等，可重复运行） |
| **运行环境** | 所有环境（dev/staging/prod） | 仅 dev/staging |
| **生产行为** | 必须执行（schema 变更） | 禁止执行 |
| **示例** | `CREATE TABLE` / `ALTER TABLE` | `INSERT INTO users VALUES (...)` |

> **反模式：** 不要在迁移 SQL 中插入种子数据。生产环境不需要测试用户和示例商品。系统运行的**必要参考数据**（如默认分类列表）放迁移文件中，用 `INSERT IGNORE INTO` 保证幂等。

#### 4.6.8 测试中的迁移

vitest 测试使用独立 MySQL 测试数据库（`campus_market_test`）：

```js
// __tests__/setup.js (全局测试配置)
const mysql = require('mysql2/promise');
const { runMigrations } = require('../src/models/migrate');

let pool;

beforeAll(async () => {
  // 创建独立测试数据库连接池
  pool = mysql.createPool({
    host: '127.0.0.1',
    user: 'root',
    password: '',
    database: 'campus_market_test',
    charset: 'utf8mb4',
    connectionLimit: 5,
  });
});

beforeEach(async () => {
  // 清空所有表（重置到干净状态）
  await pool.execute('SET FOREIGN_KEY_CHECKS = 0');
  const [tables] = await pool.execute(
    "SELECT TABLE_NAME FROM information_schema.TABLES WHERE TABLE_SCHEMA = 'campus_market_test'"
  );
  for (const { TABLE_NAME } of tables) {
    await pool.execute(`DROP TABLE IF EXISTS \`${TABLE_NAME}\``);
  }
  await pool.execute('SET FOREIGN_KEY_CHECKS = 1');

  // 重新执行全量迁移
  await runMigrations(pool, 'up');
});

afterAll(async () => {
  await pool.end();
});
```

> **为什么不用内存数据库：** MySQL 没有 SQLite 那样的进程内内存实例。独立测试数据库 + `beforeEach` 清空重建是等价方案，全量 SQL 迁移执行快速（约 200ms）。

#### 4.6.9 当前迁移规划（按开发轮次）

| 版本 | 迁移文件 | 轮次 | 变更内容 |
|:---:|------|:---:|------|
| 001 | `001_initial_schema.sql` | 1 | **全量建表：** users, products, orders, reviews, reports, admin_logs, product_images, report_evidence, notifications, user_events 共 10 张业务表 + 全部索引 |
| 002 | `002_add_fts.sql` | 5 | **全文搜索：** 添加 `FULLTEXT INDEX ft_products (title, description) WITH PARSER ngram` |
| 003 | `003_add_archive_tables.sql` | 2 | **归档表：** admin_logs_archive, reviews_archive（长期数据归档） |
| 004 | `004_add_failed_system_messages.sql` | 8 | **IM 系统消息失败重试表：** `failed_system_messages`（§8.6.8） |

#### 4.6.10 生产部署流程

每次部署涉及 schema 变更时的操作步骤：

```
部署前（staging 环境）：
  1. npm run db:migrate          # 先在 staging 执行，验证迁移正确
  2. npm test                     # 全量测试确认迁移不破坏现有功能
  3. npm run db:migrate:rollback  # 验证回滚正常
  4. npm run db:migrate           # 重新应用，准备上线

部署时（production 环境）：
  1. pm2 stop campus-market                          # 停服
  2. mysqldump -u root -p campus_market > backup-$(date +%Y%m%d-%H%M).sql  # 备份
  3. npm run db:migrate                              # 执行迁移
  4. npm run db:migrate:status                       # 验证全部 applied
  5. git pull && npm install --production            # 拉取新代码
  6. pm2 start ecosystem.config.js                   # 启动
  7. curl http://localhost:3000/api/health           # 健康检查

回滚（迁移或代码出问题时）：
  1. pm2 stop campus-market
  2. mysql -u root -p campus_market < backup-<timestamp>.sql   # 恢复备份
  3. git checkout <previous-tag>                               # 回滚代码
  4. npm install --production
  5. pm2 start ecosystem.config.js
```

#### 4.6.11 迁移编写检查清单

每次新建迁移 SQL 文件时确认：

- [ ] 文件名序号正确（当前最大 +1，3 位零填充）
- [ ] 文件内包含 `-- UP:` 和 `-- DOWN:` 两个分隔标记
- [ ] UP 包含所有正向变更（建表/加字段/加索引）
- [ ] DOWN 包含对应的逆向变更（删表/删字段/删索引）
- [ ] DOWN 中如果有不可逆数据丢失，已在文件头部标注
- [ ] 涉及外键时，UP 先建主表再建子表，DOWN 先删子表再删主表
- [ ] 本地执行 `npm run db:migrate` 通过
- [ ] 本地执行 `npm run db:migrate:rollback` + `npm run db:migrate` 通过（回滚再应用）
- [ ] `npm test` 全部通过
- [ ] 迁移中不包含种子数据（参考数据用 `INSERT IGNORE INTO`）

**订单状态流转图：**

```
pending ──┬──→ met ──────→ completed（买家确认 / 3天自动确认）
          │                  
          ├──→ cancelled（任一方取消 → 商品恢复 active）
          │                  
          └──→ timeout（7天未面交自动取消 → 商品恢复 active）

disputed 状态（客服处理中）:
  pending/met ──举报──→ disputed ──客服裁决──→ completed / cancelled
                           │
                           └── 商品同时冻结（frozen），待裁决后恢复/下架
```

```
    message: "ok",

      order_id:            100,
```

**PUT `/api/admin/tickets/:id/resolve`**

> 客服裁决工单。需指定处理结论，可选触发处罚（信誉扣分、封号、下架商品）。裁决后工单关联的 `disputed` 订单根据 `order_action` 流转。

```
Request:
  Body: {
    resolution:     "举报属实，商品已下架，被举报用户扣 30 信誉分",
    actions: {                       // 可选，客服执行的处罚操作
      deduct_credit:  30,             // 扣减被举报人信誉分（可选）
      ban_user:       false,          // 是否封禁被举报人（可选）
      off_shelf_product: true         // 是否下架关联商品（可选）
    },
    order_action:   "cancelled"      // 可选，disputed 订单裁决结果："completed" | "cancelled"
  }

Response (200):
  {
    code: 0,
    message: "ok",
    data: {
      id:              50,
      status:          "resolved",
      resolution:      "举报属实，商品已下架，被举报用户扣 30 信誉分",
      resolved_at:     "2026-06-03T20:00:00.000Z",
      updated_at:      "2026-06-03T20:00:00.000Z",
      side_effects: {                // 实际执行的副作用汇总
        credit_deducted:  30,
        user_banned:      false,
        product_off_shelf: true,
        order_status:     "cancelled"
      }
    }
  }
  // 副作用:
  //   - deduct_credit > 0 → 被举报人信誉分减少，写入 credit 变动记录
  //   - ban_user = true → 被举报人 status=banned, token_version+1
  //   - off_shelf_product = true → 关联商品 status=off_shelf
  //   - order_action 指定 → 关联 disputed 订单恢复为 completed/cancelled，商品解冻
  //   - 通过 IM 系统消息通知举报双方处理结果

可能的错误:
  - 2001: 工单不存在
  - 3002: 工单状态不允许（仅 processing 可 resolve）
  - 5001: 权限不足
```

### 5.4 API 错误码规范

#### 5.4.1 设计原则

| 原则 | 说明 |
|------|------|
| **语义分段** | 千位数表示错误大类，百位数表示子类，一眼能判断错误归属 |
| **一一对应** | 每个错误码对应唯一的 HTTP 状态码和语义场景，不存在复用 |
| **可行动** | 错误消息告诉用户"发生了什么"和"可以做什么"，而非仅技术描述 |
| **分层脱敏** | `detail` 字段仅在 development 环境输出调试信息，production 环境置 `null` |
| **前端可编程** | 前端根据 `code` 做逻辑分流（跳登录 / 弹窗 / Toast），不依赖 `message` 文本匹配 |

#### 5.4.2 错误码分段

```
1xxx — 认证与授权（未登录 / Token 无效 / 被封禁 / 越权）
2xxx — 资源（不存在 / 状态不允许）
3xxx — 业务状态冲突（订单流转 / 工单流转 / 重复操作）
4xxx — 输入与风控（参数校验 / 格式错误 / 限流 / 信誉分）
5xxx — 权限（非 cs / 非 admin / 操作系统保留用户）
6xxx — 系统与第三方（文件上传 / 敏感词 / 微信 / 腾讯云 / 未知错误）
```

#### 5.4.3 完整错误码表

##### 1xxx — 认证与授权

| code | HTTP | message | 触发场景 |
|:----:|:----:|---------|----------|
| `1001` | 401 | 请先登录 | Authorization header 缺失或格式错误 |
| `1002` | 401 | 登录已过期，请重新登录 | Access Token 过期 / 签名无效 / 被篡改 |
| `1003` | 401 | 账号已在其他设备登录，请重新登录 | `token_version` 变更（管理员封禁账号触发版本号+1，所有旧 token 失效） |
| `1004` | 403 | 账号已被限制使用 | `users.status = 'banned'`，用户被封禁 |
| `1005` | 403 | 您没有权限操作该资源 | 非资源所有者尝试修改/删除他人资源 |

> **1002 vs 1003 的区别：** 1002 是 token 自身问题（过期/伪造），前端静默跳转登录页；1003 是被动失效（管理员封禁），前端应弹窗提示"账号已在其他设备登录"再跳转。

##### 2xxx — 资源

| code | HTTP | message | 触发场景 |
|:----:|:----:|---------|----------|
| `2001` | 404 | {资源类型}不存在 | 查询的用户/商品/订单/工单 ID 在数据库中不存在 |
| `2002` | 422 | {资源}状态不允许此操作 | 资源存在但当前状态不匹配操作（如已售商品不可编辑、已完成订单不可取消） |

> **2001 消息模板：** `用户不存在` / `商品不存在或已被删除` / `订单不存在` / `工单不存在`。由调用方传入资源名。

##### 3xxx — 业务状态冲突

| code | HTTP | message | 触发场景 |
|:----:|:----:|---------|----------|
| `3001` | 409 | 订单状态不允许此操作 | 订单状态机非法转换（如从 `completed` 回到 `met`、从 `cancelled` 变为任意状态） |
| `3002` | 409 | 工单状态不允许此操作 | 工单仅 `pending`→`processing`→`resolved`，非法跳转拒绝 |
| `3003` | 200 | （幂等返回） | 同一买家对同一商品重复下单，返回已有订单（不走错误码，`code=0`） |
| `3004` | 409 | 商品已被他人锁定 | 商品 `status` 非 `active`（已被其他买家下单/保留/售出） |
| `3005` | 409 | 不能购买自己发布的商品 | 买家 ID === 卖家 ID |
| `3006` | 409 | 已存在进行中的举报 | 同一用户对同一订单的举报尚未结案 |

##### 4xxx — 输入与风控

| code | HTTP | message | 触发场景 |
|:----:|:----:|---------|----------|
| `4001` | 400 | 请求参数不完整或格式错误 | 必填字段缺失、类型错误、枚举值不在范围内 |
| `4002` | 400 | {字段}格式不正确 | 手机号非 11 位 / 价格负数 / 评分不在 1-5 范围 |
| `4003` | 400 | 图片数量超过限制（最多 6 张） | 上传图片数组长度 > 6 |
| `4004` | 400 | 单张图片大小超过 {n}MB | 单张图片 > 5MB |
| `4005` | 400 | 不支持的图片格式 | 非 image/jpeg, image/png, image/webp |
| `4006` | 429 | 操作过于频繁，请稍后再试 | 全局限流 60 req/min 触发 |
| `4007` | 429 | 操作过于频繁，请 {n} 秒后再试 | 敏感接口限流 10 req/min（登录/发布/举报）触发 |
| `4008` | 403 | 信誉分不足，无法发布商品 | `credit_score < 60` |
| `4009` | 403 | 信誉分不足，无法参与交易 | `credit_score < 30` |

> **4006 vs 4007：** 4006 是全局限流，返回 `Retry-After: 60` header；4007 是敏感接口独立限流，返回 `Retry-After: N`。

##### 5xxx — 权限

| code | HTTP | message | 触发场景 |
|:----:|:----:|---------|----------|
| `5001` | 403 | 需要客服权限 | 非 `cs` 且非 `admin` 用户访问客服工单接口 |
| `5002` | 403 | 需要管理员权限 | 非 `admin` 用户访问后台管理接口 |
| `5003` | 403 | 不可对管理员执行此操作 | 尝试封禁/操作 `role=admin` 的用户 |

##### 6xxx — 系统与第三方

| code | HTTP | message | 触发场景 |
|:----:|:----:|---------|----------|
| `6001` | 502 | 文件上传服务异常，请稍后重试 | COS STS 临时密钥获取失败 / COS 服务不可用 |
| `6002` | 400 | 内容包含违规信息，请修改后重试 | 文本命中敏感词库（不告知具体命中词） |
| `6003` | 502 | 微信服务异常，请稍后重试 | 微信 `getPhoneNumber` / `getAccessToken` 接口超时或返回错误 |
| `6004` | 502 | 消息服务异常，请稍后重试 | 腾讯云 IM REST API 调用失败 |
| `6999` | 500 | 服务器内部错误，请稍后重试 | 未预期的异常（未被 catch 的 Error / MySQL 异常 / 未知错误） |

> **6999 安全原则：** 不暴露任何堆栈信息给客户端。完整错误由 winston 记录到 error 日志，`detail` 仅在 development 环境返回。

```js
// server/src/utils/app-error.js
/**
 * 统一业务异常类。
 * 所有业务逻辑层抛出此异常，由 error-handler 中间件统一转换为 JSON 响应。
 */
class AppError extends Error {
  /**
   * @param {number} code    - 错误码（4 位数字）
   * @param {number} httpStatus - HTTP 状态码
   * @param {string} message - 人类可读错误描述（展示给用户）
   * @param {*}      detail  - 调试信息（仅 development 环境返回）
   */
  constructor(code, httpStatus, message, detail = null) {
    super(message);
    this.name = 'AppError';
    this.code = code;
    this.httpStatus = httpStatus;
    this.detail = detail;
    // 确保 instanceof 正确
    Error.captureStackTrace?.(this, AppError);
  }
}

module.exports = AppError;
```

##### 全局错误处理中间件

```js
// server/src/middleware/error-handler.js
const AppError = require('../utils/app-error');

/**
 * 全局错误处理中间件（4 参数签名）。
 * Express 根据参数个数识别为 error-handling middleware。
 */
function errorHandler(err, req, res, _next) {
  // 1. 已知业务异常 → 结构化返回
  if (err instanceof AppError) {
    return res.status(err.httpStatus).json({
      code: err.code,
      message: err.message,
      detail: process.env.NODE_ENV === 'development' ? err.detail : null,
    });
  }

  // 2. express-validator 校验失败 → 转为 4001
  if (err.type === 'entity.parse.failed' || err.name === 'ValidationError') {
    return res.status(400).json({
      code: 4001,
      message: '请求参数不完整或格式错误',
      detail: process.env.NODE_ENV === 'development' ? err.message : null,
    });
  }

  // 3. 未知异常 → 记录完整日志，返回通用错误
  console.error('[UNHANDLED ERROR]', err);
  // winston error logger 记录完整堆栈
  const logger = require('../utils/logger');
  logger.error({ message: err.message, stack: err.stack, url: req.url });

  return res.status(500).json({
    code: 6999,
    message: '服务器内部错误，请稍后重试',
    detail: process.env.NODE_ENV === 'development' ? err.message : null,
  });
}

module.exports = errorHandler;
```

##### 统一响应拦截器

```js
// miniprogram/api/request.js
const BASE_URL = 'https://your-api.com/api';

function request(url, options = {}) {
  const token = uni.getStorageSync('access_token');

  return new Promise((resolve, reject) => {
    uni.request({
      url: BASE_URL + url,
      header: {
        'Content-Type': 'application/json',
        ...(token ? { Authorization: `Bearer ${token}` } : {}),
      },
      ...options,
      success(res) {
        const { statusCode, data } = res;

        // code=0 → 成功
        if (data.code === 0) {
          return resolve(data.data);
        }

        // ---- 分类处理错误 ----
        const code = data.code;

        // 1001/1002/1003: 认证失效 → 清除 token → 跳登录
        if ([1001, 1002, 1003].includes(code)) {
          uni.removeStorageSync('access_token');
          uni.removeStorageSync('refresh_token');
          uni.reLaunch({ url: '/pages/auth/login' });
          return reject(data);
        }

        // 1004: 账号被封禁 → 弹窗提示后退出
        if (code === 1004) {
          uni.showModal({
            title: '账号已被限制',
            content: data.message,
            showCancel: false,
            success: () => {
              uni.removeStorageSync('access_token');
              uni.reLaunch({ url: '/pages/auth/login' });
            },
          });
          return reject(data);
        }

        // 4001/4002: 参数错误 → Toast 提示
        if ([4001, 4002].includes(code)) {
          uni.showToast({ title: data.message, icon: 'none' });
          return reject(data);
        }

        // 4006/4007: 限流 → Toast 提示
        if ([4006, 4007].includes(code)) {
          uni.showToast({ title: data.message, icon: 'none', duration: 2500 });
          return reject(data);
        }

        // 6002: 敏感词 → 弹窗提示修改
        if (code === 6002) {
          uni.showModal({
            title: '内容审核不通过',
            content: data.message,
            showCancel: false,
          });
          return reject(data);
        }

        // 其他错误 → 通用 Toast
        uni.showToast({ title: data.message || '请求失败', icon: 'none' });
        reject(data);
      },
      fail(err) {
        // 网络异常（小程序断开网络 / 后端未启动）
        uni.showToast({ title: '网络异常，请检查网络连接', icon: 'none' });
        reject(err);
      },
    });
  });
}

module.exports = { get: (url, data) => request(url, { method: 'GET', data }),
                    post: (url, data) => request(url, { method: 'POST', data }),
                    put: (url, data) => request(url, { method: 'PUT', data }),
                    del: (url, data) => request(url, { method: 'DELETE', data }) };
```

#### 5.4.8 新增错误码流程

当代码开发发现需要新的错误码时，按以下流程操作：

1. 判断属于哪个千位段
2. 在该段内取下一个未使用的百位数
3. 更新本节的错误码表
4. 在 `errors.js` 中添加工厂函数
5. 在测试用例中覆盖新的错误场景

> **约束：** 不修改已有错误码的语义。如现有错误码不满足需求，新增一个而不修改旧的。

```
  │                             │←──── 点击"我想要" ─────────┤
  │                             ├─ 创建订单(status=pending)    │
  │                             ├─ 商品 status=reserved        │
  │                             ├─ 返回订单ID                  │
  │                             │                             │
  │←── IM聊天(腾讯云IM SDK) ───│─── IM聊天 ─────────────────→│
  │        约定面交时间地点       │                             │
  │                             │                             │
  │  ←─── 线下面交，一手交钱一手交货 ────────────────────────→│
  │                             │                             │
  │                             │←──── 点击"已面交" ─────────┤
  │                             ├─ 订单 status=met            │
  │                             │                             │
  │                             │←──── 点击"确认收货" ────────┤  (或3天自动)
  │                             ├─ 订单 status=completed       │
  │                             ├─ 触发互评入口                │
  │                             │                             │
  │←── 提交评价 ───────────────│──── 提交评价 ──────────────→│
  │                             ├─ 双方评价入库                │
```

### 7.2 3 天超时自动确认

```
定时任务（每 30 分钟执行一次，fork 模式单进程安全）:
  1. 订单超时自动确认:
     查询 status=met AND met_at < now - 3天 的订单
     → 自动更新 status=completed, confirmed_at=now
     → 商品 status=sold
     → 触发双方互评入口

  2. pending 订单超时释放（如 7 天未面交）:
     查询 status=pending AND created_at < now - 7天 的订单
     → 自动更新 status=cancelled
     → 商品 status 从 reserved 恢复为 active
```

> 用 `node-cron` 在 Express 进程内跑定时任务。PM2 必须用 fork 模式（单进程），否则 cluster 会导致重复执行。

## 八、即时通讯集成方案

### 8.1 架构

```
小程序前端 ──TIM SDK──→ 腾讯云 IM 服务器（消息收发/存储/推送）
  │
  ├── 第一次连接前：请求服务端 /api/auth/me → 返回 IM UserSig
  │
  └── 系统消息：服务端通过 IM REST API 推送订单/举报通知
                │
        server/src/services/im/
        ├── provider.js       ← IM Provider 接口定义
        └── tencent.js        ← 腾讯云 IM 实现（SDK + REST API）
```

### 8.2 关键实现要点

- **用户 ID 映射**：后端注册用户时，用数据库 `user.id` 作为 IM 的 `UserID`
- **UserSig 生成**：后端 `utils/im.js` 用腾讯云 IM 应用密钥生成，**HMAC-SHA256 签名**，有效期 7 天
- **前端初始化**：`utils/im.js` 封装 TIM SDK 初始化、登录、创建会话、敏感词过滤 hook、系统消息监听
- **聊天入口**：商品详情页"聊一聊"按钮 → 调用 `createConversation(sellerId)` → 跳转 `chat/detail`
- **免责提示**：聊天页顶部固定 Banner —— "如私下交换微信/其他联系方式交易，发生纠纷平台不负责"
- **IM Provider 抽象**：业务层通过 `provider.js` 定义的接口调用（`sendSystemMessage`, `createConversation`, `getUserSig`），当前实现 `tencent.js` 封装腾讯云 IM。后续换 IM 服务商只需新增 Provider 实现

### 8.3 为什么不在服务端中转 IM

腾讯云 IM SDK 已经处理了：消息存储（7 天免费）、离线推送、图片/视频消息、已读回执。服务端中转没有任何好处，反而增加服务器负载。

### 8.4 系统消息通道

利用腾讯云 IM REST API 由服务端推送系统消息（TIMCustomElem）：

| 消息类型 | 触发场景 | 消息内容 |
|---------|---------|---------|
| 订单状态变更 | pending→met / met→completed / cancelled | "你的订单 #xxx 状态已更新为：已面交" |
| 评价提醒 | 订单确认收货后 | "请对本次交易进行评价" |
| 举报结果 | 客服处理举报后 | "你的举报 #xxx 已被处理：xxx" |
| 信誉分变动 | 扣分/加分 | "你的信誉分变更为：70（-30 举报属实）" |

前端 IM SDK 监听 `MESSAGE_RECEIVED` 事件，识别 `TIMCustomElem` 类型渲染系统消息卡片。

### 8.5 聊天监管

- **敏感词过滤**：消息发送前在 IM SDK hook 中调用敏感词过滤器（复用 `sensitive-filter.js` DFA 算法），命中敏感词时消息不发送并前端提示
- **举报入口**：IM 会话详情页增加"举报此用户"按钮，举报表单支持选择"聊天记录"作为证据类型
- **举报冻结**：举报类型为"疑似骗子"时，被举报用户不可删除相关 IM 会话（前端隐藏删除按钮），举报 resolved 后恢复

### 8.6 消息协议详细设计

#### 8.6.1 消息分层

```
┌─────────────────────────────────────────┐
│  用户消息（C2C）                          │
│  ├── 文本消息  TIMTextElem               │
│  ├── 图片消息  TIMImageElem              │
│  └── 自定义消息 TIMCustomElem（前端互发）  │
├─────────────────────────────────────────┤
│  系统消息（Server → C2C）                 │
│  └── 自定义消息 TIMCustomElem             │
│      └── data.ext === 'system'           │
└─────────────────────────────────────────┘
```

- **用户消息**：走腾讯云 IM SDK 默认通道，不经过服务端
- **系统消息**：服务端调用 IM REST API 发送 `TIMCustomElem`，前端通过 `ext` 字段识别

#### 8.6.2 系统消息通用信封

所有系统消息的 `TIMCustomElem` 均使用以下 JSON 结构：

```jsonc
{
  "ext": "system",                    // 前端通过此字段区分系统消息 vs 用户自定义消息
  "sound": "system_notification.aac", // 系统消息专属提示音
  "description": "订单状态更新",       // 离线推送摘要（锁屏/通知栏显示）
  "data": {
    "type": "order_update",           // 系统消息子类型（见 §8.6.3）
    "version": "1.0",                 // 协议版本，用于前端兼容降级
    "messageId": "sys_xxxxxxxxxxxx",  // 全局唯一 ID（服务端生成，UUID v4）
    "createdAt": "2026-06-03T10:30:00Z", // ISO 8601，服务端时间
    "payload": { /* 各类型专属数据 */ }
  }
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|:--:|------|
| `ext` | string | ✅ | 固定值 `"system"` |
| `sound` | string | ✅ | 系统消息提示音文件名 |
| `description` | string | ✅ | 离线推送摘要，≤50 字符 |
| `data.type` | string | ✅ | 子类型，见 §8.6.3 |
| `data.version` | string | ✅ | 协议版本，当前 `"1.0"` |
| `data.messageId` | string | ✅ | UUID v4，服务端生成 |
| `data.createdAt` | string | ✅ | ISO 8601 时间戳 |
| `data.payload` | object | ✅ | 各子类型专属数据 |

#### 8.6.3 系统消息子类型

##### 类型 1：`order_update` — 订单状态变更

**触发时机**：订单状态发生变更时（pending→met / met→completed / 任一方取消 / 超时自动取消 / 客服裁决取消）

**发送对象**：买卖双方均推送

```jsonc
{
  "type": "order_update",
  "payload": {
    "orderId": 42,
    "status": "met",                    // 新状态
    "previousStatus": "pending",        // 原状态
    "triggerBy": 1003,                  // 触发者 userId（系统触发为 0）
    "triggerRole": "buyer",             // buyer | seller | system | admin
    "productName": "《算法导论》第三版",
    "productImage": "https://cos.example.com/thumb.jpg",
    // 以下字段仅在 status=cancelled 时出现
    "cancelReason": "买家主动取消",
    // 以下字段仅在 status=completed 时出现
    "completedAt": "2026-06-03T11:00:00Z"
  }
}
```

| payload 字段 | 类型 | 必填 | 说明 |
|-------------|------|:--:|------|
| `orderId` | number | ✅ | 订单 ID |
| `status` | string | ✅ | 新状态：`met` / `completed` / `cancelled` / `timeout` / `disputed` |
| `previousStatus` | string | ✅ | 变更前状态 |
| `triggerBy` | number | ✅ | 触发操作的用户 ID，系统自动触发为 `0` |
| `triggerRole` | string | ✅ | `buyer` / `seller` / `system` / `admin` |
| `productName` | string | ✅ | 商品名称（≤100 字符） |
| `productImage` | string | ✅ | 商品封面缩略图 URL |
| `cancelReason` | string | ❌ | 取消原因（status=cancelled 时必填）。运行时透传，不持久化至 orders 表（orders 表仅记录 `cancelled_by` 枚举） |
| `completedAt` | string | ❌ | 完成时间（status=completed 时必填），对应 orders 表 `confirmed_at` 列 |

**前端渲染**：
- 卡片样式：左侧商品缩略图，右侧状态文案 + 时间
- 点击卡片：跳转对应订单详情页 `pages/order/detail?id=42`
- status 文案映射：`met→"双方已约定面交"`、`completed→"交易已完成"`、`cancelled→"订单已取消"`、`timeout→"订单超时自动取消"`、`disputed→"订单已被客服取消"`

##### 类型 2：`review_remind` — 评价提醒

**触发时机**：订单确认收货后立即推送；超时自动确认后推送

**发送对象**：买卖双方均推送

```jsonc
{
  "type": "review_remind",
  "payload": {
    "orderId": 42,
    "targetUserId": 1003,               // 需要评价的对方用户 ID
    "targetNickname": "张三",
    "productName": "《算法导论》第三版",
    "deadline": "2026-06-06T11:00:00Z", // 评价截止时间（确认后 3 天）
    "hasReviewed": false                // 当前用户是否已评价
  }
}
```

| payload 字段 | 类型 | 必填 | 说明 |
|-------------|------|:--:|------|
| `orderId` | number | ✅ | 关联订单 ID |
| `targetUserId` | number | ✅ | 待评价的对方用户 ID |
| `targetNickname` | string | ✅ | 对方用户昵称 |
| `productName` | string | ✅ | 商品名称 |
| `deadline` | string | ✅ | 评价截止时间，ISO 8601 |
| `hasReviewed` | boolean | ✅ | 当前用户是否已完成评价 |

**前端渲染**：
- 卡片样式：提示文案 + "去评价"按钮
- "去评价"按钮：跳转订单详情页 `pages/order/detail?id=42`（订单详情页内含评价入口，目标用户由订单数据确定）
- 如果 `hasReviewed=true`：按钮置灰显示"已评价"

##### 类型 3：`report_result` — 举报处理结果

**触发时机**：客服 resolve 工单后

**发送对象**：举报人 + 被举报人

```jsonc
{
  "type": "report_result",
  "payload": {
    "reportId": 15,
    "resolution": "举报属实，商品已下架",   // 客服填写的处理结论
    "result": "upheld",                   // upheld（举报成立）| dismissed（举报不成立）
    "actions": {
      "deductCredit": 30,                 // 扣除信誉分（0 表示未扣分）
      "offShelfProduct": true,            // 是否下架商品
      "cancelOrder": true                 // 是否取消关联订单
    },
    "relatedProductName": "假冒 AirPods Pro",
    "resolvedAt": "2026-06-03T14:00:00Z"
  }
}
```

| payload 字段 | 类型 | 必填 | 说明 |
|-------------|------|:--:|------|
| `reportId` | number | ✅ | 举报/工单 ID |
| `resolution` | string | ✅ | 客服处理结论，≤200 字符 |
| `result` | string | ✅ | `upheld`（成立）/ `dismissed`（不成立） |
| `actions.deductCredit` | number | ✅ | 扣除信誉分，0 表示未扣分 |
| `actions.offShelfProduct` | boolean | ✅ | 是否强制下架关联商品 |
| `actions.cancelOrder` | boolean | ✅ | 是否取消关联订单 |
| `relatedProductName` | string | ❌ | 关联商品名称（有商品关联时必填） |
| `resolvedAt` | string | ✅ | 处理完成时间 |

**前端渲染**：
- 卡片样式：结果图标（✅成立 / ⭕不成立）+ 处理结论摘要
- 点击卡片：跳转工单详情页 `pages/report/detail?id=15`
- **举报成立时**：文案附带 `actions` 说明（如"已扣除 30 信誉分，商品已下架"）

##### 类型 4：`credit_change` — 信誉分变动

**触发时机**：信誉分发生任何变动时

**发送对象**：仅变动用户本人

```jsonc
{
  "type": "credit_change",
  "payload": {
    "previousScore": 100,
    "currentScore": 70,
    "change": -30,                        // 变动值（正数为加分，负数为扣分）
    "reason": "举报属实，商品与描述严重不符",
    "reasonCode": "report_upheld",        // 机器可读原因码
    "relatedReportId": 15,                // 关联举报 ID（扣分场景）
    "changeType": "deduct"                // deduct（扣分）| reward（加分）| init（初始化）
  }
}
```

| payload 字段 | 类型 | 必填 | 说明 |
|-------------|------|:--:|------|
| `previousScore` | number | ✅ | 变动前信誉分 |
| `currentScore` | number | ✅ | 变动后信誉分 |
| `change` | number | ✅ | 变动值，正数加分，负数扣分 |
| `reason` | string | ✅ | 变动原因，人类可读 |
| `reasonCode` | string | ✅ | 机器可读原因码（见下方枚举） |
| `relatedReportId` | number | ❌ | 关联举报 ID，扣分场景提供 |
| `changeType` | string | ✅ | `deduct` / `reward` / `init` |

**reasonCode 枚举**：

| reasonCode | 说明 | change 值 |
|-----------|------|:--:|
| `report_upheld` | 举报成立 | -30 |
| `order_completed` | 完成交易（卖家） | +2 |
| `positive_review` | 收到好评 | +1 |
| `negative_review` | 获得差评 | -5 |
| `admin_manual` | 管理员手动调整 | 自定义 |
| `initial` | 新用户初始分 | +100 |

**前端渲染**：
- 卡片样式：变动数值（红色扣分 / 绿色加分）+ 原因说明
- 低信誉分预警：当 `currentScore < 60` 时，追加提示"你的信誉分低于 60，已限制发布商品"
- 点击卡片：跳转信誉分详情页 `pages/user/credit`

#### 8.6.4 敏感词过滤 Hook 协议

前端 IM SDK 发送消息前，通过 `TIM.EVENT.MESSAGE_SENDING` hook 拦截：

```
用户输入 → hook 拦截 → POST /api/sensitive/check { text } →
  ├── { ok: true }  → 继续发送（TIM SDK 正常流程）
  └── { ok: false, hit: ["xxx"], suggestion: "..." } → 阻断发送，Toast 提示
```

| 场景 | 前端行为 |
|------|---------|
| 命中敏感词 | `uni.showToast({ title: '消息包含敏感内容，请修改后重发', icon: 'none' })`，消息不入 IM |
| 接口超时（3s） | 放行发送（可用性优先，事后审计兜底） |
| 接口不可达 | 放行发送 + 上报异常日志 |

#### 8.6.5 消息发送失败重试策略

| 层级 | 策略 | 说明 |
|------|------|------|
| **用户消息** | TIM SDK 自动重试 | SDK 内置指数退避重试（2s/4s/8s，最多 3 次），前端无感知 |
| **系统消息（服务端）** | 手动重试 | 调用 IM REST API 失败后，记录到 `failed_system_messages` 表，定时任务按 `FAILED_MSG_RETRY_INTERVAL_MIN` 间隔重试（默认 5 分钟），最多 5 次后标记为 `permanent_failed` 并告警 |
| **文件消息（图片）** | 先上传 COS 再发 IM | COS 上传失败不发送 IM 消息，前端提示用户重试 |

#### 8.6.6 离线消息与已读

- **离线消息**：腾讯云 IM 免费档保留 7 天离线消息，用户重新打开小程序时 TIM SDK 自动拉取
- **系统消息离线推送**：`description` 字段用于微信服务通知的摘要文本，用户可在微信"服务通知"中看到
- **未读计数**：前端通过 `TIM.EVENT.CONVERSATION_LIST_UPDATED` 实时更新；服务端 `GET /api/notifications/unread-count` 作为补充（统计 IM 系统消息 + 站内通知）

#### 8.6.7 消息存储与合规

| 项目 | 说明 |
|------|------|
| 存储位置 | 腾讯云 IM 云端（7 天免费）+ 小程序本地 SQLite（TIM SDK 自动缓存） |
| 服务端留存 | **不做**全量消息留存。如需纠纷取证，客服通过腾讯云 IM 控制台调取 |
| 撤回 | 用户消息支持 2 分钟内撤回（TIM SDK 原生能力） |
| 删除会话 | 举报冻结期间，被举报方不可删除相关会话（前端隐藏按钮，服务端不介入） |

#### 8.6.8 失败系统消息表

```sql
CREATE TABLE IF NOT EXISTS failed_system_messages (
  id            INT       NOT NULL AUTO_INCREMENT,
  message_type  VARCHAR(32) NOT NULL,              -- order_update | review_remind | report_result | credit_change
  target_uid    VARCHAR(64) NOT NULL,              -- 目标 IM UserID（对应 users.id）
  payload       JSON       NOT NULL,               -- JSON，完整消息体
  retry_count   INT       NOT NULL DEFAULT 0,
  max_retries   INT       NOT NULL DEFAULT 5,
  last_error    VARCHAR(512),                      -- 最近一次失败原因
  status        VARCHAR(20) NOT NULL DEFAULT 'pending', -- pending | retrying | permanent_failed | sent
  created_at    DATETIME   NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updated_at    DATETIME   NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  PRIMARY KEY (id),
  INDEX idx_fsm_status (status, created_at),
  INDEX idx_fsm_target (target_uid, status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4
  COMMENT='IM 系统消息失败重试表';
```

## 九、文件上传方案

### 9.1 上传流程

```
前端                         后端                     COS
 │                            │                       │
 ├─ GET /api/upload/cos-credential ──→│               │
 │                            ├─ 调用 STS 获取临时密钥  │
 │                            ├─ 返回 credentials      │
 │←── { tmpSecretId, bucket, region, prefix } ──┤      │
 │                            │                       │
 ├─ uni.uploadFile ──────────────────────────────────→│
 │   + COS SDK 计算签名                               │
 │←── 返回图片 URL ──────────────────────────────────┤
 │                            │                       │
 ├─ POST /api/products { images: ["url1","url2"] } →│
 │                            │                       │
```

### 9.2 安全限制

- 允许格式：`image/jpeg`，`image/png`，`image/webp`（**禁止 GIF**，防止动图滥用）
- 单张最大：5MB
- 单次上传：最多 6 张
- STS 临时密钥有效期：30 分钟，限制 `content-type` 白名单
- 上传路径：`products/` + 随机文件名（防遍历）
- **前端校验**：上传前用 `uni.getImageInfo()` 校验格式（宽高 > 0）+ 文件头魔数校验
- **STS 限制**：COS STS 授权时指定 `condition` 限制 `content-type` 仅允许 image/png, image/jpeg, image/webp
- **EXIF 清理**：前端读取图片后 strip EXIF 数据再上传（防止 GPS/设备信息泄露）
- **服务端回调**：COS 上传回调到 `/api/upload/callback`，服务端二次校验文件头魔数
- **上传频控**：同一用户 1 分钟内最多上传 10 张图片

## 十、安全设计

### 10.1 认证与鉴权

| 机制 | 实现 |
|------|------|
| 登录 | 微信手机号授权（`getPhoneNumber`）。前端获取 code → 后端调微信 `phonenumber.getPhoneNumber` API 解密手机号 → 查找或创建用户 → 签发 JWT。注册与登录合一，无需密码和验证码 |
| 双 Token | access_token（JWT，7 天有效）用于业务鉴权；refresh_token（JWT，30 天有效）用于无感续期。登录时同时返回两个 token。access_token 过期后前端用 refresh_token 调 `/api/auth/refresh` 换取新的 access_token，用户无需重新授权手机号。refresh_token 也含 `token_version`，封禁时两个 token 同时失效 |
| 鉴权 | `middleware/auth.js` 解析 access_token，校验 `payload.version === user.token_version`，注入 `req.user` |
| Token 刷新 | `POST /api/auth/refresh` 接收 refresh_token，验证通过后颁发新 access_token（不颁发新的 refresh_token，refresh_token 到期后需重新授权登录） |
| 封禁即时生效 | auth 中间件解析 JWT 后，额外查询 `users.status` 和 `users.token_version`。若 `status=banned` 则返回 1004；**管理员封禁用户时 `token_version += 1`**，该用户所有旧 token 立即失效（返回 1003） |
| 客服权限 | `middleware/cs.js` 检查 role 为 `cs` 或 `admin`；客服可查看举报列表、处理工单（标记处理中/裁决），但**不可**封号、看数据看板、操作用户管理 |
| 管理员权限 | `middleware/admin.js` 检查 role 为 `admin`；拥有全部后台权限 |
| 角色权限矩阵 | 见下表 |
| 微信 access_token | `utils/wechat.js` 管理微信 access_token（内存缓存，提前 5 分钟刷新），用于手机号解密等微信 API 调用 |

**角色权限矩阵：**

| 操作 | user | cs（客服） | admin |
|------|:----:|:---------:|:-----:|
| 浏览/搜索商品 | ✓ | ✓ | ✓ |
| 发布/编辑/删除自己的商品 | ✓ (≥60分) | ✓ | ✓ |
| 发起交易 | ✓ (≥30分) | ✓ | ✓ |
| 订单操作（面交/确认/取消） | ✓ (本人) | ✓ | ✓ |
| 提交评价 | ✓ (本人) | ✓ | ✓ |
| 提交举报 | ✓ | ✓ | ✓ |
| 查看通知/信誉分 | ✓ (本人) | ✓ | ✓ |
| 编辑个人资料 | ✓ (本人) | ✓ | ✓ |
| 查看举报列表 (GET /api/reports) | ✗ | ✓ | ✓ |
| 处理工单 (process/resolve) | ✗ | ✓ | ✓ |
| 封禁/解封用户 | ✗ | ✗ | ✓ |
| 强制下架商品 | ✗ | ✗ | ✓ |
| 查看用户列表 | ✗ | ✗ | ✓ |
| 查看数据看板/统计 | ✗ | ✗ | ✓ |
| API 限流 | `middleware/rate-limiter.js`：全局 60 req/min，敏感接口（登录/举报/发布）独立 10 req/min，超限返回 429 + 冷却时间 |
| 信誉分限制 | auth 中间件校验 `user.credit_score`：< 60 禁止发布商品，< 30 禁止发起交易，返回 4008/4009 |

### 10.2 防护措施

| 威胁 | 防护 |
|------|------|
| 手机号伪造 | 手机号由微信 `getPhoneNumber` API 返回，经微信服务端签名加密，前端无法篡改。后端通过微信官方 API 解密获取真实手机号 |
| 登录接口爆破 | 同一 IP 1 分钟内最多 10 次登录请求；code 由微信生成且一次性有效，无重放风险 |
| JWT 泄露 | access_token 7 天过期，refresh_token 30 天过期；封禁时 `token_version += 1` 使所有已签发 token 立即失效（含 refresh_token）；前端检测到 1004/1005 时跳转登录页 |
| SQL 注入 | mysql2 `pool.execute(sql, params)` 内部使用 PREPARE + EXECUTE 参数化查询，天然防注入 |
| XSS | 所有用户输入的文本字段（昵称、商品描述、评价内容）存储前 HTML 实体编码，渲染时 uni-app 自动转义 |
| 越权操作 | 每个 API 在 controller 层校验资源归属（如：只能编辑自己的商品） |
| 敏感词 | 所有用户输入文本（标题/描述/昵称/评论/举报内容）经 `sensitive-filter.js` DFA 过滤，命中敏感词返回 6002 拒绝提交 |
| 图片上传 | 前端校验格式（魔数）+ 大小（≤5MB）+ 宽高（`uni.getImageInfo()`）；COS STS 限制 content-type 白名单（image/png, image/jpeg, image/webp）；禁止 GIF；服务端回调后校验文件头魔数 |
| 恶意文件 | 上传前校验文件头魔数（PNG: 89 50 4E 47 / JPEG: FF D8 FF / WebP: 52 49 46 46）；EXIF 数据上传前由客户端 strip |
| 接口限流 | 全局 60 req/min + 敏感接口独立 10 req/min，超限返回 429 + Retry-After 头 |
| 重复提交 | 订单创建使用 `idempotent_key` 唯一约束，重复点击"我想要"返回已有订单 |

### 10.3 管理员操作审计

所有敏感后台操作写入 `admin_logs` 表，事后可追溯：

| 操作 | 记录内容 |
|------|----------|
| 封禁/解封用户 | 操作人、目标用户、时间、原因 |
| 商品强制下架 | 操作人、目标商品、原因、时间 |
| 工单裁决 | 操作人、目标工单、裁决结果（含扣分/封号/下架等副作用）、时间 |

> admin_logs 表结构见数据库设计章节。用于出问题时追溯"谁在什么时候做了什么"。

### 11.4 定时任务方案

```
MVP 阶段（当前）：
  PM2 fork 模式（单进程）+ node-cron 在 Express 进程内跑定时任务

演进方案（多实例部署时）：
  抽取 scheduler.js 为独立进程，通过环境变量 ENABLE_CRON=true 控制
  只有主实例执行定时任务。分布式锁预留 Redis 方案
```

定时任务清单：

| 任务 | 频率 | 说明 |
|------|------|------|
| 订单超时自动确认 | 每 30 分钟 | status=met 且 met_at > 3 天 → completed |
| pending 超时取消 | 每 30 分钟 | status=pending 且 created_at > 7 天 → timeout |
| 数据库日备份（Son） | 每天凌晨 3 点 | mysqldump → COS `backups/daily/campus_market-${date}.sql.gz`（§11.5.1） |
| 数据库周备份（Father） | 每周日 03:00 | 同日备份额外存至 `backups/weekly/`，保留 30 天（§11.5.1） |
| 数据库月备份（Grandfather） | 每月 1 日 03:00 | 同日备份额外存至 `backups/monthly/`，保留 3 个月（§11.5.1） |
| 迁移前备份 | 每次执行 migration 前 | 调用 `backup.js --type pre-migrate`，存至 `backups/pre-migrate/`，保留 1 年（§11.5.4） |
| 备份恢复验证 | 每周一凌晨 4 点 | 下载最新日备份 → 导入临时库 → 基础查询校验 + 核心表抽样（§11.5.6） |
| 数据归档 | 每月 1 号 | admin_logs(>6月) → admin_logs_archive，reviews(>1年) → reviews_archive |
| 通知自动清理 | 每天凌晨 4 点 | `DELETE FROM notifications WHERE created_at < DATE_SUB(NOW(), INTERVAL 30 DAY)`（保留 30 天，见 §十五 通知中心） |
| 举报/工单超期清理 | 每月 1 号 | reports 及关联 report_evidence 创建 > 2 年 → 软删除（`deleted_at = NOW()`），admin_logs 中关联记录同步归档至 admin_logs_archive |
| 日志自动轮转 | winston-daily-rotate-file 内置 | `maxFiles: '90d'` — 自动删除超过 90 天的日志文件，无需额外 cron |
| IM 系统消息失败重试 | 按 `FAILED_MSG_RETRY_INTERVAL_MIN`（默认 5 分钟） | `failed_system_messages` 表中 `status='retrying'` 且 `retry_count < maxRetries` 的记录重试发送，超过上限后标记 `permanent_failed` 并告警 |

> **隐私政策保留期对照：** 上表覆盖了隐私政策 §4.2 承诺的主要保留期限。交易记录 5 年保留（数据量极低，MVP 阶段不做自动清理，留存至服务器存储上限后手动处理）；账号注销 30 天删除由注销流程手动触发（非 cron）。详见《[隐私政策](legal/隐私政策.md) §4.2》。

### 11.5 备份与恢复

> 本章节是数据安全的最后防线。以下内容覆盖备份分层、执行脚本设计、安全加密、COS 生命周期自动清理、恢复决策树（3 种故障场景）、恢复验证、以及中期 RPO 改善路线。

#### 11.5.1 备份分层策略（GFS 保留模型）

单一"每天 1 次 + 保留 7 天"不足以覆盖所有风险：如果某个逻辑错误（如误删商品）在第 8 天才被发现，7 天保留无济于事。采用简化的 GFS（Grandfather-Father-Son）分层模型，在存储成本几乎为零的前提下拉长保护窗口：

| 层级 | 备份频率 | 保留期限 | COS 路径 | 用途 |
|------|---------|---------|---------|------|
| **日备份（Son）** | 每天 03:00 | **7 天** | `backups/daily/` | 昨天误操作的快速回滚（最常见场景） |
| **周备份（Father）** | 每周日 03:00 | **30 天** | `backups/weekly/` | 发现较晚的逻辑错误（如一周前上架了违规商品未被及时发现） |
| **月备份（Grandfather）** | 每月 1 日 03:00 | **3 个月** | `backups/monthly/` | 合规审计最低追溯期。若涉及交易纠纷（如 2 个月前的一笔争议订单），有据可查 |
| **迁移前备份** | 每次执行 `npm run db:migrate` 前 | **1 年** | `backups/pre-migrate/` | migration 引入的 schema 变更（如误删列）可通过此备份恢复原始表结构与数据 |

**存储成本估算**：假设数据库 50 MB（校园量级），gzip 压缩后约 10 MB/份。日 7 + 周 4 + 月 3 = 14 份 × 10 MB = **140 MB**。腾讯云 COS 标准存储 ¥0.118/GB/月，月费不到 **¥0.02**。即使数据增长到 200 MB，月费仍不足 ¥0.10。**结论：多保留比少保留明智，成本可忽略不计。**

**周备份与月备份的实现**：`backup.js` 在执行日备份时，额外检查当天是否为周日（`new Date().getDay() === 0`）或每月 1 日（`new Date().getDate() === 1`），若是则将该份备份复制到对应的 `weekly/` 或 `monthly/` 目录。不额外执行一次 `mysqldump`——同一份文件，多个目录。

#### 11.5.2 备份执行（backup.js 设计）

`backup.js` 不是简单的 shell 一行封装，它有明确的职责边界和错误处理：

```
backup.js 执行流程（共 10 步）：

  ┌─────────────────────────────────────────────┐
  │ 1. 检查 MySQL 连接 (SELECT 1)              │
  │    → 失败：记录 error log，exit(1)          │
  ├─────────────────────────────────────────────┤
  │ 2. 检查 COS 可达性 (headBucket)            │
  │    → 失败：记录 error log，exit(1)          │
  ├─────────────────────────────────────────────┤
  │ 3. 确定目标目录                             │
  │    • 每日：backups/daily/                   │
  │    • 周日额外 → backups/weekly/             │
  │    • 1日额外 → backups/monthly/             │
  │    • --type pre-migrate → pre-migrate/      │
  ├─────────────────────────────────────────────┤
  │ 4. 执行 mysqldump                           │
  │    --single-transaction --routines          │
  │    --triggers --no-tablespaces              │
  │    --skip-lock-tables                       │
  │    | gzip > campus_market-${date}.sql.gz    │
  │    → 失败：记录 error log，exit(1)          │
  ├─────────────────────────────────────────────┤
  │ 5. 计算 .sql.gz 的 MD5 校验值               │
  │    → 写入 campus_market-${date}.sql.gz.md5  │
  ├─────────────────────────────────────────────┤
  │ 6. 上传两个文件至 COS                       │
  │    • .sql.gz（带 Content-MD5 header）       │
  │    • .sql.gz.md5（供恢复时校验完整性）      │
  ├─────────────────────────────────────────────┤
  │ 7. 验证上传：headObject 获取 Content-Length │
  │    对比本地文件大小 → 不一致则重试一次      │
  │    → 两次均失败：记录 error log，exit(1)    │
  ├─────────────────────────────────────────────┤
  │ 8. 写入备份记录日志                          │
  │    { ts, type, file, size, md5, duration }  │
  │    → 存入 server/src/data/backup-log.json   │
  ├─────────────────────────────────────────────┤
  │ 9. 清理本地临时文件                          │
  │    删除 .sql.gz 和 .sql.gz.md5              │
  ├─────────────────────────────────────────────┤
  │ 10. 记录成功日志（info level）               │
  │     "backup ok: daily 2.3MB 4.2s md5=xxx"  │
  └─────────────────────────────────────────────┘
```

**mysqldump 参数说明**：

| 参数 | 作用 |
|------|------|
| `--single-transaction` | 在单个事务中导出，保证一致性快照（InnoDB），不锁表 |
| `--routines` | 导出存储过程和函数 |
| `--triggers` | 导出触发器（本项目未使用，但保留以防未来添加） |
| `--no-tablespaces` | 省略表空间定义（MySQL 8.0+ 有此信息但恢复时可能导致权限错误） |
| `--skip-lock-tables` | 不锁表（与 `--single-transaction` 配合） |

**错误处理原则**：
- 任何步骤失败 → 记录 `error` 级别日志（winston 会写入 `error.log`）
- PM2 日志监控可捕获 `error.log` 中的 `[BACKUP_FAIL]` 关键字
- `exit(1)` 确保 cron 可检测到失败（cron 通过 `$?` 判断），后续可接入告警
- **不静默失败**：哪怕只是 COS 不可达也记录日志，避免"以为在备份其实一周没备份"的情况

#### 11.5.3 备份安全

备份文件包含用户手机号（PII），不能以明文形式无防护存放：

| 安全层 | 措施 | 实现方式 | 优先级 |
|--------|------|---------|--------|
| **传输加密** | HTTPS | COS SDK 默认使用 HTTPS，无需额外配置 | ✅ 默认 |
| **存储加密** | COS SSE-COS 服务端加密 | 腾讯云 COS 控制台 → 存储桶 → 安全管理 → 服务端加密 → 开启 SSE-COS。**免费，一键开启。** | 🔴 必须 |
| **访问控制** | 备份 Bucket 独立 + 最小权限 | 生产环境使用独立的 COS Bucket（不与图片上传共用），仅授予 `backup.js` 所需的最小权限（PutObject + GetObject + DeleteObject + HeadBucket） | 🟡 推荐 |
| **备份文件加密**（可选增强） | AES-256-CBC 加密后上传 | `mysqldump \| gzip \| openssl enc -aes-256-cbc -pass env:BACKUP_KEY \| cos upload`。恢复时反向解密。密钥存储在 `.env` 中，与备份分离。 | 🟢 中期可选 |

**MVP 阶段最低要求**：SSE-COS 开启（控制台 1 分钟操作）。这是免费的，没有理由不开启。

**CAM 最小权限策略示例**（生产环境用独立子账号操作备份 Bucket）：

```json
{
  "Statement": [
    { "Effect": "Allow", "Action": ["cos:PutObject", "cos:GetObject", "cos:DeleteObject", "cos:HeadObject", "cos:HeadBucket"], "Resource": ["qcs::cos:ap-guangzhou:uid/xxxxx:backup-bucket-xxxxx/*"] }
  ]
}
```

#### 11.5.4 COS 生命周期自动清理

不依赖 `backup.js` 手动删除过期备份——脚本可能因 bug 漏删或误删。使用 COS 生命周期规则实现**全自动、零代码**的过期清理：

| 路径前缀 | 自动删除时间 | 说明 |
|---------|------------|------|
| `backups/daily/` | **7 天**后删除 | 快照级别保护，过期自动清理 |
| `backups/weekly/` | **30 天**后删除 | 比实际保留需求略宽松，留有缓冲 |
| `backups/monthly/` | **90 天**后删除 | 合规审计最低保留期 |
| `backups/pre-migrate/` | **365 天**后删除 | migration 回滚需求随时间递减 |

**配置方式**：腾讯云 COS 控制台 → 存储桶 → 生命周期 → 新增规则（4 条，分别匹配上述前缀）。配置后 COS 每天凌晨自动执行删除，无需任何代码。

**为什么不用 backup.js 手动清理**：
- 脚本异常时旧文件堆积，存储成本微乎其微（140 MB = ¥0.02/月），不是主要问题
- 真正的问题是：手动清理逻辑写错了可能**删除不该删的文件**（比如删了还在保留期内的日备份）
- COS 生命周期是声明式的——"7 天后删除"就是这个语义，没有 off-by-one 的风险
- 额外收益：生命周期规则包含**不完全分条上传**的自动清理（`IncompleteMultipartUpload`），防止上传中断的碎片文件堆积

#### 11.5.5 恢复 SOP（决策树）

恢复操作不是只有一种情况。按故障严重程度分为 3 种场景，选择对应的恢复流程。

**恢复决策树**：

```
发现问题
    │
    ├─→ 确认故障范围
    │       │
    │       ├─ 单表误删/误改（如 DELETE products WHERE id=123 多删了）
    │       │       └─→ 场景 A：单表恢复（5-10 分钟，不停服）
    │       │
    │       ├─ 多表损坏 / 全库数据异常（如 DROP TABLE、批量 UPDATE 写错）
    │       │       └─→ 场景 B：全库恢复（15-25 分钟，停服约 15 分钟）
    │       │
    │       └─ 服务器完全丢失（磁盘损坏、到期释放、不可恢复的系统故障）
    │               └─→ 场景 C：服务器重建（1-2 小时，完全停服）
    │
    └─→ ⚠️ 无论哪种场景，第一步永远是：抢救备份当前状态
```

---

**场景 A：单表误删/误改（最常见，约 80% 的数据恢复需求）**

特点：不停服，不影响其他功能，快速恢复。

```bash
# === 步骤 1：确认受损表名与大致时间范围 ===
# 从应用日志中确认是哪个表、大概什么时候出的问题

# === 步骤 2：抢救备份当前数据（必须！）===
# 即使当前数据有问题，也要备份。万一恢复选错文件，还能回退
mysqldump -u root -p campus_market --single-transaction \
  --skip-lock-tables | gzip > emergency-$(date +%Y%m%d-%H%M).sql.gz

# === 步骤 3：从 COS 下载最新日备份 ===
# 假设 COS CLI 已配置（或使用 cos-nodejs-sdk 封装的下载脚本）
cos-cli cp cos://bucket/backups/daily/campus_market-2026-06-03.sql.gz ./
# 验证完整性：对比 MD5
cos-cli cp cos://bucket/backups/daily/campus_market-2026-06-03.sql.gz.md5 ./
md5sum -c campus_market-2026-06-03.sql.gz.md5  # 必须输出 OK

# === 步骤 4：解压并提取单表 ===
gunzip campus_market-2026-06-03.sql.gz
# 从完整 SQL 中提取目标表的 CREATE + INSERT 语句
sed -n '/^-- Table structure for table .products./,/^-- Table structure for table/p' \
  campus_market-2026-06-03.sql > products_recover.sql

# === 步骤 5：导入单表（应用正常在线）===
mysql -u root -p campus_market < products_recover.sql

# === 步骤 6：验证恢复结果 ===
mysql -u root -p campus_market -e "SELECT COUNT(*) FROM products;"
mysql -u root -p campus_market -e "SELECT id, title FROM products ORDER BY id DESC LIMIT 5;"
# 抽样检查：抽查 3-5 条记录的字段完整性

# === 步骤 7：清理 ===
rm campus_market-2026-06-03.sql products_recover.sql emergency-*.sql.gz
```

> 预估耗时：**5-10 分钟**。服务全程在线，不影响其他功能。注意：如果目标表通过外键被其他表引用，需同步检查关联表完整性。

---

**场景 B：全库损坏/误删（如 DROP TABLE、批量 UPDATE 写错 WHERE）**

特点：必须停服，操作需谨慎，但恢复后数据完整。

```bash
# === 步骤 1：立即停服（阻止新数据写入）===
pm2 stop app
# 确认服务已停止
curl -s http://localhost:3000/api/health  # 预期无响应或拒绝连接

# === 步骤 2：抢救备份当前残留数据（必须！）===
# 即使数据已损坏，可能仍有部分表可读。这是恢复的"后悔药"
mysqldump -u root -p campus_market --single-transaction --skip-lock-tables \
  | gzip > emergency-before-restore-$(date +%Y%m%d-%H%M).sql.gz
# 将此抢救备份保存到本地，不要上传 COS（避免混淆）

# === 步骤 3：选择恢复目标 ===
# 优先级：最新日备份 → 前一天 → 上周备份 → 上月备份
# 选择原则：优先使用最新的确定无问题的备份
# 对于"误操作发生在 2 小时前"，选今天凌晨的日备份
# 对于"一周前上架的违规商品今天才发现"，选 7 天前的那份日备份（或上周周备份）
ls cos://bucket/backups/daily/    # 列出可用的日备份
ls cos://bucket/backups/weekly/   # 列出可用的周备份

# === 步骤 4：下载并验证 ===
cos-cli cp cos://bucket/backups/daily/campus_market-2026-06-03.sql.gz ./
cos-cli cp cos://bucket/backups/daily/campus_market-2026-06-03.sql.gz.md5 ./
md5sum -c campus_market-2026-06-03.sql.gz.md5  # ⚠️ 不通过则换另一份备份

# === 步骤 5：执行恢复 ===
gunzip campus_market-2026-06-03.sql.gz
mysql -u root -p campus_market < campus_market-2026-06-03.sql

# === 步骤 6：启动服务 ===
pm2 start app
sleep 3  # 等待 Node.js 启动完成

# === 步骤 7：验证恢复完整性 ===
# 不止是 health check！
curl http://localhost:3000/api/health
# 核心表行数对比（与备份前的预期行数对比）
mysql -u root -p campus_market -e "
  SELECT 'users' AS tbl, COUNT(*) AS cnt FROM users
  UNION ALL SELECT 'products', COUNT(*) FROM products
  UNION ALL SELECT 'orders', COUNT(*) FROM orders
  UNION ALL SELECT 'reviews', COUNT(*) FROM reviews;"
# 最新订单 ID 连续性检查（确认没有主键断裂）
mysql -u root -p campus_market -e "SELECT MAX(id) FROM orders; SELECT MAX(id) FROM reviews;"
# 抽样验证：手动查询 3 条最新订单，确认 buyer_id, seller_id, product_id 非 NULL

# === 步骤 8：通知用户 ===
# 如果停服超过 10 分钟，通过系统消息告知用户恢复完成
# （或至少发布一条"服务已恢复"的公告）

# === 步骤 9：事后复盘 ===
# 将 emergency-before-restore-*.sql.gz 保留至少 1 周，供复盘分析根因
```

> 预估耗时：**15-25 分钟**，其中停服窗口约 15 分钟。如果在凌晨 2-5 点执行（用户量最低），几乎无感知。**RTO 目标 < 30 分钟可达成。**

---

特点：服务器磁盘损坏、到期释放、被入侵后系统不可信。需要从零重建。


#### 11.5.6 恢复验证

两层验证：自动（每周） + 人工（每季度）。

**自动验证（每周一凌晨 4:00，cron）**：

```
1. 从 COS 下载最新日备份（.sql.gz）
2. 验证 MD5 校验 → 不通过则报警并跳过
3. 解压 → 导入临时 MySQL 实例（campus_market_verify，用完即删）
4. 执行基础查询：
   • SELECT COUNT(*) FROM users, products, orders, reviews
   • 对比 backup-log.json 中上次记录的 COUNT 值，波动 ±5% 内算正常
   • 抽样：SELECT * FROM products ORDER BY RAND() LIMIT 5
5. 记录验证结果到 server/src/data/verify-log.json
   { ts, backup_file, checks: { users, products, orders, reviews }, status: "pass|fail" }
6. 删除临时库 DROP DATABASE campus_market_verify
```

**人工演练（每季度一次）**：

| 演练项目 | 内容 | 验收标准 |
|---------|------|---------|
| 场景 A 演练 | 模拟误删 products 表 10 行数据，执行单表恢复 | 10 分钟内恢复，无停服 |
| 场景 B 演练 | 模拟 DROP TABLE orders 后全库恢复 | 25 分钟内完成，RTO < 30min |

#### 11.5.7 RPO 改善路线（中期增量备份）

当前 RPO = 24 小时（最坏情况丢失一天交易数据）。对于交易系统，这个数字不够理想。改善方案：

**第一阶段（MVP 后，约第 14-15 轮）**：

开启 MySQL binlog（二进制日志），实现增量备份：

```ini
# my.cnf
[mysqld]
log_bin = /var/log/mysql/mysql-bin.log
binlog_format = ROW          # 行级日志，最精确
expire_logs_days = 7         # binlog 保留 7 天（配合日备份的 7 天）
max_binlog_size = 100M       # 单个 binlog 文件最大 100MB
```

恢复流程变为：**全量恢复最新日备份 + binlog 重放到指定时间点**，可将 RPO 缩短到 **< 1 小时**（取决于 binlog 上传频率）。

```bash
# 时间点恢复示例：恢复到 2026-06-03 14:30:00
mysqlbinlog --stop-datetime="2026-06-03 14:30:00" mysql-bin.* | mysql -u root -p
```

**第二阶段（成熟期，用户量增长后）**：
- binlog 准实时上传 COS（每 5 分钟增量同步）
- 支持 Point-in-Time Recovery（PITR），RPO → **< 5 分钟**
- 需要：独立备份服务器或云数据库（腾讯云 CDB 自带 PITR）

**MVP 阶段**：保持 RPO = 24h，在文档中记录改善路线。这是务实的取舍——单人开发、百级用户，数据丢失 24h 的影响可控（最多丢失几十条订单，可通过客服人工补偿处理）。

## 十六、Moderation 内容审核模块

### 16.1 审核分层

```
第一层：自动审核（发布/修改时触发）
  ├── 敏感词过滤（DFA，详见 §16.3 词库规范）
  ├── 图片格式校验（魔数 + content-type + 宽高）
  └── 结果：通过 → 发布 / 不通过 → 拒绝（6002）并提示

第二层：事后抽查（管理员定时）
  ├── 每日随机抽查 5% 新发布商品
  └── 异常商品 → 强制下架 + 通知用户

第三层：举报触发人工审核
  ├── 用户提交举报 → 客服工单后台
  └── 客服审核 → 通过（处理） / 驳回（恢复）
```

### 16.2 审核状态机

```
用户发布 → pending_auto
  ├── auto_approved（直接上架）
  ├── auto_rejected（敏感词/格式问题，拒绝发布）
  └── manual_review（可疑内容，待人工审核）
       ├── approved（上架）
       └── rejected（下架 + 通知）
```

> **说明：** `auto_rejected` 为发布时即时拦截（如命中敏感词、图片格式校验失败），直接拒绝发布，不进入上架流程。`manual_review` 仅用于触发高风险启发式规则但不确定的极少数情况（如命中多个模糊规则但未达直接拒绝阈值），正常情况下 99%+ 的商品走 `auto_approved` 即时上架。整体策略与 PRD"发布即上架 + 事后抽查/举报下架"一致——`manual_review` 是自动审核的兜底安全阀，而非前置人工审核。

#### 16.3.3 DFA 加载与数据结构

```js
// server/src/utils/sensitive-filter.js
const fs = require('fs');
const path = require('path');

class SensitiveFilter {
  constructor() {
    this.trie = {};            // DFA 字典树根节点
    this.wordCount = 0;        // 已加载词数
    this.loadedAt = null;      // 加载时间戳
  }

  // 从文件构建 DFA 字典树
  load(filePath) {
    const content = fs.readFileSync(filePath, 'utf-8');
    const lines = content.split('\n');
    const wordSet = new Set();
    let skipped = 0;

    for (const raw of lines) {
      const line = raw.trim();
      // 跳过空行和注释行
      if (!line || line.startsWith('#')) continue;
      // 非法字符检查
      if (!/^[一-龥a-zA-Z0-9]+$/.test(line)) {
        console.warn(`[sensitive-filter] Skipping invalid word: "${line}"`);
        skipped++;
        continue;
      }
      wordSet.add(line.toLowerCase());
    }

    // 构建字典树
    this.trie = {};
    for (const word of wordSet) {
      let node = this.trie;
      for (const char of word) {
        if (!node[char]) node[char] = {};
        node = node[char];
      }
      node.isEnd = true;      // 词尾标记
    }

    this.wordCount = wordSet.size;
    this.loadedAt = new Date().toISOString();
    if (skipped > 0) console.warn(`[sensitive-filter] Skipped ${skipped} invalid lines`);
    console.log(`[sensitive-filter] Loaded ${this.wordCount} words`);
  }

  // 检测文本是否命中敏感词，返回命中的词列表
  check(text) {
    const hits = [];
    const str = text.toLowerCase();
    const len = str.length;

    for (let i = 0; i < len; i++) {
      let node = this.trie;
      let j = i;
      while (j < len && node[str[j]]) {
        node = node[str[j]];
        j++;
        if (node.isEnd) {
          hits.push(str.slice(i, j));
          break;  // 找到最短匹配即停止（也可改为继续找最长匹配）
        }
      }
    }

    return {
      ok: hits.length === 0,
      hits: [...new Set(hits)],  // 去重
      suggestion: hits.length > 0 ? '输入内容包含敏感信息，请修改后重试' : '',
    };
  }

  // 热重载（不重启服务更新词库）
  reload(filePath) {
    console.log('[sensitive-filter] Reloading...');
    this.load(filePath);
  }

  // 获取统计信息
  stats() {
    return { wordCount: this.wordCount, loadedAt: this.loadedAt };
  }
}

// 单例
module.exports = new SensitiveFilter();
```

**性能特征**：

| 指标 | 值 |
|------|------|
| 时间复杂度 | O(n × m)，n = 文本长度，m = 最长敏感词长度（通常 ≤ 10） |
| 空间复杂度 | O(字符总数)，约 500 词 × 平均 3 字符 = 1500 节点 |
| 内存占用 | < 1MB（500 词级别） |
| 加载时间 | < 5ms（从文件加载 500 词构建字典树） |
| 检测吞吐 | > 10,000 字符/ms |

#### 16.3.4 防绕过策略

| 绕过手段 | 示例 | 防御方式 |
|---------|------|---------|
| 插入空格 | "裸 聊" | 检测前 `removeSpacing()` 去除连续空格和全角空格 |
| 插入特殊符号 | "裸*聊" | 检测前 `removeSymbols()` 去除 `* - _ . , / \ | @ # $ % ^ &` |
| 同音/形近字 | "果聊"（果=裸） | 维护形近字映射表 `{果:裸, ...}`，检测时展开替换 |
| 拼音 | "luoliao" | 词库补充常见拼音变体 |
| 拆分 | "衤果 耳卯" | 维护拆字映射表（P2，暂不处理，人工举报兜底） |

```js
// 预处理管道（check() 方法入口）
function preprocess(text) {
  let str = text.toLowerCase().trim();
  str = str.replace(/\s+/g, '');                     // 去空格
  str = str.replace(/[*\-_.,/\\|@#$%^&()[\]{}]/g, ''); // 去特殊符号
  // 形近字映射（按需扩展）
  const shapeMap = { '果':'裸', '扣':'枪', '馬':'马' };
  str = str.split('').map(c => shapeMap[c] || c).join('');
  return str;
}
```

> MVP 阶段只做去空格 + 去符号 + 去重。形近字和拆字由用户举报 + 客服人工处理兜底（见 §8.5）。

| 某词连续 1 个月无命中 | 标记 `# deprecated`，保留 1 个月后删除 |
| 某词误拦截 ≥ 5 次/月 | 立即移除，加入白名单 |
| 词库超过 1000 词 | 审计并清理低频词 |

#### 16.3.6 误拦截申诉处理

```
用户被敏感词拦截
  │
  ├── 提示："输入内容包含敏感信息，请修改后重试"
  │    不告知具体敏感词（防止针对性绕过）
  │
  ├── 用户认为误拦截 → "我" → "联系客服" → 提交申诉
  │
  └── 客服审查：
        ├── 确认误拦截 → 从词库移除该词 + 通知用户（1 个工作日内）
        └── 非误拦截 → 维持拦截，通知用户"内容经审核确认违规"
```

#### 16.3.7 管理 API

| 端点 | 方法 | 权限 | 说明 |
|------|:--:|:--:|------|
| `/api/admin/sensitive/stats` | GET | admin | 返回词库统计：词数、分类分布、上次加载时间 |
| `/api/admin/sensitive/reload` | POST | admin | 热重载词库（从文件重新读取并重建 DFA 字典树） |
| `/api/admin/sensitive/check` | POST | admin | 测试模式：输入文本 → 返回命中词列表（调试用，不记录） |

#### 16.3.8 监控指标

| 指标 | 采集方式 | 告警 |
|------|---------|------|
| 每日命中次数 | `user_events` 埋点 `sensitive_hit` | 单日 > 500 次 → 疑似刷量攻击 |
| 各词命中频率 | 命中日志聚合（仅记录词名，不记录用户文本） | 某词 > 100 次/日 → 审查是否过于宽泛 |
| 误拦截申诉数 | 工单系统 count（type=sensitive_appeal） | > 10 次/月 → 全量审计词库 |
| 词库加载失败 | `load()` 方法 try-catch + error 日志 | 任何失败立即告警 |

## 十八、Risk Control 风控模块

### 18.1 信誉分模型

| 事件 | 分值变动 |
|------|----------|
| 初始分 | 100 |
| 完成一笔交易（买方确认收货，卖家 +2） | +2 |
| 收到好评（4 星及以上） | +1 |
| 被举报属实 | -30 |
| 获得差评（2 星及以下） | -5 |

> **评分计算：** 评价包含沟通态度、守时程度、描述一致度三个维度，各 1-5 分。总分取三维度**平均分**（四舍五入）：≥ 4 为好评、≤ 2 为差评。
> 
> **上下限：** 信誉分最低 0（不跌破），最高 200（由 `CREDIT_MAX` 环境变量控制，不超上限）。

### 18.2 阈值策略

| 信誉分区间 | 限制 |
|-----------|------|
| ≥ 60 | 正常使用 |
| 30 ~ 59 | 禁止发布商品（返回 4008），可浏览/购买/聊天 |
| < 30 | 禁止发起交易（返回 4009），仅可浏览 |
| ≤ 0 | 自动标记为高风险，通知管理员人工审核决定是否封禁 |

### 18.3 黑名单机制

- 封禁用户自动进入黑名单（通过 status=banned 实现）
- 管理员可设置封禁时长（临时/永久）
- 临时封禁到期后自动解封（cron job 检查 `banned_at + duration < now`）
- 信誉分 ≤ 0 → 自动标记为高风险用户，通知管理员人工审核决定是否封禁（与 §18.2 一致——信誉分 ≤ 0 单独触发，无需额外举报条件）

## 二十、Search 全文检索模块

### 20.1 MySQL FULLTEXT 方案

利用 MySQL InnoDB 原生全文索引实现搜索：

```sql
-- 建表时或通过迁移添加（已在 §4.2 定义）
ALTER TABLE products ADD FULLTEXT INDEX ft_products (title, description) WITH PARSER ngram;
```

- **索引字段**：title（权重最高）、description、category
- **搜索方式**：`SELECT *, MATCH(title, description) AGAINST(? IN NATURAL LANGUAGE MODE) AS relevance FROM products WHERE MATCH(title, description) AGAINST(? IN NATURAL LANGUAGE MODE) ORDER BY relevance DESC`
- **中文支持**：`WITH PARSER ngram` 使用 MySQL 内置的 ngram 分词器（默认 token 大小 2），无需额外引入分词库
- **最小词长**：`innodb_ft_min_token_size=1`（ngram 场景下生效）
- **停用词**：使用 MySQL 默认停用词列表（可自定义 `innodb_ft_user_stopword_table`）

### 20.2 搜索策略

```
用户输入 keyword
  ├── FULLTEXT MATCH ... AGAINST (IN NATURAL LANGUAGE MODE) — 按相关性排序
  ├── Fallback：LIKE '%keyword%'（FULLTEXT 无结果时）
  │     └── 卖家昵称模糊匹配
  └── 结果去重 + 过滤（status='active'，软删除通过 status='deleted' 实现）
```

> **BOOLEAN MODE vs NATURAL LANGUAGE MODE：** MVP 阶段用 NATURAL LANGUAGE MODE（按相关性排序）。后续如果需要 `+必须包含 -排除` 高级语法，升级为 BOOLEAN MODE。

### 20.3 热词统计

- 搜索时记录 keyword 到 `user_events`（event=search）
- 每小时聚合 top 20 搜索词存入缓存
- Dashboard 展示热门搜索词排名

### 20.4 ES 迁移预留

当搜索量和数据量增长到 MySQL FULLTEXT 瓶颈时（预估 > 10 万商品或 QPS > 100），迁移路径：
1. 部署 ElasticSearch 实例
2. `search.js` 新增 `ElasticSearchProvider`
3. 数据同步：商品变更时通过消息队列同步到 ES
4. 前端搜索 API 无需改动

| CI 中每个测试文件独立线程 | vitest `pool: 'threads'`，`isolate: true` |

### 23.7 外部依赖 Mock 策略

| 外部依赖 | Mock 方式 | 说明 |
|---------|----------|------|
| 微信 `getPhoneNumber` API | `vi.fn().mockResolvedValue({ phone: '13800138000' })` | 单元/集成测试均 mock，不调真实微信 |
| 微信 `getAccessToken` | `vi.fn().mockResolvedValue({ access_token, expires_in })` | 测试微信 token 缓存逻辑时 mock |
| 腾讯云 IM REST API | `vi.fn().mockResolvedValue({ ActionStatus: 'OK' })` | 系统消息发送 mock |
| COS STS | `vi.fn().mockResolvedValue({ credentials, expiredTime })` | 临时密钥获取 mock |
| 腾讯云 COS 上传 | `vi.fn().mockResolvedValue({ Location: 'https://...' })` | 文件上传 mock |

> 原则：**只 mock 网络边界**。`config/`、`repository/`、`services/` 内部的 MySQL 调用、业务逻辑**不做 mock**，用真实测试数据库验证。

### 23.8 覆盖率目标

| 类型 | 目标 | 说明 |
|------|:--:|------|
| 整体行覆盖率 | **≥ 70%** | MVP 阶段务实目标 |
| `services/` 业务逻辑 | **≥ 85%** | 订单状态机、信誉分、JWT 等核心逻辑 |
| `repository/` 数据访问 | **≥ 60%** | 简单 CRUD 可通过集成测试覆盖 |
| `middleware/` 中间件 | **≥ 90%** | 限流器、错误处理、JWT 验证 |
| `utils/` 工具函数 | **≥ 80%** | DFA 敏感词、校验函数、加密工具 |
| `routes/` 路由 | **≥ 50%** | 通过集成测试间接覆盖即可 |

不计入覆盖率统计：
- `migrations/` — 配置性质
- `seeds/` — 数据性质
- `config/index.js` — env 加载无业务逻辑
- 第三方库代码（`node_modules/`）

### 23.10 CI 集成

```yaml
# .github/workflows/ci.yml
name: CI
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '18' }
      - run: npm ci
      - run: npx vitest run --coverage
      - name: Upload coverage
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: coverage
          path: coverage/
```

| CI 规则 | 说明 |
|---------|------|
| 触发条件 | 每个 PR / push 到 main 自动运行 |
| 失败策略 | 任一测试失败 → CI 标红 → 禁止合并 |
| 覆盖率门禁 | 整体行覆盖率 < 70% → CI 警告（不阻断，MVP 阶段） |
| 超时 | 单测试文件 10s，整体 60s |
| 通知 | 失败时 GitHub PR 页面可见，暂不接微信通知 |

### 23.12 测试编写优先级（与开发轮次对应）

| 开发轮次 | 先写什么测试 | 说明 |
|---------|-------------|------|
| 第 1 轮（骨架） | `health.test.js` 集成测试 | 验证 app 能启动，DB 能连接 |
| 第 2 轮（认证） | `auth.test.js` + `jwt.test.js` + `rate-limiter.test.js` | 注册/登录/Token/限流 |
| 第 3 轮（商品） | `products.test.js` 集成 + `validate.test.js` | 商品 CRUD + 字段校验 |
| 第 5 轮（IM） | IM 相关 mock 验证 | 不测试 TIM SDK 本身，只测系统消息 payload 格式 |
| 第 6 轮（交易） | `order-state.test.js` + `orders.test.js` | **本轮测试最重要**，状态机必须全覆盖 |
| 第 7 轮（举报） | `reports.test.js` + `credit.test.js` | 举报流转 + 信誉分联动 |
| 第 9 轮（通知） | 通知清理 cron 测试 | 30 天自动清理逻辑 |
| 第 13 轮（收尾） | 全部补到目标覆盖率 | 补漏 + CI 跑通 |

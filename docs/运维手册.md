## 一、服务器基础信息

### 1.1 服务器规格

| 项目 | 配置 |
|------|------|
| 云服务商 | 腾讯云轻量应用服务器 |
| 实例规格 | 2 核 CPU / 2 GB 内存 |
| 操作系统 | Ubuntu 22.04 LTS |
| 预估月费 | 约 ¥50（学生优惠） |
| 地域 | 广州（ap-guangzhou） |

### 1.2 网络与安全组

生产环境安全组规则遵循最小开放原则：

| 端口 | 协议 | 来源 | 用途 |
|:----:|------|------|------|
| 22 | TCP | 运维人员 IP（建议指定，非 0.0.0.0/0） | SSH 远程管理 |
| 80 | TCP | 0.0.0.0/0 | HTTP（用于 Let's Encrypt 证书验证 + 到 443 的重定向） |
| 443 | TCP | 0.0.0.0/0 | HTTPS（小程序 API 请求入口） |

> **禁止开放 3306（MySQL）：** 数据库仅监听 `127.0.0.1`，不对外暴露。所有数据库操作通过 SSH 登录服务器后在本地执行。

### 1.3 访问方式

```bash
# SSH 登录（使用密钥，禁用密码登录）
ssh -i ~/.ssh/campus-market.pem root@<服务器公网IP>

# 健康检查（公网）
curl https://<域名>/api/health
```

## 二、环境体系

### 2.1 三环境定义

| 环境 | 用途 | 数据库 | 配置文件 | 部署方式 |
|------|------|--------|---------|---------|
| **development** | 本地开发、断点调试 | `campus_market_dev`（本地 MySQL） | `.env.development` | `npm run dev`（nodemon） |
| **staging** | 上线前验证 | `campus_market_staging`（服务器 MySQL） | `.env.staging` | PM2（端口 3001） |
| **production** | 线上正式环境 | `campus_market`（服务器 MySQL） | `.env.production` | PM2（端口 3000） |

> staging 环境部署在与 production 同一台轻量服务器上（端口 3001），上线前必须在 staging 完成全量验证。staging 使用独立的 IM 测试应用和 COS 测试 Bucket，避免污染生产数据。

### 2.2 环境变量差异

| 变量 | development | staging | production |
|------|:----------:|:-------:|:----------:|
| `NODE_ENV` | `development` | `staging` | `production` |
| `PORT` | `3000` | `3001` | `3000` |
| `DB_NAME` | `campus_market_dev` | `campus_market_staging` | `campus_market` |
| `CORS_ORIGIN` | `*` | 子域名 | 正式域名 |
| `LOG_LEVEL` | `debug` | `info` | `warn` |
| `ENABLE_CRON` | `false` | `true` | `true` |
| `RATE_LIMIT_GLOBAL` | `999` | `60` | `60` |
| `RATE_LIMIT_SENSITIVE` | `999` | `10` | `10` |

### 2.3 密钥管理

| 规则 | 说明 |
|------|------|
| `.env.*` 不入库 | 所有 `.env.*` 文件在 `.gitignore` 中，仓库仅保留 `.env.example` 模板 |
| 密钥分离 | `JWT_ACCESS_SECRET` 和 `JWT_REFRESH_SECRET` 使用不同的随机值 |
| 密钥强度 | `WX_APPSECRET`、`IM_SECRET_KEY`、`COS_SECRET_KEY` 均 ≥ 32 字符随机字符串 |
| 轮换周期 | 核心密钥至少每 6 个月更换一次 |
| 分发方式 | 生产密钥通过加密信道（如 1Password / 私密聊天）分发，不以明文邮件/微信发送 |
| 泄露响应 | 立即在云服务控制台重置密钥 → 更新 `.env.production` → `pm2 reload` |

## 三、部署流程

### 3.1 首次部署（服务器从零开始）

#### 步骤一：购买并初始化服务器

1. 腾讯云控制台 → 轻量应用服务器 → 新建
2. 选择：地域"广州"、镜像"Ubuntu 22.04"、套餐"2C2G"
3. 创建后记录公网 IP，在 DNS 管理后台添加 A 记录指向该 IP

#### 步骤二：执行环境初始化脚本

项目中维护了 `server/setup.sh`，使用前确认内容无误后执行：

```bash
# 将脚本上传到服务器
scp server/setup.sh root@<服务器IP>:/root/
# SSH 登录后执行
ssh root@<服务器IP>
bash /root/setup.sh
```

`setup.sh` 自动完成以下安装：

```
1. apt update && apt upgrade -y           # 系统更新
2. Node.js 18 LTS                         # deb.nodesource.com 官方源
3. MySQL 8.0（mysql-server）              # 启动 + 开机自启
4. Nginx + certbot + python3-certbot-nginx # 反向代理 + HTTPS 证书
5. PM2（npm install -g pm2）              # 进程管理
6. 防火墙：ufw allow 22/80/443 && enable
```

#### 步骤三：安全加固 MySQL

```bash
# 运行 MySQL 安全配置向导
mysql_secure_installation

# 交互式设置：
# - 密码强度插件：选 Y（建议开启）
# - root 密码：设置复杂密码（≥ 16 位，含大小写字母 + 数字 + 符号）
# - 移除匿名用户：Y
# - 禁止 root 远程登录：Y
# - 删除 test 数据库：Y
# - 重新加载权限表：Y
```

```bash
# 创建应用专用数据库用户（不使用 root）
mysql -u root -p <<EOF
CREATE DATABASE campus_market CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'campus_app'@'127.0.0.1' IDENTIFIED BY '<应用数据库密码>';
GRANT SELECT, INSERT, UPDATE, DELETE, EXECUTE ON campus_market.* TO 'campus_app'@'127.0.0.1';
FLUSH PRIVILEGES;
EOF
```

#### 步骤四：部署应用代码

```bash
# 克隆仓库
git clone <仓库地址> /opt/campus-market
cd /opt/campus-market/server

# 切换到发布版本
git checkout main

# 安装生产依赖
npm ci --production

# 配置环境变量
cp .env.example .env.production
vim .env.production  # 填入真实密钥值
```

#### 步骤五：执行数据库迁移

```bash
# 先在 staging 数据库验证迁移（如有 staging 环境）
NODE_ENV=staging npm run db:migrate

# 生产环境迁移（按 §3.2 的带备份流程执行）
npm run db:migrate
```

#### 步骤六：配置 Nginx + HTTPS

```bash
# 复制 Nginx 配置文件
cp /opt/campus-market/server/nginx/campus-market.conf /etc/nginx/sites-available/
ln -s /etc/nginx/sites-available/campus-market.conf /etc/nginx/sites-enabled/

# 测试 Nginx 配置
nginx -t

# 重载 Nginx
systemctl reload nginx

# 申请 SSL 证书（Let's Encrypt 免费证书，90 天有效期，certbot 自动续期）
certbot --nginx -d <你的域名>

# 验证自动续期
certbot renew --dry-run
```

#### 步骤七：启动服务

```bash
# 启动 PM2
pm2 start /opt/campus-market/server/ecosystem.config.js --env production

# 设置 PM2 开机自启
pm2 startup systemd
pm2 save

# 验证
curl http://localhost:3000/api/health
# 预期：{"status":"ok","timestamp":"...","db":"connected"}
```

### 3.2 日常更新部署

以下流程适用于不涉及数据库 schema 变更的代码更新。涉及 schema 变更时，必须按"带迁移的部署流程"（见 §3.3）执行。

```bash
# 日常更新流程（纯代码变更，无 schema 改动）
cd /opt/campus-market/server
git pull origin main
npm ci --production
pm2 reload ecosystem.config.js --env production

# 部署后 30 秒验证
sleep 3
curl http://localhost:3000/api/health
```

### 3.3 带迁移的部署流程

当本次发布涉及 schema 变更（新增迁移 SQL 文件）时，必须按以下流程执行：

```
部署前（staging 环境）：
  1. npm run db:migrate          # 先在 staging 执行，验证迁移正确
  2. npm test                     # 全量测试确认迁移不破坏现有功能
  3. npm run db:migrate:rollback  # 验证回滚正常
  4. npm run db:migrate           # 重新应用，准备上线

部署时（production 环境）：
  1. pm2 stop campus-market                          # 停服
  2. mysqldump -u campus_app -p campus_market \
       --single-transaction --skip-lock-tables \
       | gzip > backup-pre-migrate-$(date +%Y%m%d-%H%M).sql.gz  # 备份（必须！）
  3. npm run db:migrate                              # 执行迁移
  4. npm run db:migrate:status                       # 验证全部 applied
  5. git pull && npm ci --production                 # 拉取新代码
  6. pm2 start ecosystem.config.js --env production  # 启动
  7. curl http://localhost:3000/api/health           # 健康检查

回滚（迁移或代码出问题时）：
  1. pm2 stop campus-market
  2. mysql -u campus_app -p campus_market < backup-pre-migrate-<timestamp>.sql  # 恢复备份
  3. git checkout <previous-tag>                     # 回滚代码
  4. npm ci --production
  5. pm2 start ecosystem.config.js --env production
```

### 3.4 版本标记

每次发布打 git tag：

```bash
git tag -a v1.0.0 -m "发布说明"
git push origin v1.0.0
```

版本号遵循语义化版本（SemVer）：`主版本.次版本.修订号`。

### 3.5 回滚方案（纯代码）

不涉及 schema 变更的代码回滚：

```bash
cd /opt/campus-market/server
git checkout <上一个可用版本的tag>  # 如 v0.9.0
npm ci --production
pm2 reload ecosystem.config.js --env production
```

> 涉及 schema 变更的回滚：按 §3.3 中"回滚"步骤执行（需同时恢复数据库备份）。

## 四、进程管理（PM2）

### 4.1 配置文件（ecosystem.config.js）

```js
module.exports = {
  apps: [{
    name: 'campus-market',
    script: 'app.js',
    instances: 1,              // fork 模式，单进程 —— 必须！
    exec_mode: 'fork',         // 不得使用 cluster，否则 cron 重复执行
    max_memory_restart: '500M',// 内存超限自动重启
    env: {
      NODE_ENV: 'production',
      ENABLE_CRON: 'true',
      PORT: 3000,
    },
    env_staging: {
      NODE_ENV: 'staging',
      ENABLE_CRON: 'true',
      PORT: 3001,
    },
    // 日志配置
    log_date_format: 'YYYY-MM-DD HH:mm:ss Z',
    error_file: '/opt/campus-market/server/logs/pm2-error.log',
    out_file:   '/opt/campus-market/server/logs/pm2-out.log',
    merge_logs: true,
    max_size: '10M',           // 单个日志文件最大 10MB，自动滚动
  }],
};
```

> **为什么必须 fork 而非 cluster：** 项目使用 `node-cron` 在 Express 进程内执行定时任务（自动确认、超时取消、数据库备份等）。cluster 模式会启动多个进程，每个进程都会注册 `node-cron`，导致同一定时任务被重复执行——例如同一笔订单被自动确认多次。fork 模式单进程 = 单次执行，安全可靠。

### 4.2 常用 PM2 命令

```bash
# 查看进程状态
pm2 status
pm2 show campus-market        # 详细信息（CPU、内存、重启次数、运行时间）

# 启动 / 停止 / 重启
pm2 start ecosystem.config.js --env production
pm2 stop campus-market        # 优雅停止（SIGTERM → 等待连接关闭 → 退出）
pm2 reload ecosystem.config.js --env production  # 零停机重载

# 日志
pm2 logs campus-market        # 实时日志（Ctrl+C 退出）
pm2 logs campus-market --lines 100  # 最近 100 行
pm2 logs campus-market --err  # 仅错误日志

# 监控面板（终端内实时资源使用）
pm2 monit

# 开机自启
pm2 startup systemd           # 生成 systemd 服务文件
pm2 save                      # 保存当前进程列表

# 清空日志（日志文件过大时）
pm2 flush campus-market
```

### 4.3 内存告警与自动恢复

| 条件 | 动作 |
|------|------|
| 内存使用 > 400 MB | `perf.js` 每 5 分钟采集 `process.memoryUsage()`，超过阈值写入 warn 日志 |
| 内存使用 > 500 MB | PM2 `max_memory_restart` 触发自动重启 |
| 进程意外崩溃 | PM2 自动重启（默认最多 15 次连续重启，超过后停止并告警） |

## 五、Nginx 反向代理

### 5.1 配置文件（campus-market.conf）

Nginx 作为反向代理层，职责：HTTPS 终端、静态资源缓存、`/api/*` 转发、安全头注入。

关键配置要点：

```nginx
server {
    listen 443 ssl http2;
    server_name <你的域名>;

    # SSL 证书（certbot 自动管理）
    ssl_certificate     /etc/letsencrypt/live/<域名>/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/<域名>/privkey.pem;

    # 安全头
    add_header X-Content-Type-Options nosniff;
    add_header X-Frame-Options DENY;
    add_header X-XSS-Protection "1; mode=block";

    # API 反向代理
    location /api/ {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 超时配置
        proxy_connect_timeout 5s;
        proxy_read_timeout 30s;
        proxy_send_timeout 10s;
    }

    # 请求体大小限制（API 不接收文件上传，防止大 body 攻击）
    client_max_body_size 10m;
}

# HTTP → HTTPS 重定向
server {
    listen 80;
    server_name <你的域名>;
    return 301 https://$host$request_uri;
}
```

### 5.2 SSL 证书管理

```bash
# 查看证书有效期
certbot certificates

# 手动续期（certbot 通常会通过 systemd timer 自动续期）
certbot renew

# 测试自动续期流程（不实际续期）
certbot renew --dry-run

# 检查自动续期 timer 是否正常
systemctl status certbot.timer
```

> Let's Encrypt 证书有效期 90 天。certbot 安装时会自动创建 systemd timer，每天检查两次，距过期 30 天内自动续期。部署后务必执行一次 `certbot renew --dry-run` 确认自动续期正常。

### 5.3 Nginx 常用运维命令

```bash
nginx -t                         # 配置语法检查
systemctl reload nginx           # 重载配置（零停机）
systemctl restart nginx          # 重启（短暂中断）
systemctl status nginx           # 查看运行状态
tail -f /var/log/nginx/access.log  # 访问日志
tail -f /var/log/nginx/error.log   # 错误日志
```

## 六、定时任务管理

### 6.1 任务清单

所有定时任务通过 `node-cron` 在 Express 进程中执行。PM2 fork 单进程保证不重复执行。

| 任务 | Cron 表达式 | 频率 | 说明 |
|------|:--:|------|------|
| 订单超时自动确认 | `*/30 * * * *` | 每 30 分钟 | `status='met'` 且 `met_at > 3 天` → `completed` |
| pending 超时取消 | `*/30 * * * *` | 每 30 分钟 | `status='pending'` 且 `created_at > 7 天` → `timeout` |
| 数据库日备份 | `0 3 * * *` | 每天 03:00 | `backup.js --type daily` → COS `backups/daily/` |
| 通知自动清理 | `0 4 * * *` | 每天 04:00 | `DELETE FROM notifications WHERE created_at < DATE_SUB(NOW(), INTERVAL 30 DAY)` |
| 备份恢复验证 | `0 4 * * 1` | 每周一 04:00 | 下载最新日备份 → 导入临时库 → 验证 |
| IM 系统消息重试 | `*/5 * * * *` | 每 5 分钟 | `failed_system_messages` 中 `status='retrying'` 且未超重试上限的记录 |
| 数据库周备份 | （日备份中判断） | 每周日 03:00 | 同日备份额外存至 `backups/weekly/`（保留 30 天） |
| 数据库月备份 | （日备份中判断） | 每月 1 日 03:00 | 同日备份额外存至 `backups/monthly/`（保留 90 天） |
| 数据归档 | `0 3 1 * *` | 每月 1 日 | `admin_logs`(>6月) → `admin_logs_archive`；`reviews`(>1年) → `reviews_archive` |
| 举报/工单超期清理 | `0 3 1 * *` | 每月 1 日 | `reports` 及关联 `report_evidence` > 2 年 → 软删除 |
| 日志自动轮转 | — | winston 内置 | `maxFiles: '90d'`，自动删除超 90 天的日志 |

### 6.2 验证定时任务是否正常

```bash
# 方式一：查看 PM2 日志中的 cron 执行记录
pm2 logs campus-market | grep -E "\[CRON\]|backup ok|auto.confirm|cleanup"

# 方式二：检查备份文件是否按时生成
# 登录 COS 控制台 → 存储桶 → backups/daily/ → 检查最新文件日期

# 方式三：检查数据库中的定时任务执行痕迹
mysql -u campus_app -p campus_market -e "
  SELECT COUNT(*) AS pending_orders
  FROM orders WHERE status IN ('met', 'pending');
  SELECT COUNT(*) AS old_notifications
  FROM notifications WHERE created_at < DATE_SUB(NOW(), INTERVAL 30 DAY);
"
```

### 6.3 定时任务故障排查

| 症状 | 可能原因 | 排查步骤 |
|------|---------|---------|
| 订单逾期未自动确认 | cron 未执行 | 1. 确认 `ENABLE_CRON=true`；2. PM2 是否为 fork 模式；3. 检查 PM2 日志是否有 cron 注册信息 |
| 备份文件缺失 | `backup.js` 执行失败 | 1. 查看 error.log 中 `[BACKUP_FAIL]` 标记；2. 手动执行 `node backup.js --type daily` 观察报错；3. 检查 COS 密钥是否过期 |
| 通知未清理 | SQL 执行失败 | 1. 查看 error.log 中的数据库错误；2. 手动执行清理 SQL 确认语法；3. 检查 MySQL 用户权限 |
| IM 消息永久失败堆积 | 腾讯云 IM 服务异常 | 1. 查看 `failed_system_messages` 表中 `status='permanent_failed'` 的记录；2. 登录腾讯云 IM 控制台检查服务状态 |

## 七、日志管理

### 7.1 日志体系架构

| 日志类型 | Logger 名称 | 输出目标 | 级别 | 用途 |
|---------|:----:|------|------|------|
| **access** | `access` | `logs/access-%DATE%.log` | info | 每个 HTTP 请求的方法、路径、状态码、耗时、IP |
| **error** | `error` | `logs/error-%DATE%.log` | error | 应用异常、数据库错误、外部服务调用失败 |
| **business** | `business` | `logs/business-%DATE%.log` | info及以上 | 关键业务事件：用户注册、商品发布、订单状态变更、举报提交 |

### 7.2 日志配置要点

- **库：** winston v3 + winston-daily-rotate-file v5
- **切割方式：** 每天午夜 00:00 自动切割为新文件
- **保留期限：** `maxFiles: '90d'` — 超过 90 天的日志文件自动删除
- **命名格式：** `access-2026-06-04.log`、`error-2026-06-04.log`
- **日志级别（production）：** `warn` — 仅记录 warn 和 error，减少 I/O
- **日志脱敏：** winston 配置中过滤 `*SECRET*`、`*_KEY` 开头的环境变量值，禁止写入日志
- **PM2 进程日志：** 与 winston 业务日志分开——PM2 输出 stdout/stderr 到 `pm2-error.log`/`pm2-out.log`，winston 写入 `logs/` 目录

### 7.3 常用日志查看命令

```bash
# 查看今天的 access 日志（按需调整日期）
tail -100 /opt/campus-market/server/logs/access-$(date +%Y-%m-%d).log

# 查看今天的 error 日志
cat /opt/campus-market/server/logs/error-$(date +%Y-%m-%d).log

# 搜索特定错误
grep "ERR_DB_QUERY_TIMEOUT" /opt/campus-market/server/logs/error-*.log

# 搜索特定用户的请求
grep "uid=xxx" /opt/campus-market/server/logs/access-$(date +%Y-%m-%d).log

# 统计今日 HTTP 状态码分布
grep -oP '"\s+\d{3}\s+' /opt/campus-market/server/logs/access-$(date +%Y-%m-%d).log \
  | sort | uniq -c | sort -rn

# 查找慢请求（> 1s）
grep -E "duration.*[0-9]{4,}ms" /opt/campus-market/server/logs/access-$(date +%Y-%m-%d).log
```

### 7.4 磁盘空间管理

```
日志目录预估增长（校园量级）：
  - access 日志：约 5–20 MB / 天
  - error 日志：约 0.5–5 MB / 天（正常情况极低）
  - business 日志：约 1–3 MB / 天
  - 90 天总占用：约 600 MB–2.5 GB

检查磁盘使用：
```

## 八、备份与恢复

> 详细设计见 [技术架构文档 §11.5](技术架构文档.md)。本章节聚焦于运维操作——如何执行备份、如何恢复。

### 8.1 备份分层（GFS 模型）

| 层级 | 备份频率 | 保留期限 | COS 路径 | 用途 |
|------|---------|:------:|---------|------|
| **日备份（Son）** | 每天 03:00 | 7 天 | `backups/daily/` | 昨天误操作的快速回滚（最常见场景） |
| **周备份（Father）** | 每周日 03:00 | 30 天 | `backups/weekly/` | 发现较晚的逻辑错误 |
| **月备份（Grandfather）** | 每月 1 日 03:00 | 90 天 | `backups/monthly/` | 合规审计最低追溯期 |
| **迁移前备份** | 每次 migration 前 | 365 天 | `backups/pre-migrate/` | migration schema 变更回滚 |

**存储成本估算：** 数据库 50 MB → gzip 压缩后约 10 MB/份。日 7 + 周 4 + 月 3 = 14 份 × 10 MB = **140 MB**。COS 标准存储 ¥0.118/GB/月 → 月费不到 ¥0.02。**成本可忽略不计，多保留比少保留明智。**

### 8.2 备份安全

| 措施 | 说明 | 优先级 |
|------|------|:--:|
| **传输加密** | COS SDK 默认 HTTPS | ✅ 默认 |
| **SSE-COS 服务端加密** | COS 控制台 → 存储桶 → 安全管理 → 开启 SSE-COS（免费，一键） | 🔴 必须 |
| **备份 Bucket 独立** | 生产环境使用独立 COS Bucket，不与图片上传共用 | 🟡 推荐 |
| **CAM 最小权限** | 备份 Bucket 使用独立子账号，仅授予 PutObject + GetObject + DeleteObject + HeadBucket | 🟡 推荐 |

### 8.3 COS 生命周期自动清理

不依赖脚本手动删除——使用 COS 生命周期规则声明式管理：

| 路径前缀 | 自动删除时间 | 配置方式 |
|---------|:----------:|------|
| `backups/daily/` | 7 天后 | COS 控制台 → 存储桶 → 生命周期 → 新增规则 |
| `backups/weekly/` | 30 天后 | 同上 |
| `backups/monthly/` | 90 天后 | 同上 |
| `backups/pre-migrate/` | 365 天后 | 同上 |

> COS 生命周期是声明式的，没有 off-by-one 风险。同时自动清理上传中断的碎片文件（IncompleteMultipartUpload）。

### 8.4 手动执行备份

```bash
# 日备份
cd /opt/campus-market/server
node src/jobs/backup.js --type daily

# 迁移前备份
node src/jobs/backup.js --type pre-migrate

# 查看备份记录
cat /opt/campus-market/server/src/data/backup-log.json
```

### 8.5 恢复 SOP

#### 恢复决策树

```
发现问题
    │
    └─→ 确认故障范围
            │
            ├─ 单表误删/误改 → 场景 A：单表恢复（5-10 分钟，不停服）
            │
            ├─ 多表损坏/全库异常 → 场景 B：全库恢复（15-25 分钟，停服~15 分钟）
            │
            └─ 服务器完全丢失 → 场景 C：服务器重建（1-2 小时，完全停服）

    ⚠️ 无论哪种场景，第一步永远是：抢救备份当前状态
```

#### 场景 A：单表恢复（最常见，约 80% 的需求）

```bash
# 1. 确认受损表名
# 2. 抢救备份当前数据（必须！）
mysqldump -u campus_app -p campus_market --single-transaction \
  --skip-lock-tables | gzip > emergency-$(date +%Y%m%d-%H%M).sql.gz

# 3. 从 COS 下载最新日备份
# （使用 COS CLI 或控制台下载）
# 验证 MD5：md5sum -c campus_market-YYYY-MM-DD.sql.gz.md5

# 4. 提取单表数据
gunzip campus_market-YYYY-MM-DD.sql.gz
sed -n '/^-- Table structure for table .<表名>./,/^-- Table structure for table/p' \
  campus_market-YYYY-MM-DD.sql > <表名>_recover.sql

# 5. 导入单表（应用正常在线）
mysql -u campus_app -p campus_market < <表名>_recover.sql

# 6. 验证
mysql -u campus_app -p campus_market -e "SELECT COUNT(*) FROM <表名>;"
# 抽样检查 3-5 条记录

# 7. 清理
rm campus_market-*.sql *.sql.gz <表名>_recover.sql
```

> 预估耗时：**5–10 分钟**。服务全程在线。注意：如果目标表通过外键被其他表引用，需同步检查关联表完整性。

#### 场景 B：全库恢复

```bash
# 1. 立即停服
pm2 stop campus-market
curl -s http://localhost:3000/api/health  # 确认无响应

# 2. 抢救备份当前残留（必须！）
mysqldump -u campus_app -p campus_market --single-transaction --skip-lock-tables \
  | gzip > emergency-before-restore-$(date +%Y%m%d-%H%M).sql.gz

# 3. 选择恢复目标
#    优先级：最新日备份 → 前一天 → 上周备份 → 上月备份

# 4. 下载并验证
#    COS 下载 → md5sum 校验 → 不通过则换另一份

# 5. 执行恢复
gunzip campus_market-YYYY-MM-DD.sql.gz
mysql -u campus_app -p campus_market < campus_market-YYYY-MM-DD.sql

# 6. 启动服务
pm2 start ecosystem.config.js --env production
sleep 3

# 7. 验证恢复完整性
curl http://localhost:3000/api/health
# 核心表行数对比
mysql -u campus_app -p campus_market -e "
  SELECT 'users' AS tbl, COUNT(*) AS cnt FROM users
  UNION ALL SELECT 'products', COUNT(*) FROM products
  UNION ALL SELECT 'orders', COUNT(*) FROM orders
  UNION ALL SELECT 'reviews', COUNT(*) FROM reviews;"

# 8. 事后复盘
#    emergency-before-restore-*.sql.gz 保留至少 1 周
```

> 预估耗时：**15–25 分钟**，停服窗口约 15 分钟。RTO < 30 分钟可达成。凌晨 2–5 点操作几乎无感知。

#### 场景 C：服务器完全重建

```bash
# 1. 腾讯云控制台新建轻量服务器（同地域、同镜像）
# 2. 执行 setup.sh 环境初始化（10-15 分钟）
# 3. 从 COS 下载最新日备份 → 恢复数据库
# 4. git clone → npm ci → 配置 .env.production
# 5. 配置 Nginx + certbot 申请 SSL 证书
# 6. pm2 start → 验证 → DNS 切换（如有域名）
# 7. 全功能冒烟测试
```

> 预估耗时：**1–2 小时**。概率极低（腾讯云轻量服务器数据可靠性 99.9999999%），但 SOP 必须就位。参见 `server/setup.sh` 环境初始化脚本。

### 8.6 恢复演练

#### 自动验证（每周一 04:00，cron）

```
1. 从 COS 下载最新日备份 → 校验 MD5
2. 导入临时 MySQL 库 campus_market_verify
3. 执行：SELECT COUNT(*) FROM users, products, orders, reviews
4. 对比 backup-log.json 中上次记录的 COUNT 值（波动 ±5% 内正常）
5. 抽样验证：SELECT * FROM products ORDER BY RAND() LIMIT 5
6. 记录验证结果 → 删除临时库
```

#### 人工演练（每季度一次）

| 演练项目 | 内容 | 验收标准 |
|---------|------|---------|
| 场景 A | 模拟误删 products 表 10 行数据，执行单表恢复 | 10 分钟内恢复，无停服 |
| 场景 B | 模拟 DROP TABLE orders 后全库恢复 | 25 分钟内完成，RTO < 30min |
| 复盘 | 记录实际耗时、遇到的问题、改进点 | SOP 持续优化 |

> 只有人工演练过，RTO < 30min 才不是文档上的空话。如果从未演练，真实故障时大概率超时。

### 9.1 监控体系总览

| 监控层 | 工具 | 检查频率 | 告警方式 |

| 健康检查 | UptimeRobot（免费） | 每 5 分钟 | 微信/邮件通知 |

### 9.2 健康检查（UptimeRobot 配置）

UptimeRobot 是免费的外部监控服务，从全球多个节点探测你的 API。

**配置步骤：**
1. 注册 [uptimerobot.com](https://uptimerobot.com)（免费套餐：50 个监控项，5 分钟间隔）
2. 添加监控 → HTTP(s) → URL: `https://<你的域名>/api/health`
3. 监控间隔：5 分钟
4. 告警联系人：微信（通过 UptimeRobot 微信公众号）/ 邮件
5. 超时设置：15 秒（5 分钟内连续 2 次超时判定为 down）

**预期响应：**
```json
{"status":"ok","timestamp":"2026-06-04T10:00:00Z","db":"connected"}
```

### 9.3 告警阈值一览

| 监控项 | 阈值 | 级别 | 动作 |
|--------|------|:--:|------|
| 健康检查无响应 | 连续 2 次（10 分钟） | P0 | UptimeRobot 推送告警 → 立即排查 |
| 进程内存 > 400 MB | `perf.js` 采集 | warn | 写入 warn 日志 |
| 进程内存 > 500 MB | PM2 检测 | 自动 | PM2 自动重启进程 |
| 慢查询 > 200 ms | access-log 检测 | warn | 写入 warn 日志（含 SQL 文本、参数摘要、耗时） |
| 慢查询 > 1,000 ms | access-log 检测 | error | 写入 error 日志（告警级别） |
| 慢查询 > 3,000 ms | mysql2 驱动超时 | 终止 | 抛出 `ERR_DB_QUERY_TIMEOUT` → 返回 6999 |
| 错误率 > 10/min | winston error 计数 | 人工 | 排查 error log |
| 备份失败 | `backup.js` exit(1) | 人工 | error log 中搜索 `[BACKUP_FAIL]` |
| IM 消息永久失败 | `failed_system_messages` `status='permanent_failed'` | 人工 | 检查表记录数 |
| 磁盘使用 > 80% | `df -h` | 人工 | 清理旧日志/备份 |

## 十、故障处理手册

### 10.1 故障分级

| 级别 | 定义 | 响应时间 | 示例 |
|:----:|------|:------:|------|
| **P0** | 全站不可用 | 立即（目标 < 30 min） | 进程崩溃未重启、MySQL 宕机、服务器到期 |
| **P1** | 核心链路中断 | 1 小时内 | 无法发布商品/下单/聊天、IM 欠费停服、COS 密钥过期 |
| **P2** | 非核心功能异常 | 下一工作日 | 搜索慢、部分页面报错、定时任务未执行 |
| **P3** | 体验降级 | 下一迭代 | 图片加载慢、页面偶发抖动 |

### 10.2 常见故障处理 SOP

#### P0-01：服务无响应

```
症状：curl /api/health 无响应或连接拒绝

排查流程：
1. SSH 登录服务器（如无法登录，先检查服务器是否到期/欠费）
2. pm2 status                           → 确认进程是否运行
3. pm2 logs campus-market --lines 50    → 查看最近日志
4. systemctl status mysql               → 确认 MySQL 是否运行
5. systemctl status nginx               → 确认 Nginx 是否运行

常见原因与解决：
• PM2 进程未启动
  → pm2 start ecosystem.config.js --env production
• MySQL 服务停止
  → systemctl start mysql
• 服务器磁盘满
  → df -h → 清理旧日志和临时文件
• 服务器到期/欠费
  → 登录腾讯云控制台续费
```

#### P0-02：数据库连接失败

```
症状：API 返回 6999，error log 中大量 ECONNREFUSED 或 ER_CON_COUNT_ERROR

排查流程：
1. systemctl status mysql               → MySQL 进程状态
2. mysql -u campus_app -p -e "SELECT 1" → 手动连接测试
3. mysql -u root -p -e "SHOW PROCESSLIST" → 查看连接数

常见原因与解决：
• MySQL 未运行
  → systemctl start mysql
• 连接数耗尽
  → mysql -u root -p -e "SET GLOBAL max_connections = 100;"
  → 排查是否有慢查询堆积
• 密码/权限变更
  → 检查 .env.production 中 DB_PASSWORD 是否正确
  → mysql -u root -p -e "SELECT user,host FROM mysql.user;"
```

#### P1-01：腾讯云 IM 服务异常

```
症状：聊天消息发送失败、IM SDK 初始化失败

排查流程：
1. 登录腾讯云 IM 控制台 → 查看应用状态
2. 检查 IM 应用是否欠费/停服
3. 检查 IM_SDK_APP_ID 和 IM_SECRET_KEY 是否过期

常见原因与解决：
• IM 应用欠费
  → 登录腾讯云控制台充值
• 免费档 100 DAU 超限
  → 升级付费套餐（约 ¥500/月）或等待次日重置
• SecretKey 在控制台被重置
  → 更新 .env.production 中的 IM_SECRET_KEY → pm2 reload
```

#### P1-02：COS 图片上传失败

```
症状：用户发布商品时图片上传失败，返回 403/6001

排查流程：
1. 登录腾讯云 COS 控制台 → 检查存储桶状态
2. 检查 COS_SECRET_ID 和 COS_SECRET_KEY 是否有效
3. 检查 CAM 子账号权限是否被修改

常见原因与解决：
• COS 密钥过期/被重置
  → 腾讯云 API 密钥管理 → 更新密钥 → 更新 .env.production → pm2 reload
• 存储桶被删除
  → 检查是否有生命周期规则误删了 bucket（与备份 bucket 规则区分）
• STS 临时密钥获取失败
  → 检查 error log 中 COS 相关错误
```

#### P2-01：定时任务未执行

```
症状：订单逾期未自动确认、备份文件缺失

排查流程：
1. pm2 show campus-market | grep "exec mode"  → 确认 fork 模式
2. pm2 logs campus-market | grep "cron"       → 确认 cron 注册日志
3. 检查 .env.production 中 ENABLE_CRON=true
4. 手动触发验证：
   node -e "const cron = require('node-cron'); console.log(cron.getTasks());"

常见原因与解决：
• PM2 为 cluster 模式 → 改为 fork 模式（修改 ecosystem.config.js）
• ENABLE_CRON=false → 修改 .env.production → pm2 reload
• node-cron 任务注册失败 → 查看启动日志中的错误
```

#### P2-02：备份文件持续缺失

```
症状：COS 中连续多日无新备份文件

排查流程：
1. tail -100 /opt/campus-market/server/logs/error-*.log | grep "BACKUP_FAIL"
2. 手动执行备份：
   cd /opt/campus-market/server && node src/jobs/backup.js --type daily
3. 观察手动执行的输出和错误

常见原因与解决：
• COS 密钥过期 → 更新 .env.production 中的 COS 密钥
• COS 存储桶不存在 → COS 控制台检查
• MySQL dump 权限不足 → 确认 campus_app 有 SELECT 权限
• 磁盘空间不足 → df -h 检查
```

### 10.3 压力与容量

| 指标 | 上限 | 超过后的措施 |
|------|:--:|------|
| 峰值 QPS | 300 | 超过后考虑：PM2 cluster 模式 + scheduler 独立进程（见 §11.1） |
| 并发 TCP 连接 | 200 | Node.js event loop 轻松应对，超限后 Nginx 排队 |
| MySQL 连接池利用率 | 70%（活跃 ≤ 7/10） | 调高 `connectionLimit` 或排查慢查询堆积 |
| 磁盘使用率 | 80% | 清理旧日志、删除临时文件 |

## 十一、扩容与演进

### 11.1 MVP 后架构演进（多实例部署时）

当前 MVP 阶段：PM2 fork 单进程（`instances: 1`）。当 QPS 逼近 300 或需要零停机部署时，可升级为：

```js
// ecosystem.config.js（演进版）
module.exports = {
  apps: [
    {
      name: 'api',
      script: 'app.js',
      instances: 2,              // 多实例处理 HTTP
      exec_mode: 'fork',
      env: { ENABLE_CRON: 'false', PORT: 3000 }
    },
    {
      name: 'scheduler',
      script: 'scheduler.js',    // 独立定时任务进程
      instances: 1,              // 单实例！防止重复执行
      exec_mode: 'fork',
      env: { ENABLE_CRON: 'true', PORT: 3001 }
    }
  ]
};
```

**关键约束：** scheduler 必须 `instances: 1`，否则 cron 重复执行。后续可接入 Redis 分布式锁进一步保证唯一性。

### 11.2 RPO 改善路线

| 阶段 | RPO | 方案 |
|------|:--:|------|
| **MVP（当前）** | < 24 小时 | 每日全量 `mysqldump` |
| **第一阶段（MVP 后）** | < 1 小时 | 开启 MySQL binlog（`binlog_format=ROW`），恢复 = 全量 + binlog 重放到指定时间点 |
| **第二阶段（成熟期）** | < 5 分钟 | binlog 准实时上传 COS（每 5 分钟增量同步），支持 PITR。需腾讯云 CDB 或独立备份服务器 |

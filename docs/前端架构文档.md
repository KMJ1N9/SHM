# 1. 视觉风格 (Visual Style)

## 1.1 设计理念

### 1.1.1 核心哲学

**简约至上，信息高效。** 校园二手交易小程序面向广应科肇庆校区在校学生，核心诉求是快速浏览商品、快速决策、快速沟通。视觉设计的首要目标是减少认知负荷，让用户将注意力集中在商品内容上，而非界面装饰。

设计遵循三大原则：

| 原则 | 说明 | 具体体现 |
|------|------|---------|
| 简约至上 | 以灰色系为主色调，减少视觉干扰 | 页面背景统一为浅灰，不使用装饰性图案或渐变背景 |
| 信息高效 | 紧凑布局，最大化信息密度 | 瀑布流双列布局，卡片信息层级分明，关键数据（价格/成色）视觉突出 |
| 交互友好 | 舒缓过渡动画，清晰操作反馈 | 所有可交互元素均有 hover/active 态，页面切换 300ms 过渡，加载/错误/空数据均有对应 UI 状态 |

### 1.1.2 设计价值观

- **清晰可读**：正文与背景对比度 ≥ 4.5:1（WCAG AA），标题字重 600，正文行高 1.5
- **一致性**：全局统一的设计语言——同一组件在所有页面中外观和行为一致，同一色值不出现多种写法
- **易用性**：触控区域最小 44rpx × 44rpx（符合微信小程序人机界面指南），关键操作（"我想要"/"聊一聊"）拇指热区优先

### 1.1.3 竞品参考

设计参考了闲鱼（瀑布流 + 卡片布局）、悠悠有品（侧边栏筛选器），但做了针对校园场景的特化——用蓝色替代橙色/黄色主色调，匹配校园的理性、可信赖感。

## 1.2 色彩系统

色彩系统分为四层：灰色系（主色调，占 70%+ 面积）、蓝色系（点缀色，引导操作）、功能色（状态传达）、透明度（层级表达）。所有色值必须在 tokens.scss 中以 SCSS 变量定义，禁止在组件中硬编码原始色值。

### 1.2.1 灰色系 — 主色调

灰色系承担页面结构、信息层级、视觉呼吸感的职责。遵循"越重要的内容越深"原则。

| 名称 | 色值 | CSS 变量 | SCSS 变量 | 用途 |
|------|------|---------|-----------|------|
| 页面背景 | `#F8F8F8` | `--color-bg` | `$color-bg` | 所有页面的底层背景色。不直接用于卡片或输入框 |
| 卡片/弹窗背景 | `#FFFFFF` | `--color-surface` | `$color-surface` | 卡片、列表项、弹窗、导航栏、输入框的背景 |
| 分割线/占位区 | `#F0F0F0` | `--color-divider` | `$color-divider` | 1px 分割线、图片占位背景、禁用态背景 |
| 辅助文字/占位符 | `#999999` | `--color-muted` | `$color-muted` | 输入框 placeholder、辅助说明文字、图标默认色 |
| 正文文字 | `#666666` | `--color-body` | `$color-body` | 所有正文段落、列表项主文本 |
| 标题文字 | `#333333` | `--color-title` | `$color-title` | 页面标题、卡片标题、导航栏标题 |

**使用规则：**
- 正文文字必须在白色背景上使用 `#666666`，不得使用 `#999999`（对比度不足）
- 分割线统一使用 `#F0F0F0`，1px 实线，不允许使用其他灰色替代

### 1.2.2 蓝色系 — 点缀色

蓝色系约占页面面积的 5%-8%，用于引导用户注意力到关键操作和当前状态。

| 名称 | 色值 | CSS 变量 | SCSS 变量 | 用途 |
|------|------|---------|-----------|------|
| 主蓝 | `#4A90D9` | `--color-primary` | `$color-primary` | 主按钮、链接文字、选中态图标/Tab、导航栏背景 |
| 深蓝 | `#3A7BC8` | `--color-primary-dark` | `$color-primary-dark` | 按钮 press 态、按钮 hover 态 |
| 浅蓝 | `#E8F1FB` | `--color-primary-light` | `$color-primary-light` | 选中态背景（如筛选标签选中）、系统提示 Banner 背景 |
| 蓝色渐变 | `#4A90D9 → #3A7BC8` | — | `$color-primary-gradient` | 重要主按钮背景、数据看板图表 |

**渐变使用规范：**
```scss
// 135deg 对角线渐变，仅用于以下场景：
// 1. 主按钮（大/中尺寸）
// 2. 首页导航栏背景
// 3. 管理后台数据图表填充
// 禁止在文字、图标、边框上使用渐变
background: linear-gradient(135deg, #4A90D9, #3A7BC8);
```

### 1.2.3 功能色

功能色用于传达操作结果和状态，每种颜色有明确的语义，不可混用。

| 名称 | 色值 | CSS 变量 | SCSS 变量 | 用途 |
|------|------|---------|-----------|------|
| 成功 | `#52C41A` | `--color-success` | `$color-success` | 成功 Toast、完成状态标签、提交成功图标 |
| 警告 | `#FAAD14` | `--color-warning` | `$color-warning` | 警告提示、待处理状态标签、信誉分低于 60 提示 |
| 错误 | `#FF4D4F` | `--color-error` | `$color-error` | 错误 Toast、删除按钮、表单验证错误、封禁操作 |
| 信息 | `#4A90D9` | `--color-info` | `$color-info` | 信息提示、帮助文案 |

**功能色使用规则：**
- 同一界面中功能色出现种类 ≤ 3 种
- 红色（`#FF4D4F`）仅用于错误/删除/封禁，不得用于价格标签或其他装饰
- 价格文字不使用功能色——价格使用 `#FF4D4F`（红色）表示，原价使用 `#999999` 划线

### 1.2.4 透明度

透明度用于表达层级关系和可用性状态，不使用额外的灰色色值。

| 透明度 | SCSS 变量 | 用途 |
|--------|-----------|------|
| 80% | `$opacity-overlay` | 弹窗/侧边栏遮罩层背景（`rgba(0,0,0,0.8)`） |
| 60% | `$opacity-disabled` | 禁用按钮、禁用输入框、不可操作元素 |
| 40% | `$opacity-placeholder` | 图片加载中占位、骨架屏 |

## 1.3 字体系统

### 1.3.1 字体族

| 用途 | 字体栈 | SCSS 变量 | 说明 |
|------|--------|-----------|------|
| 中文正文/标题 | `"PingFang SC", "Noto Sans SC", sans-serif` | `$font-chinese` | iOS 用 PingFang SC，Android 回退 Noto Sans SC，均为系统内置字体，无需下载 |
| 英文/数字（价格/时间） | `"SF Mono", "DIN", monospace` | `$font-mono` | 等宽字体，确保价格数字对齐、时间戳可读性。iOS 用 SF Mono，Android/其他回退 DIN |

**注意：** 小程序不支持自定义字体文件（包体积限制 2MB），统一使用系统字体栈。严禁引入 Web Font。

### 1.3.2 字号层级

基于 375px 逻辑宽度，1px = 2rpx。所有字号使用 rpx 单位以确保多设备适配。

| 层级 | 字号 (rpx) | 对应 px | SCSS 变量 | 字重 | 用途 |
|------|-----------|---------|-----------|------|------|
| 特大 | 64rpx | 32px | `$text-3xl` | 600 | 数据看板核心指标数字 |
| 超大 | 48rpx | 24px | `$text-2xl` | 600 | 价格展示（商品详情页主价格） |
| 页面标题 | 40rpx | 20px | `$text-xl` | 600 | 页面级标题（如弹窗标题、设置页 section 标题） |
| 卡片标题 | 36rpx | 18px | `$text-lg` | 500 | 商品卡片名称、订单卡片标题 |
| 正文 | 28rpx | 14px | `$text-base` | 400 | 所有正文段落、列表项文本、按钮文字、输入框文字 |
| 辅助文字 | 24rpx | 12px | `$text-sm` | 400 | 辅助说明、时间戳、卖家信息、字数统计 |
| 标签/状态 | 22rpx | 11px | `$text-xs` | 400 | 分类标签、状态标签、角标 |

### 1.3.3 字重规范

| 字重 | SCSS 变量 | 适用场景 |
|------|-----------|---------|
| 600 (Bold) | `$weight-bold` | 页面标题、卡片标题、价格数字、核心指标 |
| 500 (Medium) | `$weight-medium` | 卡片标题（商品卡片内）、按钮文字 |
| 400 (Regular) | `$weight-regular` | 正文、辅助文字、标签、输入框内容 |

**限制：** 小程序不支持自定义字重文件。系统字体仅支持 400 和 600 两档字重（PingFang SC Regular / Semibold），500 在大多数设备上会回退为 400，但保留 SCSS 变量以便后续扩展。

### 1.3.4 行高

| 场景 | 行高 | 说明 |
|------|------|------|
| 正文段落 | 1.5 | 所有 body 文本、描述文本 |
| 标题 | 1.3 | 标题通常不换行，1.3 提供合理的上下间距 |
| 单行文本 | 1.0 | 按钮文字、标签文字 |

## 1.4 间距系统 (8pt Grid)

本项目采用 **8pt 网格系统**——所有间距值为 8 的倍数（即 16rpx 的倍数）。这保证了视觉节奏的一致性，并降低了间距决策成本。

### 1.4.1 间距尺度

| 级别 | rpx 值 | 对应 px | SCSS 变量 | 用途 |
|------|--------|---------|-----------|------|
| xs | 8rpx | 4px | `$space-xs` | 图标与文字间距、标签内 padding |
| sm | 16rpx | 8px | `$space-content` | 元素内部间距（如卡片内 padding、表单项间距） |
| md | 24rpx | 12px | `$space-card` | 卡片之间间距、列表项间距 |
| lg | 32rpx | 16px | `$space-page` | 页面左右边距、section 之间间距 |
| xl | 48rpx | 24px | — | 大 section 之间的留白（如用户信息区与功能列表区间距） |
| 2xl | 64rpx | 32px | — | 页面顶部/底部大留白（如登录页 Logo 区域） |

### 1.4.2 间距使用规则

- **页面边距**：左右统一使用 `$space-page`（32rpx），不得为特定页面单独调整
- **卡片内边距**：统一 `$space-content`（16rpx），商品卡片、订单卡片、用户卡片均遵循此规则
- **卡片间距**：瀑布流/列表中的卡片之间统一 `$space-card`（24rpx）
- **组件内元素间距**：图标与文字 8rpx，标签之间 8rpx，表单行之间 16rpx

**反例：** 禁止出现 10rpx、15rpx、30rpx 等非 8 的倍数的间距值。

## 1.5 圆角系统

| 元素类型 | rpx 值 | 对应 px | SCSS 变量 | 说明 |
|---------|--------|---------|-----------|------|
| 标准圆角 | 16rpx | 8px | `$radius-card` | 卡片、按钮、图片缩略图、输入框、标签 |
| 大圆角 | 24rpx | 12px | `$radius-modal` | 弹窗/模态框 |
| 全圆角 | 9999rpx | — | `$radius-full` | 圆形头像、圆形按钮、胶囊标签 |

**规则：** 所有可交互元素的圆角统一为 16rpx。弹窗/模态框使用 24rpx 以区分于页面内元素。圆形元素（头像、部分标签）使用 9999rpx。

## 1.6 阴影/高程系统

阴影用于表达元素在 Z 轴上的层级关系。共三级：

| 层级 | CSS 值 | SCSS 变量 | 用途 |
|------|--------|-----------|------|
| 卡片悬浮 | `0 4rpx 16rpx rgba(0,0,0,0.06)` | `$shadow-card` | 商品卡片、订单卡片、用户卡片（z-index: 1-10） |
| 按钮悬浮 | `0 4rpx 8rpx rgba(74,144,217,0.30)` | `$shadow-button` | 主按钮（含蓝色半透明，强调操作感） |
| 弹窗/遮罩 | `0 16rpx 48rpx rgba(0,0,0,0.15)` | `$shadow-modal` | 模态弹窗、侧边栏筛选器（z-index: 100+） |

**规则：**
- 同一页面中不使用超过 2 种阴影层级
- 卡片阴影仅用于浮于页面背景之上的元素（如卡片、列表项），不用于导航栏或 TabBar
- 弹窗阴影必须配合 80% 不透明度遮罩层使用

## 1.8 动效规范

### 1.8.1 过渡动画

| 场景 | 动画属性 | 时长 | 缓动函数 | 说明 |
|------|---------|------|---------|------|
| 页面切换 | opacity + translateX | 300ms | `ease-out` | 页面进入：从右滑入 + 淡入；页面返回：向右滑出 + 淡出 |
| 列表加载 | opacity + translateY | 400ms | `ease-out` | 列表项逐个出现，每项延迟 50ms（stagger） |
| 模态框出现 | scale(0.9→1) + opacity | 250ms | `ease-out` | 弹窗从中心放大出现 |
| 模态框消失 | scale(1→0.95) + opacity | 200ms | `ease-in` | 弹窗缩小消失 |
| 侧边栏 | translateX | 250ms | `ease-out` | 从右侧滑入 |
| 下拉刷新 | translateY + 弹性 | 500ms | 自定义弹性曲线 | 回弹效果 |
| Toast 提示 | opacity + translateY | 300ms | `ease-out` | 从底部弹出，2s 后自动消失 |

### 1.8.2 触摸反馈（微交互）

| 交互 | 效果 | CSS 实现 |
|------|------|---------|
| 按钮按下 | scale(0.98) + 颜色加深 + 阴影内收 | `:active { transform: scale(0.98); }` |
| 卡片点击 | scale(0.98) + 阴影减弱 | `:active { transform: scale(0.98); box-shadow: none; }` |
| 列表项点击 | 背景变浅蓝 | `:active { background-color: #E8F1FB; }` |
| 开关切换 | 滑块平移动画 | `transition: transform 0.2s ease` |

**规则：** 所有 touch 反馈必须使用 CSS `:active` 伪类，禁止使用 JS 监听 touch 事件实现基础反馈。`transform: scale()` 优于修改 `width/height`（性能更好，不触发 reflow）。

### 1.8.3 动画性能约束

- 仅对 `transform` 和 `opacity` 做动画——这两个属性不触发 layout/paint，仅触发 composite
- 禁止对 `width`、`height`、`top`、`left`、`margin`、`padding` 做过渡动画
- 复杂动画（如骨架屏闪烁）使用 CSS animation，不用 JS setInterval
- 所有动画时长 ≤ 500ms

# 2. 技术方案 (Technical Solution)

## 2.1 框架选型：uni-app (Vue 3 Composition API)

### 2.1.1 选型理由

| 考量维度 | 决策 | 理由 |
|---------|------|------|
| 开发效率 | uni-app | 一套代码编译为微信小程序；Vue 3 Composition API 提供更好的逻辑复用和 TypeScript 支持 |
| 目标平台 | 仅微信小程序 | MVP 阶段只做微信小程序，但保留 uni-app 跨端能力（后续可编译为 H5/App） |
| 团队规模 | 单人开发 | uni-app 社区成熟，HBuilder X 开发体验好，问题排查成本低 |
| 未来扩展 | 预留跨端 | 如需扩展到其他平台（支付宝小程序、H5），仅需调整少量平台差异代码 |

### 2.1.2 Composition API 规范

所有页面和组件必须使用 `<script setup>` + Composition API。禁止使用 Options API。

```vue
<!-- 正确：<script setup> + Composition API -->
<script setup lang="ts">
import { ref, computed, onMounted } from 'vue'

interface ProductInfo {
  id: number
  name: string
  price: number
}

const products = ref<ProductInfo[]>([])
const loading = ref(false)

const displayCount = computed(() => products.value.length)

onMounted(async () => {
  loading.value = true
  // 加载数据...
  loading.value = false
})
</script>
```

```vue
<!-- 禁止：Options API -->
<script lang="ts">
export default {
  data() { return { products: [] } },
  computed: { displayCount() { return this.products.length } }
}
</script>
```

### 2.1.3 TypeScript 集成

- 所有 `.vue` 文件的 `<script>` 标签必须标注 `lang="ts"`
- 所有 `.ts` 文件必须定义完整的类型，禁止 `any`
- Props 和 Emits 使用 TypeScript 类型推导：

```vue
<script setup lang="ts">
// Props 定义
interface Props {
  productId: number
  showPrice?: boolean
}
const props = withDefaults(defineProps<Props>(), { showPrice: true })

// Emits 定义
interface Emits {
  (e: 'buy', productId: number): void
  (e: 'favorite', productId: number): void
}
const emit = defineEmits<Emits>()
</script>
```

## 2.2 状态管理：Pinia

### 2.2.1 Store 架构

本项目使用 3 个 Pinia Store，遵循"按领域拆分"原则：

| Store | 文件 | 职责 | 数据范围 |
|-------|------|------|---------|
| `useUserStore` | `store/user.js` | 用户认证状态（token/手机号/个人信息）、登录/登出逻辑 | 全局持久化（`uni.setStorageSync`） |
| `useChatStore` | `store/chat.js` | IM 连接状态、未读消息计数、会话列表缓存 | 全局，内存级缓存 |
| `useAppStore` | `store/app.js` | 全局 UI 状态（网络状态、全局 loading、错误信息） | 全局，内存级 |

```js
// store/user.js 示例结构
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'

export const useUserStore = defineStore('user', () => {
  // State
  const token = ref<string | null>(null)
  const refreshToken = ref<string | null>(null)
  const userInfo = ref<UserInfo | null>(null)

  // Getters
  const isLoggedIn = computed(() => !!token.value)
  const isAdmin = computed(() => userInfo.value?.role === 'admin')

  // Actions
  async function login(phoneCode: string) { /* ... */ }
  async function logout() { /* ... */ }
  function restoreSession() { /* 从 storage 恢复 */ }

  return { token, refreshToken, userInfo, isLoggedIn, isAdmin, login, logout, restoreSession }
})
```

### 2.2.2 Store 使用规则

- **页面中使用**：通过 `storeToRefs` 解构响应式数据
- **持久化**：仅 `useUserStore` 的 token/userInfo 写入 `uni.setStorageSync`；`useChatStore` 和 `useAppStore` 仅内存存储
- **禁止**：在 Store 中直接操作 DOM、调用 `uni.navigateTo`（Side effect 应在页面层处理）

### 2.4.1 请求封装

基于 `uni.request` 封装统一的 HTTP 客户端：

```
api/
├── client.js       ← 基础 HTTP 客户端（拦截器、Token 附加、错误处理、请求日志）
├── auth.js         ← 认证模块（login/logout/refreshToken/getUserInfo）

└── admin.js        ← 管理后台（数据看板/用户管理/商品管理/工单处理）
```

### 2.4.2 拦截器策略

```js
// api/client.js 核心逻辑
const BASE_URL = 'https://api.campus.example.com'

// 请求拦截器
function request(config) {
  // 1. 附加 Access Token
  const userStore = useUserStore()
  if (userStore.token) {
    config.header = {
      ...config.header,
      'Authorization': `Bearer ${userStore.token}`
    }
  }

  // 2. 请求日志（开发环境）
  if (__DEV__) {
    console.log(`[HTTP] ${config.method} ${config.url}`)
  }

  return config
}

// 响应拦截器
async function response(res) {
  // 1. 统一错误处理
  if (res.statusCode === 401) {
    // Token 过期 → 尝试刷新
    const refreshed = await refreshAccessToken()
    if (refreshed) {
      // 重试原请求
      return request(res.config)
    } else {
      // 刷新失败 → 清除登录态 → 跳转登录页
      const userStore = useUserStore()
      await userStore.logout()
      uni.reLaunch({ url: '/pages/auth/login' })
    }
  }

  // 2. 业务错误码处理
  if (res.data.code !== 0) {
    const message = ERROR_CODE_MAP[res.data.code] || '请求失败'
    uni.showToast({ title: message, icon: 'none' })
    return Promise.reject(res.data)
  }

  return res.data
}
```

### 2.4.3 无感刷新 Token 机制

```
请求 → 401? → 是 → refreshToken → 成功? → 重试原请求
                → 否 → 失败? → 重试次数 < 3? → 是 → 500ms 后重试
                                        → 否 → 清除登录态
                → 否 → 正常返回
```

- Access Token 有效期：2 小时
- Refresh Token 有效期：7 天
- 刷新时使用互斥锁（`refreshingPromise`），避免并发 401 触发多次刷新
- 刷新成功后，等待队列中的请求全部重试

```js
// utils/cos.js
export function validateImage(file) {
  // 1. 格式校验（文件头魔数）
  const validTypes = ['image/jpeg', 'image/png', 'image/webp']
  if (!validTypes.includes(file.type)) {
    uni.showToast({ title: '仅支持 JPG/PNG/WebP 格式', icon: 'none' })
    return false
  }

  // 2. 大小校验
  if (file.size > 5 * 1024 * 1024) {
    uni.showToast({ title: '图片大小不能超过 5MB', icon: 'none' })
    return false
  }

  // 3. 尺寸校验
  uni.getImageInfo({ src: file.path, success: (info) => {
    if (info.width === 0 || info.height === 0) {
      uni.showToast({ title: '图片已损坏，请重新选择', icon: 'none' })
      return false
    }
  }})

  return true
}
```

# 3. UI 组件库 (UI Component Library)

## 3.1 组件库选型：零第三方 UI 库

### 3.1.1 决策

本项目**不引入任何第三方 UI 组件库**（如 uni-ui、uView、Vant Weapp）。所有 UI 组件均为自定义开发。

### 3.1.2 理由

| 考量 | 分析 |
|------|------|
| 包体积 | 微信小程序主包限制 2MB。第三方 UI 库通常 500KB-1MB，加上业务代码后接近上限 |
| 设计一致性 | 第三方组件库有自身的默认样式，覆盖成本高，容易出现"一半 UI 库样式、一半自定义样式"的视觉割裂 |
| 维护成本 | 依赖第三方库的版本升级可能引入 breaking changes，单人开发应尽量减少外部依赖 |
| 实际需求 | 本项目仅需 6 个通用组件 + 15 个页面组件，组件数量少，自研成本低 |
| 学习成本 | uni-app 的第三方组件库通常需要额外的适配和配置，不如直接写 Vue 组件 |

### 3.1.3 例外

以下**非 UI 组件**的依赖允许引入：
- `pinia`（状态管理）
- `tim-wx-sdk`（腾讯云 IM SDK）
- `cos-wx-sdk-v5`（腾讯云 COS SDK）

- 每个格子 160rpx × 160rpx，圆角 16rpx
- "+"格子：`$color-divider` 背景 + `$color-muted` 加号图标
- 已上传格子：缩略图填充 + 右上角删除按钮（半透明圆形，`×` 图标）
- 当 `images.length < maxCount` 时显示"+"格子；等于 maxCount 时隐藏

**行为规则：**
- 点击"+"触发 `uni.chooseImage`，支持拍照/相册
- 上传前校验（格式/大小/尺寸，见 §2.6.2）
- 上传中显示格子内进度条（蓝色渐变）
- 上传失败格子显示红色边框 + 刷新图标，点击重新上传
- 长按图片可拖拽排序（使用 `movable-view`）
- 点击已上传图片可全屏预览（`uni.previewImage`）

### 3.3.4 StarRating — 五星评分

**用途：** 互评页面（沟通态度/守时程度/描述一致度三个维度评分）、商品详情页卖家信誉分展示（只读模式）。

**Props:**

| Prop | 类型 | 必填 | 默认值 | 说明 |
|------|------|:--:|--------|------|
| `value` | `number` | ✅ | `0` | 当前评分（0-5，支持半星，如 3.5） |
| `readonly` | `boolean` | ❌ | `false` | 只读模式（仅展示，不可交互） |
| `size` | `number` | ❌ | `40` | 每颗星尺寸（rpx） |
| `count` | `number` | ❌ | `5` | 星星总数 |

**Events:**

| Event | 参数 | 说明 |
|-------|------|------|
| `update:value` | `rating: number` | 评分变更（仅交互模式触发） |

**内部结构：**

```
┌──┐ ┌──┐ ┌──┐ ┌──┐ ┌──┐
│★│ │★│ │★│ │☆│ │☆│    ← 3/5 星（已选中蓝色填充，未选中灰色空心）
└──┘ └──┘ └──┘ └──┘ └──┘
```

- 每颗星尺寸由 `size` prop 控制
- 选中态：`$color-primary` 填充
- 未选中态：`$color-divider` 描边，无填充
- 只读模式：不响应点击，鼠标样式为 default
- 交互模式：hover 时高亮到当前位置的星星（预选效果），点击确认选择

**行为规则：**
- 点击第 N 颗星 → 评分为 N
- 再次点击同一颗星 → 取消评分（归零）
- 从左到右连续选中（不可跳过中间星）
- 半星支持：点击星星左半部分 → N - 0.5 分；右半部分 → N 分

### 3.3.5 AppNavbar — 自定义导航栏

**用途：** 替代微信原生导航栏，提供更灵活的自定义能力（如透明导航栏、带搜索的导航栏、带操作按钮的导航栏）。

**Props:**

| Prop | 类型 | 必填 | 默认值 | 说明 |
|------|------|:--:|--------|------|
| `title` | `string` | ❌ | `''` | 导航栏标题 |
| `showBack` | `boolean` | ❌ | `true` | 是否显示返回按钮 |
| `backgroundColor` | `string` | ❌ | `'#4A90D9'` | 导航栏背景色 |
| `titleColor` | `string` | ❌ | `'#FFFFFF'` | 标题文字颜色 |
| `transparent` | `boolean` | ❌ | `false` | 是否透明（背景渐变到透明） |

**Slots:**

| Slot | 说明 |
|------|------|
| `default` | 自定义标题区域（替代 title prop） |

|-------|------|------|

- 自动适配状态栏高度（`uni.getSystemInfoSync().statusBarHeight`）

**用途：** 列表无数据、网络错误、搜索无结果等场景的统一空状态展示。

**默认文案映射：**

| type | 图标 | 默认文案 |
|------|------|---------|
| `empty` | 📦 | 暂无内容 |
| `network-error` | 🌐 | 网络异常，请检查网络后重试 |
| `search-empty` | 🔍 | 未找到相关商品，换个关键词试试 |
| `order-empty` | 📋 | 暂无订单 |

**Events:**

| Event | 参数 | 说明 |
|-------|------|------|
| `retry` | — | 点击重试按钮 |

**内部结构：**

```
┌─────────────────────────────────┐
│                                 │
│         [图标]  (80rpx)         │
│                                 │
│      提示文字 (正文, 深灰)       │
│                                 │
│    ┌───────────────────┐        │
│    │      重试          │        │  ← 仅在 showRetry=true 时显示
│    └───────────────────┘        │
│                                 │
└─────────────────────────────────┘
```

## 3.4 组件设计原则

1. **Props Down, Events Up**：父组件通过 Props 向子组件传递数据，子组件通过 Events 向父组件通信。禁止子组件直接修改 Props。
2. **单一职责**：每个组件只负责一个明确定义的功能。如果一个组件的职责超过 2 个，拆分为子组件。
3. **无副作用**：组件不应直接调用 API 或修改 Store。数据获取和状态变更由页面层负责，组件仅做展示和用户交互。
4. **可组合**：组件之间可以嵌套组合，但不形成强耦合。例如 ProductCard 内部可以使用 StarRating，但不依赖 ProductCard 的特定上下文。
5. **样式隔离**：所有组件使用 `scoped` 样式 + BEM 命名，确保样式不泄漏到父/子组件。

## 3.5 组件通信模式

```
父组件 (页面)
  │
  ├─ Props (数据) ──────→ 子组件
  │
  ├─ Events (操作) ←────── 子组件
  │
  └─ provide/inject ────→ 深层嵌套子组件（仅跨 2 层以上时使用）
```

**禁止的通信方式：**
- 子组件直接访问父组件的 ref
- 组件间通过 `uni.$on` / `uni.$emit` 全局事件总线通信
- 组件直接修改 Pinia Store（Store 变更由页面层或 actions 执行）

## 4.4 页面路由配置说明

`pages.json` 中的 `pages` 数组定义了所有页面的路由、窗口样式和下拉刷新配置。

**路由规则：**
- 首页（`pages/index/index`）排在 `pages` 数组的第一位，作为应用入口页

- 404 页面（`not-found`）和网络异常页（`network`）在 `App.vue` 中通过全局拦截自动跳转

**全局样式配置：**

- 商品详情页和登录页使用白色导航栏（`navigationBarBackgroundColor: "#FFFFFF"`），其他页面使用蓝色导航栏
- 登录页使用 `navigationStyle: "custom"` 自定义导航（隐藏原生导航栏）
- 首页启用下拉刷新（`enablePullDownRefresh: true`）

# 5. 模块边界 (Module Boundaries)

## 5.1 分层架构

```
┌──────────────────────────────────────────────────┐
│                    页面层 (Pages)                  │
│  组装组件、编排 API 调用、处理用户交互、响应路由    │
├──────────────────────────────────────────────────┤
│             状态管理层 (Stores)                    │
│  全局共享状态（用户/IM/应用）、跨页面数据传递       │
├──────────────────────────────────────────────────┤
│               API 层 (api/)                       │
│  HTTP 请求封装、Token 附加、错误统一处理            │
├──────────────────────────────────────────────────┤
│              工具层 (utils/)                       │
│  IM SDK 封装、COS 上传、第三方服务初始化            │
├──────────────────────────────────────────────────┤
│             组件层 (Components)                    │
│  纯展示 + 用户交互，不直接调 API 或 Store           │
└──────────────────────────────────────────────────┘
```

**核心约束：**
- 上层可以依赖下层，下层**禁止**依赖上层
- 页面层 → 可以调 API 层 + Store + 组件
- 组件层 → 只能接收 Props 和发送 Events，不能调 API 或直接改 Store
- API 层 → 只做 HTTP 请求，不含业务逻辑
- Store → 管理状态，可被页面层使用，但不能导入组件

## 5.2 API 模块边界

### 5.2.1 模块职责

| API 模块 | 拥有的接口 | 对应的后端路由前缀 |
|---------|-----------|------------------|
| `api/auth.js` | login / logout / getPhoneNumber / refreshToken / getUserInfo | `/api/auth/*` |
| `api/product.js` | getProductList / getProductDetail / createProduct / updateProduct / offlineProduct / searchProducts | `/api/products/*` |
| `api/order.js` | createOrder / getOrderList / getOrderDetail / confirmMet / confirmReceived / cancelOrder | `/api/orders/*` |
| `api/review.js` | submitReview / getReviewList / getReviewStats | `/api/reviews/*` |
| `api/report.js` | submitReport / getReportList / getReportDetail | `/api/reports/*` |
| `api/admin.js` | getDashboard / getUserList / banUser / unbanUser / getProductList / offlineProduct / getTicketList / resolveTicket | `/api/admin/*` |

### 5.2.2 模块隔离规则

- **禁止跨模块导入**：`api/product.js` 不应导入 `api/order.js` 的函数
- **统一出口**：所有 API 模块共享 `api/client.js` 的基础请求封装
- **返回格式统一**：所有 API 函数返回 Promise，resolve 时返回 `response.data`（即 `{ code, message, data }`），reject 时返回错误对象

```js
// api/product.js 示例
import { get, post } from './client'

export function getProductList(params) {
  return get('/api/products', params)  // → Promise<{ code, message, data }>
}

export function getProductDetail(id) {
  return get(`/api/products/${id}`)
}
```

## 5.3 Store 模块边界

### 5.3.1 Store 职责划分

| Store | 拥有的数据 | 禁止触碰 |
|-------|-----------|---------|
| `useUserStore` | token / refreshToken / userInfo / role | 订单/商品/聊天数据 |
| `useChatStore` | IM 连接状态 / 未读计数 / 会话列表 | 用户认证状态 |
| `useAppStore` | 全局 loading / 网络状态 / 错误信息 | 具体业务数据 |

### 5.3.2 跨 Store 访问规则

- 允许：一个 Store 的 actions 中读取另一个 Store 的 state（如 `useChatStore` 中读取 `useUserStore` 的 userId）
- 禁止：一个 Store 直接修改另一个 Store 的 state（必须通过目标 Store 的 actions）

```js
// 正确：在 chat store 中读取 user store
const userStore = useUserStore()
const userId = userStore.userInfo.id

// 错误：直接修改
userStore.token = 'xxx' // 禁止
```

## 5.4 组件所有权与导入规则

### 5.4.1 组件归属

| 组件 | 所属层级 | 可被谁使用 |
|------|---------|-----------|
| ProductCard | 通用业务组件 | 首页/搜索页/我发布的/他人主页 |
| FilterSidebar | 通用业务组件 | 搜索页 |
| ImageUploader | 通用组件 | 发布商品页/举报表单页 |
| StarRating | 通用组件 | 评价创建页/商品详情页（只读） |
| AppNavbar | 通用组件 | 任何需要自定义导航栏的页面 |
| EmptyState | 通用组件 | 任何列表页/详情页的空状态 |

### 5.4.2 导入规则

- 页面导入组件：通过 easycom 自动解析（`c-` 前缀），无需手动 `import`
- 页面导入 API：`import { getProductList } from '@/api/product'`
- 页面导入 Store：`import { useUserStore } from '@/store/user'`
- 页面导入 Utils：`import { initIM } from '@/utils/im'`
- 组件内部：**禁止**导入 API 模块或 Store 模块

## 5.5 跨模块通信规则

### 5.5.1 页面间通信

| 场景 | 方式 | 说明 |
|------|------|------|
| 页面跳转并传参 | URL query 参数 | `uni.navigateTo({ url: '/pages/order/detail?id=42' })` |
| 页面返回并刷新 | `uni.$emit` + `onShow` 中监听 | 如：发布商品完成后返回首页，触发首页刷新 |
| 跨页面共享数据 | Pinia Store | 如：用户信息在所有页面中统一从 `useUserStore` 读取 |

### 5.5.2 页面与组件通信

```
页面 ──Props──→ 组件          (数据流向: 单向)
页面 ←──Events── 组件          (事件流向: 单向)
```

- 组件**不能**通过 `$parent` 或 `getCurrentInstance` 访问页面实例
- 组件**不能**调用 `uni.navigateTo` 等路由 API（页面跳转由页面层控制）

### 5.5.3 全局事件

仅在以下场景使用 `uni.$emit` / `uni.$on`：
- App.vue 拦截网络状态变化 → 通知当前页面显示/隐藏网络异常提示
- Token 刷新失败 → 通知所有页面跳转登录页

禁止将全局事件用于组件间通信或页面间常态数据传递。

## 5.6 模块拥有矩阵 (OWNERSHIP MATRIX)

| 模块 | 拥有 (OWNS) | 禁止触碰 (MUST NOT TOUCH) |
|------|------------|--------------------------|
| 页面 (Pages) | 自己的路由参数、自己的 UI 状态、自己的数据获取/提交逻辑 | 其他页面的内部状态、原生导航栏（应使用 AppNavbar 组件） |
| API 模块 | 自己对应的 HTTP 端点 | 其他 API 模块的端点、直接操作 Store |
| Store (user) | token / userInfo / 登录登出逻辑 | IM 连接、会话列表、商品/订单数据 |
| Store (chat) | IM 状态 / 未读计数 / 会话缓存 | 用户认证 Token、订单商品数据 |
| Store (app) | 全局 UI 标志位 | 任何业务数据 |
| 组件 | 自己的 Props 类型接口、自己的内部 UI 状态、Events 签名 | API 调用、Store 修改、路由跳转、全局事件发射 |
| Utils | IM SDK 封装、COS 上传工具 | 业务逻辑判断、UI 状态 |
| Styles | 设计 Token 定义、全局重置样式 | 组件特定的样式（应在组件 scoped 中定义） |

# 6. 复用规则 (Reuse Rules)

## 6.1 组件复用

### 6.1.1 何时创建新组件

满足以下**任一**条件时，将代码抽取为独立组件：

1. **相同 UI 在 3 个及以上页面中出现**（如：商品卡片出现在首页、搜索页、我发布的、他人主页 → 抽取为 ProductCard）
2. **逻辑自成体系且超过 80 行**（如：图片上传逻辑包含选择/校验/上传/预览/删除/排序 → 抽取为 ImageUploader）
3. **需要通过 Props 配置多种变体**（如：空状态有 4 种类型 → 抽取为 EmptyState）

### 6.1.2 何时不创建新组件

以下情况**不要**创建新组件：

1. **仅在一个页面中使用且小于 80 行** → 直接在页面内实现
2. **仅为一个页面服务的小 UI 片段**（如：某个页面的特定 header） → 在页面内用 `<template>` 片段
3. **逻辑过于简单**（如：仅封装了 `<view>` + 一个 class） → 不值得创建组件

### 6.1.3 组件 Props 设计原则

- Props 优先使用基本类型（string/number/boolean），复杂数据使用 interface 定义
- 每个 Props 必须有明确的默认值或标注必填
- 布尔 Props 默认值应为 `false`
- Props 数量 ≤ 8 个。超过 8 个说明组件职责过多，应拆分

## 6.2 函数复用：Composables (useXxx)

### 6.2.1 Composable 模式

将可复用的有状态逻辑抽取为 Composable（组合式函数），遵循 `useXxx` 命名规范。

**项目中需要创建的 Composable：**

| Composable | 文件 | 用途 |
|-----------|------|------|
| `usePageScroll` | `composables/usePageScroll.ts` | 分页加载通用逻辑（列表页复用） |
| `useCountdown` | `composables/useCountdown.ts` | 倒计时逻辑（订单超时/评价截止） |
| `useImagePreview` | `composables/useImagePreview.ts` | 图片预览（商品详情/聊天图片） |
| `useFormValidation` | `composables/useFormValidation.ts` | 表单实时校验（发布商品/举报表单） |

### 6.2.2 Composable 规范

```ts
// composables/usePageScroll.ts 示例
import { ref, onMounted } from 'vue'

export function usePageScroll<T>(fetchFn: (page: number, size: number) => Promise<T[]>) {
  const list = ref<T[]>([])
  const page = ref(1)
  const loading = ref(false)
  const hasMore = ref(true)
  const PAGE_SIZE = 20

  async function loadMore() {
    if (loading.value || !hasMore.value) return
    loading.value = true
    try {
      const data = await fetchFn(page.value, PAGE_SIZE)
      list.value.push(...data)
      hasMore.value = data.length === PAGE_SIZE
      page.value++
    } finally {
      loading.value = false
    }
  }

  function refresh() {
    page.value = 1
    list.value = []
    hasMore.value = true
    return loadMore()
  }

  onMounted(() => loadMore())

  return { list, loading, hasMore, loadMore, refresh }
}
```

**使用示例：**
```vue
<script setup lang="ts">
import { usePageScroll } from '@/composables/usePageScroll'
import { getProductList } from '@/api/product'
import type { ProductCardData } from '@/components/product-card/types'

const { list: products, loading, hasMore, loadMore, refresh } =
  usePageScroll<ProductCardData>((page, size) =>
    getProductList({ page, size }).then(res => res.data)
  )
</script>
```

### 6.2.3 Composable vs 组件 vs Store 判断

| 如果... | 则使用... |
|---------|----------|
| 需要复用一段**有状态的 JS 逻辑**（无模板） | Composable (`useXxx`) |
| 需要复用一段**UI 模板 + 逻辑** | 组件 |
| 需要**跨页面共享状态** | Pinia Store |

## 6.3 样式复用：三层级联

详见 [§7. 样式系统](#7-样式系统-style-system)。

简要规则：
- **tokens.scss**：定义 SCSS 变量 → 全局自动注入，无需手动 import
- **common.scss**：全局重置 + 高频工具类 → 在 App.vue 中全局导入
- **组件级 scoped**：组件特有样式 → 每个 `.vue` 文件的 `<style scoped>`

**DRY 阈值：** 同一样式模式在 3 个及以上组件中出现时，提升到 `common.scss` 作为工具类或提取为 SCSS mixin。

## 6.4 API 复用：模块封装规则

### 6.4.1 API 模块拆分原则

- 按 REST 资源拆分（`auth`、`product`、`order`、`review`、`report`、`admin`）
- 一个 API 模块对应一个后端路由前缀
- 同一资源的 CRUD 操作放在同一模块中

### 6.4.2 禁止模式

- 禁止在页面中直接使用 `uni.request`（必须通过 API 模块封装）
- 禁止在组件中调用 API 模块
- 禁止相同的 API 调用出现在 2 个不同文件中（提取到对应 API 模块）

### 6.4.3 API 函数签名规范

```js
// 标准 CRUD 命名
getXxxList(params)       // GET    /api/xxx        → 列表
getXxxDetail(id)         // GET    /api/xxx/:id    → 详情
createXxx(data)          // POST   /api/xxx        → 创建
updateXxx(id, data)      // PUT    /api/xxx/:id    → 更新
deleteXxx(id)            // DELETE /api/xxx/:id    → 删除（软删除）

// 自定义操作命名（动词 + 资源）
confirmMet(orderId)      // POST   /api/orders/:id/met
cancelOrder(orderId)     // POST   /api/orders/:id/cancel
```

## 6.5 Store 复用

| Store | 服务的页面 |
|-------|-----------|
| `useUserStore` | 全部页面（全局用户状态） |
| `useChatStore` | 聊天列表（`chat/list`）、聊天详情（`chat/detail`）、首页（未读角标） |
| `useAppStore` | 全部页面（网络状态提示、全局 loading） |

**规则：**
- 页面中只能使用其职能范围内的 Store。如：订单列表页不应使用 `useChatStore`
- Store 之间不应循环依赖（A → B → A）

## 6.6 DRY 阈值

| 重复次数 | 动作 |
|:--:|------|
| 1-2 次 | 允许重复，不提取 |
| 3+ 次 | **必须提取** — 相同 UI → 组件；相同逻辑 → Composable；相同样式 → common.scss / tokens.scss；相同 HTTP 调用 → API 模块 |

# 7. 样式系统 (Style System)

## 7.1 三层级联架构

本项目采用严格的三层样式级联，确保样式复用、一致性、可维护性：

```
┌─────────────────────────────────────────────┐
│ Layer 1: tokens.scss                        │
│ 设计 Token（SCSS 变量）                       │
│ 范围：颜色/字体/间距/圆角/阴影/组件尺寸        │
│ 注入：vite.config.js additionalData 全局注入  │
│ 内容：仅有 SCSS 变量，无 CSS 规则              │
├─────────────────────────────────────────────┤
│ Layer 2: common.scss                        │
│ 全局公共样式                                  │
│ 范围：reset / 工具类 / 高频 mixin              │
│ 导入：App.vue 中全局导入（无 scoped）           │
│ 内容：作用于 page/全局元素的 CSS 规则           │
├─────────────────────────────────────────────┤
│ Layer 3: Component <style scoped>           │
│ 组件级样式                                    │
│ 范围：单个组件的专属样式                       │
│ 命名：BEM (Block__Element--Modifier)         │
│ 内容：仅作用于当前组件的 CSS 规则              │
└─────────────────────────────────────────────┘
```

### 7.1.1 级联规则

1. **Layer 1 (tokens.scss)** 仅定义变量，不输出任何 CSS 规则
2. **Layer 2 (common.scss)** 可以引用 Layer 1 的变量，输出全局 CSS 规则
3. **Layer 3 (组件 scoped)** 通过 Vite 自动获得 Layer 1 的变量（`additionalData`），可引用 Layer 1 和 Layer 2。组件样式通过 `scoped` 属性隔离

## 7.5 响应式设计：rpx 单位

### 7.5.1 rpx 换算

微信小程序 rpx（responsive pixel）是响应式单位。规定屏幕宽度为 750rpx。

本项目设计基准为 375px 逻辑宽度，因此 **1px = 2rpx**。

| 设计稿 px | rpx 值 | 说明 |
|-----------|--------|------|
| 375px (屏幕宽) | 750rpx | 全屏宽度 |
| 16px | 32rpx | 页面边距 |
| 14px | 28rpx | 正文 |
| 12px | 24rpx | 辅助文字 |
| 8px | 16rpx | 元素间距 |

### 7.5.2 rpx 使用规则

- **必须使用 rpx 的场景**：所有 `font-size`、`padding`、`margin`、`width`、`height`、`border-radius`、`gap`
- **可以使用 px 的场景**：`border-width`（1px 边框）、`box-shadow` 偏移量
- **禁止混合使用**：同一组件中不应混用 rpx 和 px 表示同类尺寸

### 7.5.3 屏幕适配

| 设备 | 逻辑宽度 | 1rpx 实际像素 |
|------|---------|--------------|
| iPhone SE (320px) | 320px | 0.427px |
| iPhone 6/7/8 (375px) | 375px | 0.5px |
| iPhone 12/13/14 (390px) | 390px | 0.52px |
| iPhone 14 Pro Max (430px) | 430px | 0.573px |

rpx 自动处理上述适配，无需手动编写媒体查询。

## 7.6 主题一致性强制规则

### 7.6.1 颜色使用强制检查

**CI/CD 或提交前检查（推荐添加 stylelint）：**

```json
// .stylelintrc.json 关键规则
{
  "rules": {
    "color-no-hex": true,                // 禁止原始 hex 色值（必须使用 SCSS 变量）
    "declaration-no-important": true,    // 禁止 !important
    "declaration-property-value-disallowed-list": {
      "/^margin/": ["/px/"],             // margin 禁止使用 px（必须用 rpx）
      "/^padding/": ["/px/"],            // padding 禁止使用 px
      "/^font-size/": ["/px/"]           // font-size 禁止使用 px
    }
  }
}
```

### 7.6.2 样式 Code Review 检查项

- [ ] 是否使用了原始 hex 色值？（应使用 SCSS 变量）
- [ ] 是否使用了 px 而非 rpx？（border 除外）
- [ ] 间距是否为 8 的倍数？（8rpx / 16rpx / 24rpx / 32rpx / 48rpx / 64rpx）
- [ ] 圆角是否为 16rpx 或 24rpx？（不要出现 10rpx、14rpx 等自定义值）
- [ ] 颜色数量是否 ≤ tokens.scss 中定义的色值？
- [ ] 是否有内联样式？(`<view style="color: red">`)
- [ ] 字号是否来自字号层级？（必须使用 $text-xs ~ $text-3xl）

## 7.7 反模式 (Anti-Patterns)

### 7.7.1 内联样式

```vue
<!-- ❌ 禁止 -->
<view style="color: #FF0000; font-size: 14px; margin-top: 10px;">

<!-- ✅ 正确 -->
<view class="my-component__error-text">
<style lang="scss" scoped>
.my-component__error-text {
  color: $color-error;
  font-size: $text-base;
  margin-top: $space-content;
}
</style>
```

### 7.7.2 魔数

```scss
/* ❌ 禁止：无意义的数字 */
.product-card {
  margin-top: 13rpx;    // 不是 8 的倍数
  border-radius: 7rpx;  // 不在圆角系统中
}

/* ✅ 正确：使用 tokens.scss 变量 */
.product-card {
  margin-top: $space-content;  // 16rpx
  border-radius: $radius-card; // 16rpx
}
```

### 7.7.3 颜色硬编码

```scss
/* ❌ 禁止：原始 hex 值 */
.my-button {
  background: #4A90D9;
  color: #FFFFFF;
}

/* ✅ 正确：使用 SCSS 变量 */
.my-button {
  background: $color-primary;
  color: $color-surface;
}
```

### 7.7.4 过度嵌套

```scss
/* ❌ 禁止：过度嵌套（> 3 层） */
.page {
  .section {
    .card {
      .header {
        .title {
          .text {
            color: $color-title;  // 6 层嵌套
          }
        }
      }
    }
  }
}

/* ✅ 正确：使用 BEM 扁平化 */
.page-section__card-title {
  color: $color-title;
}
```

### 7.7.5 !important 滥用

```scss
/* ❌ 禁止 */
.my-text {
  color: $color-primary !important;  // 绝不使用 !important
}

/* ✅ 正确：通过提高选择器特异性解决 */
.product-detail .product-detail__price {
  color: $color-primary;
}
```

### 7.7.6 弹窗套弹窗

```
❌ 禁止：在弹窗中再弹一个弹窗
（来源：rules/ui-rules.md — "弹窗套弹窗"）

✅ 正确：在弹窗中需要二次确认时，使用弹窗内按钮的状态变化
（如：点击"封禁" → 按钮变为"确认封禁" + "取消"两个按钮）
```

- [ ] 确定商业模式（可后置）

### 0.2 工程优先级矩阵 🆕🆕

> ⚠️ AI 项目最大死因之一：「还没用户，先做多 Agent 平台。」什么都想做，结果没一个真正完成。

| 等级 | 时机 | 内容 |
|------|------|------|
| **P0 必须优先** | 立刻 | 核心用户闭环、MVP、登录、核心 Agent、安全、数据一致性 |
| **P1 上线前** | 内测前 | 监控、测试、成本控制、缓存、限流 |
| **P2 有用户后** | 1-100 用户 | A/B Test、数据分析、自动化运营、Prompt 优化 |
| **P3 规模化后** | 1000+ 用户 | 多 Agent、工作流平台、Agent 市场、团队协作、平台生态 |

### 2.3 安全架构设计 🆕

> 🕳️ AI Agent 最大风险不是代码 Bug，而是「模型被用户骗了」。很多 Agent 死于 Prompt Injection、Tool 越权、API Key 泄露。

- [ ] **API Key 安全管理** — 环境变量隔离，禁止硬编码，定期轮换
- [ ] **Prompt Injection 防护** — 输入清洗、系统指令隔离、用户输入边界标记
- [ ] **用户权限系统（RBAC）** — 角色-权限模型，Tool 级别访问控制
- [ ] **Tool 越权防护** — 每个 Tool 执行前做权限校验，禁止用户注入 Tool 调用
- [ ] **敏感词过滤** — 输入输出双向过滤
- [ ] **文件上传安全检查** — 类型白名单、大小限制、病毒扫描
- [ ] **SQL 注入防护** — 参数化查询，禁止拼接 SQL
- [ ] **XSS / CSRF 防护** — 输出编码、CSRF Token
- [ ] **Webhook 签名验证** — 防伪造回调
- [ ] **数据加密** — 传输层 TLS、存储层 AES
- [ ] **用户隐私合规** — 数据最小化、用户删除权

### 2.4 高级 AI 安全 🆕🆕

> 🔴 AI Agent 还有一套普通安全覆盖不到的攻击面。

- [ ] **Indirect Prompt Injection 防护** — 防止通过 RAG 文档/网页/邮件注入恶意指令
- [ ] **RAG 污染检测** — 检测知识库是否被恶意数据污染
- [ ] **恶意 Embedding 检测** — 防止对抗性向量攻击
- [ ] **Tool Chain Attack 防护** — 防止多个 Tool 串联执行恶意操作
- [ ] **越狱 Prompt 数据集** — 积累对抗样本持续测试
- [ ] **Prompt Leak 防护** — 防止用户诱导 Agent 泄露系统 Prompt
- [ ] **Agent 自我修改限制** — 禁止 Agent 修改自身 Prompt 或配置
- [ ] **模型输出安全过滤** — 输出端内容安全检查
- [ ] **敏感信息脱敏** — PII/密钥/内部信息自动打码
- [ ] **AI 红队测试** — 定期模拟攻击验证安全性

### 5.4 成本控制系统 🆕

> 💰 「能跑」≠ 「成本可控」。AI 产品的边际成本是 Token，不是服务器。无限 Token 消耗能一夜烧光预算。

- [ ] **Token 成本预算系统** — 按用户/会话/日设定 Token 上限
- [ ] **用户额度系统** — 每个用户独立的 Token 配额
- [ ] **模型降级策略** — 简单任务用小模型，复杂任务用大模型
- [ ] **模型路由系统** — 根据任务类型自动选择最优模型
- [ ] **缓存命中策略** — 相同 Prompt 复用缓存结果
- [ ] **Tool 调用成本监控** — 每次 Tool 调用的 Token 消耗统计
- [ ] **长上下文截断策略** — 超过阈值自动摘要压缩
- [ ] **高峰限流策略** — 突发流量保护

### 5.5 Agent 治理系统 🆕🆕

> 🔒 Agent 最大风险不是「不会干活」，而是「太会干活」——无限递归、Tool 狂调用、自主执行危险操作、Token 爆炸。

- [ ] **最大递归深度** — 防止 Agent 无限循环
- [ ] **最大 Tool 调用次数** — 单次任务 Tool 调用上限
- [ ] **最大 Token 消耗限制** — 单次/单用户/单日 Token 硬上限
- [ ] **高风险操作人工确认** — 删除/支付/发邮件等操作必须人工批准
- [ ] **自动执行白名单** — 只有白名单内的 Tool 可以自动执行
- [ ] **Agent 行为审计日志** — 完整记录每一步决策和原因
- [ ] **多 Agent 权限隔离** — 每个 Agent 独立权限空间
- [ ] **自主任务终止条件** — 定义明确的终止规则
- [ ] **Agent Sandbox** — 高风险 Agent 隔离运行
- [ ] **Tool 风险等级系统** — 每个 Tool 标注风险等级（低/中/高/危险）

### 5.6 优雅降级系统（Fault Tolerance）🆕🆕

> 🛡️ 真正成熟系统不是「永不崩溃」，而是「崩了还能活」。线上一定会出现：OpenAI 超时、Anthropic 限流、DeepSeek 宕机、向量库异常、Tool 超时、Token 超限。没有降级 = 一个 API 超时 = 整个产品死亡。

- [ ] **模型不可用自动切换** — OpenAI 挂了切 Anthropic，再切 DeepSeek
- [ ] **Tool 超时 fallback** — Tool 不可用时返回预设降级回复
- [ ] **RAG 失败自动降级普通对话** — 知识库不可用时用模型自身知识
- [ ] **高峰期自动关闭高级功能** — 负载过高时只保留核心对话
- [ ] **API 超限后的用户提示** — 明确告知用户当前状态和预计恢复时间
- [ ] **流式输出中断恢复** — 断流后自动重连
- [ ] **熔断机制（Circuit Breaker）** — 连续失败 N 次后自动暂停服务
- [ ] **Retry Backoff 策略** — 指数退避重试，避免雪崩
- [ ] **部分功能降级机制** — 非核心功能可独立关闭
- [ ] **离线缓存机制** — 网络异常时返回缓存内容

### 5.7 工作流编排 🆕🆕

> 🔀 后期 Agent 会越来越复杂，单个 Agent 已不够，需要编排多个 Agent 协作。

- [ ] **DAG Workflow** — 有向无环图定义任务依赖关系
- [ ] **可恢复 Workflow** — 中断后从断点继续
- [ ] **Workflow Retry** — 失败步骤自动重试
- [ ] **Step Checkpoint** — 关键步骤保存中间结果
- [ ] **Human-in-the-loop** — 关键决策点人工介入
- [ ] **Workflow Timeout** — 全局超时保护
- [ ] **Workflow Debugger** — 可视化追踪每个步骤的执行过程
- [ ] **可视化 Workflow** — 图形化展示和编辑工作流

### 8.3 数据生命周期管理 🆕🆕

> 🫧 AI 项目后期最容易腐化的是数据层。对话历史、Embedding、向量数据、上传文件堆积 → 数据库 = AI 沼泽。

- [ ] **对话历史归档** — 超过 N 天的对话自动归档冷存储
- [ ] **Embedding 过期策略** — 设定向量数据的有效期

- [ ] **自动数据清理** — 定时任务清理过期数据
- [ ] **用户数据删除机制** — GDPR「被遗忘权」支持
- [ ] **冷热数据分层** — 热数据（最近7天）→ 温数据（30天）→ 冷数据（归档）
- [ ] **向量库重建机制** — 数据变更后重建索引的方案
- [ ] **数据压缩策略** — 历史对话摘要化存储
- [ ] **长期存储策略** — 法律要求/分析价值的长期保留

## AI Native 工程哲学 🆕🆕

> 🧭 顶级工程体系最后一定不是「功能清单」，而是「原则」。工具会变、模型会变、框架会变——**但原则不会**。

### 10 条 AI Native Engineering Principles

**1. AI 不可信，必须可验证**
所有 AI 输出都必须可验证。禁止盲信 Agent。

**2. Prompt 是代码**
必须版本化、必须测试、必须 Review。

**3. 小步迭代优于大爆炸开发**
一次只改一个模块，每次修改必须可回滚。

**4. 能观测比能运行更重要**
看不见的问题无法修复。AI 系统的第一优先级不是功能，是可观测性。

**5. 先稳定，再智能**
稳定的 80 分 Agent 强于随机发疯的 95 分 Agent。

**6. Agent 必须被约束**
Agent 不能拥有无限自由。Tool 必须有权限边界，自动执行必须有终止条件。

**7. 成本是 AI 产品的一部分**
Token 就是服务器成本，Prompt 长度就是资源消耗。成本意识从 Day 1 开始。

**8. 所有自动化都必须可中断**
自动执行必须支持人工接管。永远保留「紧急停止」按钮。

**9. 所有复杂系统最终都会腐化**
必须持续重构、持续清理、持续简化。熵增是宇宙规律，架构腐化是必然——对抗它需要持续投入。

**10. AI 系统本质是概率系统**
不存在 100% 稳定，只能提高置信度。接受不确定性，设计容错机制。

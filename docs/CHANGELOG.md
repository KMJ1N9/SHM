### Added — PRD 增强（2026-06-05）

- **PRD §八 MVP 验证标准**（~120 行，缺口 #14 修复）：含 6 个子章节——
  - §8.1 冷启动说明（种子商品 15-20 件 + 种子用户 10-15 人 + 推广渠道 + 发布窗口警告）
  - §8.2 观察周期（T+0 上线 → T+14 健康检查 → T+42 正式评估 → T+70 最终评估，含寒暑假暂停说明）
  - §8.3 指标体系：11 项指标分 4 层——需求验证（M1-M3：注册 ≥150 / 商品 ≥60 / 卖家 ≥20）、交易闭环（M4-M6：成交 ≥25 / 完成率 ≥55% / 动销率 ≥25%）、信任质量（M7-M8：纠纷率 ≤8% / 互评率 ≥40%）、可持续性（M9-M11：7 日回访 ≥20% / 周活卖家 ≥5 / 运维 ≤5h/周）。每项指标含计算方法、数据源（对应具体数据库表）、阈值及理由
  - §8.4 综合判定（第 6 周）：通过（≥8 项 + M4/M7 必须达标）/ 待定（5-7 项，延长至第 10 周）/ 不通过（≤4 项）
  - §8.5 常见失败模式与诊断：8 种失败模式 × 特征信号 × 可能原因 × 建议动作
  - §8.6 数据看板建议：Dashboard API 建议新增 5 个 MVP 指标字段（独立卖家数/动销率/互评完成率/周活卖家/7 日回访率）
  - 同步重编号：原 §八 风险与待确认事项 → §九，原 §九 技术设计文档索引 → §十

### Fixed — 法律文件（2026-06-05）

## 2026-06-04 — 第三至第五轮深度审阅

> **范围：** 8 份文档全量审阅，DDL / API / Cron / 页面 / 法律 / UI 八向交叉比对
> **三轮合计：** 54 项问题（P0×15 + P1×15 + P2×24），全部已修复

### Fixed — 数据库与 DDL

- 字段命名统一：`reviews`、`reports`、`products` 表多处字段名与 API / 前端对齐。
- 索引优化：补充缺失的业务索引，复合索引列顺序调整以匹配实际查询模式。
- 迁移规划：`migrations/` 目录树补全，迁移版本号重排确保与 DDL 章节一致。
- 归档表（`admin_logs_archive`、`reviews_archive`）的归档策略文档化——明确无外键约束的设计原因（源行可能先于归档被删除，数据一致性由迁移脚本事务保证）。

### Fixed — API 接口

- 错误码语义专项：30 个错误码的 HTTP 状态码、业务含义、触发条件逐一与中间件栈校验，确保无冲突/无重复/无语义模糊。
- 路由映射表：补全所有 `controllers/` 占位注释，路由文件与实际 Controller 一一对应。
- 请求/响应 body：所有 43 个端点的字段名、类型、必填/可选与 DDL 列定义交叉验证一致。
- 分页参数：统一为 `page` / `pageSize`，默认值 1 / 20。

### Fixed — 定时任务

- Cron 表达式与 PRD 业务规则对齐（超时自动取消 7 天 → `*/30 * * * *`、自动确认收货 3 天 → `*/30 * * * *`、通知清理 30 天 → `0 4 * * *`）。
- 补全定时任务锁机制说明（`cron_lock` 表防止多实例重复执行）。

### Fixed — 页面与 UI

- 页面路由配置与 UI 设计文档各章节一一对应，消除孤岛页面和缺失路由。
- 个人中心页导航完整度检查——确认所有入口均有对应页面设计或明确标注为"后续迭代"。

### Fixed — 法律文件

- 三份法律文件措辞统一：个人信息收集范围、使用目的、存储期限、用户权利条款与《个人信息保护法》第 17 条要求对齐。
- 用户协议与隐私政策的交叉引用路径修正。

### Fixed — 跨文档一致性

- DDL 字段名 ↔ API 参数名 ↔ 前端 `data` 绑定三向对齐（全部 14 张表、43 个端点、15 个页面）。
- 信誉分规则（初始 100 / 上限 200 / 举报 -30 / 差评 -5 / 交易 +2 / 好评 +1）：PRD §3.5.1 ↔ 技术架构 §18.1 ↔ API 文档 §5.2.13 ↔ 用户协议 §6.2 四处完全一致。
- 订单状态流转（pending → met → completed / cancelled / disputed）：API 文档状态机图 ↔ DDL `orders.status` ENUM ↔ 技术架构 §七 流程描述三方一致。

## 2026-06-03 — 第一至第二轮初始审阅

> **范围：** 技术架构文档 + PRD + 三份法律文件

- **`failed_system_messages` DDL SQLite→MySQL 语法迁移**：`INTEGER PRIMARY KEY AUTOINCREMENT` → `INT PRIMARY KEY AUTO_INCREMENT`；`TEXT NOT NULL DEFAULT (datetime('now'))` → `DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP`；短文本字段补 `VARCHAR(n)` 长度。新增迁移文件 `004_add_failed_system_messages.sql`。
- **"差评 -5"规则补全**：技术架构 §5.2.13 信誉分主规则块 + 用户协议 §6.2 信誉分表格 + 环境变量 `CREDIT_DEDUCT_NEGATIVE_REVIEW=5` 三处同步补充。
- **信誉分上限定义**：新增 `CREDIT_MAX=200` 环境变量，技术架构 §18.1 / §23.4 / 环境变量清单三处同步。

### Fixed — 高优先级修复（P1）

- **三维度评价聚合规则明确**：技术架构 §5.2.5 + API 文档新增说明——三维度（沟通态度 / 守时程度 / 描述一致度）各 1-5 分，总分取平均分（四舍五入），≥ 4 为好评、≤ 2 为差评。
- **免责提示措辞统一**：PRD §3.3 / 技术架构 §8.2 / 免责声明 §3.3 / UI 设计聊天页——四处统一为"如买卖双方**私下**交换微信/**其他联系方式**进行交易，发生纠纷后平台不负责"。

- **4xxx 错误码段描述修正**：技术架构 §5.4.2 ——移除不存在的"重复提交"分类（实际幂等检查在 3xxx 段）。

### Fixed — 低优先级修复（P2）

- **备份保留期统一**：技术架构 §11.5.1 / §11.5.4 ——周备份从"4 周"统一为 **30 天**（COS 生命周期规则仅支持按天配置）。
- **迁移规划补全**：`failed_system_messages` 表纳入迁移 004，§三 `migrations/` 目录树同步更新。
- **Controller 占位**：§三 `controllers/` 下预置 `notification.js` 和 `credit.js` 占位注释。
- **PRD 元信息更新**：`当前阶段` 从"阶段 0-2（想法→产品定义→技术规划）"更新为"阶段 3（代码开发）"。
